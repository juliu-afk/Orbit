3. 5 大可学点 → 5 个 ContextPrebuilder

核心改造：在 TaskRunner._build_context() → Agent.run() 之间插入一层 ContextPrebuilder——按 Agent 角色 + 任务类型，确定性预处理后再交给 Agent。

📌 学习点 1：Pre-dispatch Context Builder per Role

现状：TaskRunner._build_context() 构建通用 TaskContext → 所有 Agent 共享同一份 context dict。

改造：新增 src/orbit/context/prebuilder.py——ContextPrebuilder 基类 + 5 个角色子类。

角色ContextPrebuilder裁剪策略
ClarifierClarifierContextPrebuilder保留：用户原始需求 + 项目说明书 + 关键词。丢弃：代码细节、测试输出
ArchitectArchitectContextPrebuilder保留：影响面分析 + 候选文件清单 + 依赖图。丢弃：完整文件内容
DeveloperDeveloperContextPrebuilder保留：变更范围 + 相关代码片段(≤5) + 现有测试。丢弃：无关模块代码
ReviewerReviewerContextPrebuilder保留：diff 摘要 + 权限变更 + schema 变更 + 风险点。丢弃：完整 diff（类比报告 reviewer-input.md）
QAQAContextPrebuilder保留：应测清单 + 覆盖率缺口 + 边界条件。丢弃：完整测试输出

预期节省：每 Agent 调用 -30~50% context 体积。

📌 学习点 2："不含完整 Diff"原则下沉到 Agent 间通信

现状：Orchestrator dispatch 时，input_data.context 可塞任意大段原始输出。无规则要求"不塞完整大文件"。

改造：在 TaskContext 层加约束——

  1. TaskContext 新增 max_chars_per_field 配置，默认 5000 chars
  2. 超过阈值的内容自动截断 + 摘要替代（用已有 _truncate_output 逻辑）
  3. Agent 间消息（MessageBus）也应用此约束——不转发超过 N chars 的消息体

类比：PR 审核中 reviewer-input.md 不含完整 diff（6KB vs 1397行 diff），Orbit 的 ReviewerAgent 也应该收到 diff 摘要而非完整 diff。

预期节省：Agent 间通信 -40~60% token。

📌 学习点 3：7 个 Phase 2 脚本 → Orbit 的 7 个 Context Builder

报告 §11 的 7 个脚本直接映射到 Orbit 的功能模块。

报告脚本Orbit 模块输入输出
build_test_input.pycontext/builders/test_builder.py变更文件列表 + git diff应测清单（按优先级排序）
build_design_input.pycontext/builders/design_builder.py需求 PRD + 代码图谱影响面 + 候选文件 + 依赖图
build_implementation_input.pycontext/builders/impl_builder.py技术方案 + 代码图谱最小变更任务书（文件:函数 映射）

每个 builder 是纯函数——无 LLM 调用，确定性处理。产物是结构化 Markdown，直接注入对应 Agent 的 context。

📌 学习点 4：确定性预扫描器（per-Agent concern）

类比：scan_permissions.py 用正则扫 Python 文件找 require_permission("xxx") 字符串，交叉比对 rbac.py 注册表——确定性工具做，不让 LLM 猜。

Orbit 可做的预扫描器：

扫描器用途消费方
AffectedFilesScannergit diff → 分类文件列表（新增/修改/删除，按模块分组）Developer, Reviewer, QA
ImportDependencyScannerPython AST → 受影响 import 依赖图Architect, Developer
TestCoverageScannercoverage.json → 变更文件的覆盖率缺口QA
SchemaChangeScannerAlembic migration → 表/列变更摘要Reviewer
PermissionStringScanner正则扫 Python → 权限字符串 + 注册表比对Reviewer（已有模式，直接复用）

每个扫描器返回结构化 dict，注入对应 ContextPrebuilder。LLM 不参与扫描——只消费结果。

📌 学习点 5：变更范围感知 → 测试粒度决策

现状：Orbit 的 ReActAgent 没有"先看变更范围再决定跑什么"的逻辑——直接进 think→act→observe 循环。QA Agent 可能跑全量测试而非增量测试。

改造：在 TaskRunner 状态机中插入一个轻量 DiffScope 步骤——

  1. TaskState.SCOPING 新增状态（在 CODING 前）
  2. Scoping agent（规则引擎，非 LLM）分析 git diff → 输出变更范围报告
  3. 变更范围决定测试粒度：
    • 仅改了前端 → 冒烟级别
    • 改了核心模块 → 回归级别
    • 改了工具函数 → 单元+集成

类比：Token 节省规则 #5 "Diff 先于全量测试——不盲目全量回归"。

预期节省：QA Agent 每次调用 -50~70% 测试执行 token（不跑无关测试）。

6. Token 节省估算

来源节省 token/Agent调用机制
Pre-dispatch Context 裁剪-30~50%角色特定裁剪，不塞无关内容
"不含完整 diff"约束-40~60% (Reviewer)diff 摘要替代完整 diff
确定性预扫描-500~1500不用 LLM 判断——脚本直接给出结论
变更范围感知-50~70% (QA Agent)增量测试替代全量回归
Agent 间消息约束-20~30%大消息体自动截断+摘要
单任务典型节省：一个完整任务链（Clarifier → Architect → Developer → Reviewer → QA）
改造前：~5 次 Agent 调用 × 平均 8000 token context = ~40,000 token
改造后：~5 次 × 平均 4500 token context = ~22,500 token
单任务 -44% ≈ 17,500 token 节省

  1. 5 个 ContextPrebuilder 子类全部实现并通过单元测试
  2. 7 个 Context Builder 全部实现，用真实 Orbit PR 跑通
  3. 5 个预扫描器全部实现，输出结构化 dict
  4. TaskContext max_chars_per_field 生效——所有字段 ≤ 5000 chars
  5. Agent 间消息经 MessageBus 传递时自动截断
  6. SCOPING 状态正确识别变更范围 → 测试粒度决策正确
  7. Token 节省基准测试——改造后同任务 token 消耗 ≤ 改造前 70%
  8. 已有压缩管线（L1-L5）不受影响——压缩兜底仍正常工作
  9. 开发计划 + CLAUDE.md 更新
