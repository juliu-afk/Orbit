## 战略定调

两个减熵维度：

1. **内部减熵**——Orbit 自身代码质量，消除重复/空壳/上帝类
2. **业务层减熵**——Orbit 作为 AI 编程工具，降低用户项目的熵

**核心原则**：减熵不是大重构。每改一处——删掉的行数 > 新增的行数，就是对的。

## 一、内部减熵（5 类行动）

### 1.1 消重——同一概念两个实现（P0）

| 问题 | 位置 | 行动 |
|------|------|------|
| CircuitBreaker 双重实现 | `resource_guard/resource_guard.py` + `gateway/circuit_breaker.py` | ResourceGuard 改用 gateway CircuitBreaker 组合 |
| ClarifierAgent vs ClarificationEngine | `agents/clarifier.py` + `scheduler/clarifier.py` | 保留 scheduler/clarifier.py，agents/clarifier.py 的增强逻辑移植过去后删除 |

**目标**：-2 个文件，-~200 行重复代码。

### 1.2 拆大类——一个类做 5+ 件事（P1）

| 问题 | 位置 | 行动 |
|------|------|------|
| SchedulerOrchestrator 697行/7职责 | `scheduler/orchestrator.py` | 拆为 TaskRunner + DagRunner，复用已有 CheckpointManager |
| ResourceGuard 6职责 | `resource_guard/resource_guard.py` | 熔断委托给 CircuitBreaker，审计+指标抽独立 Observer |

**目标**：orchestrator.py → -300 行，拆为 3 个文件。

### 1.3 去空壳——只有 `__init__.py` 的目录（P1）

5 个空目录删除或填内容：
```
api/dependencies/   # 0 字节 → 删除
graph/schemas/      # 0 字节 → 删除
graph/models/       # 0 字节 → 删除
core/               # 0 字节 → 删除
infrastructure/     # 0 字节 → 删除
```

`knowledge/ontology/` 扁平化：`accounting.py` 提到 `knowledge/` 层。

**目标**：-5 个空目录，-1 层嵌套。

### 1.4 修循环依赖（P2）

`orchestrator.py` 5 个 TYPE_CHECKING + 4 处惰性导入。行动：
1. 画依赖图（`importlab` 或手工）
2. 找出环：`scheduler → agents → compression → tools → scheduler`
3. 切一刀：引入 protocol/接口打破环

**目标**：消除所有惰性导入 + TYPE_CHECKING guard。

### 1.5 去重复模板——防幻觉层相同 guard（P2）

5 个防幻觉层的 3 段重复 guard 代码抽为基类装饰器：

```python
# 现在：每个 layer 重复
if not code.strip():
    return ValidationResult(passed=True, warnings=["empty code, skipped"])

# 目标：装饰器
@skip_if_empty
@skip_if_no_sandbox
async def validate(self, code: str) -> ValidationResult: ...
```

**目标**：-~30 行重复代码，5 个文件各 -6 行。

## 二、业务层减熵（8 项策略）

### 2.1 上下文裁剪——相关性打分代替全文注入

**现状**：`context/matcher.py` 硬截断 5000 字符。`prompt/builder.py` 三层缓存，但 context 层是整文件注入。

**目标**：Tree-sitter 解析任务关键词 → 只注入包含该关键词的函数/类，按相关性排序注入。

**指标**：上下文注入量 -50%，任务完成率不变或提升。

### 2.2 决策日志——Agent 不可逆选择被持久化

**模型**：

```python
class DesignDecision:
    id: str              # "DD-042"
    choice: str          # "PostgreSQL over MySQL"
    why: str             # 理由
    constraints: list    # 约束条件
    alternatives: list   # 考虑过的替代方案
    made_by: AgentRole   # 决策者
    timestamp: datetime
```

**注入时机**：后续 Agent 调度时，`prompt/builder.py` 注入相关决策 → 不会重选。

### 2.3 编辑摇摆检测——高熵文件触发需求重确认

代码图谱追踪文件变更历史 → 检测编辑摇摆模式：
- 同一函数 7 天内改 4 次以上
- 同一逻辑改过又改回来
- 多个 Agent 对同一段代码反复修改

> 判定高熵 → 触发 ClarificationEngine 重新确认需求。

### 2.4 依赖膨胀拦截——安装前检查已有替代

Agent 提议新增依赖 → `resource_guard/` 拦截检查：
- 已有标准库替代？（如 `fuzzywuzzy` → `difflib`）
- 已有其他包替代？（如 `rapidfuzz` 已在项目中）
- 新增几个传递依赖？
- 包的活跃度/许可证？

→ 输出替代方案，确认后放行或拦截。

### 2.5 测试覆盖空洞检测——参数 × 输入组合找未覆盖

代码图谱追踪 `function → tests → 输入值`：

```
函数: calculate_tax(amount, rate, category)
已有测试覆盖: amount=100/0, rate=13%/0%, category="goods"/"service"
空洞: amount 负数 / rate>100% / category=None
```

→ 输出空洞列表，QA Agent 自动生成补洞测试。

### 2.6 Agent 输出简洁约束——PonyTail 哲学植入

1. `RULES_BLOCK` 加第 9 条：
```
9. 简洁优先：用最少代码行数实现功能但保持可读性。
   写完检查：有能删掉的行吗？
```

2. 后置检查：Agent 生成 >200 行/文件 → Reviewer Agent 做简洁审计。

### 2.7 CLAUDE.md 自动生成——六图谱扫描 → 项目说明书

Orbit 的六图谱有全部数据源：
- 代码图谱 → 目录结构 + 模块清单
- 配置图谱 → 技术栈 + 环境变量
- 知识图谱 → 领域约束

→ 自动生成初始 CLAUDE.md + 项目迭代中自动更新。

### 2.8 基础代码模板库——Agent 优先调用已验证模板

`knowledge/templates/` 下预置高频代码模式：
- FastAPI 路由模板（校验→service→响应）
- Service 类模板（事务边界/异常映射）
- pytest 模板（fixture/mock 模式）
- 配置模板（.env/docker-compose 片段）

Agent 拿到任务 → `knowledge/engine.py` 搜模板 → 命中则填充参数，未命中再自由生成。

## 三、技术约束

| # | 约束 | 理由 |
|---|------|------|
| 1 | 减熵改动不得破坏现有测试 | 先跑全量回归，改完再跑 |
| 2 | 内部减熵和业务减熵分 PR 提交 | 独立审查，独立回滚 |
| 3 | 每个 PR 包含对应的回归测试更新 | CI 门禁不降 |
| 4 | 删文件前 grep 全仓库确认无隐式引用 | 防断链 |
