## 逐文件审查

### 1. `review/progressive.py` ✅

- **ReviewCheckpoint 模型**: 三列对照清晰——expectation/adr_constraint/actual 渐进填充
- **ProgressiveReviewEngine**: build_from_spec() → fill_code_column() 两阶段 API 简洁
- **边缘处理**: 空 spec.task → 回退 description 检查点；PRD 无结构化 AC → "no structured AC found"
- **性能**: _extract_keywords() 正则提取，无 AST 递归遍历，O(n) 复杂度
- **问题**: _search_in_output() 的关键词匹配过于简单——可能误匹配（如函数名 "run" 匹配到 "runtime"）。不阻塞——渐进式审查定位是"辅助信号"而非"精确判定"

### 2. `testing/strategies/evolutionary_selector.py` ✅

- **EvolutionaryRouter**: MIN_SAMPLES_FOR_ROUTING=3 + WIN_RATE_DELTA_THRESHOLD=0.20 参数合理
- **缓存**: 内存缓存减少 knowledge/ 查询——合理优化
- **持久化**: record_ab_result() 同时写缓存 + knowledge/ ——双写保证
- **降级**: knowledge/ 不可用时缓存仍工作——fail-open
- **问题**: 无。设计成熟，对标 Google MuRS 的反馈驱动自适应

### 3. `testing/triangle.py` ✅

- **TriangleConnector**: 测试+审查双确认 → failure pattern → evolution/ 信号
- **去重**: PATTERN_FREQUENCY_THRESHOLD=2 — 同模式出现 2 次才触发 Prompt 调整（防止噪声）
- **边缘处理**: knowledge/ 不可用时静默降级；evolution/ 注入失败不阻塞主流程
- **问题**: _upsert_pattern() 的相似模式匹配依赖 memory.search() —— 对中文描述可能召回不准。不阻塞——Phase 3 加 BGE 向量相似度改善

### 4. `compose/orchestrator.py`（修改部分） ✅

- **_code_review() 重写**: 状态机转换完整——create_review → record_decision → transition_status
- **异常处理**: ValueError（活跃审查 → 复用）捕获不抛；review_service 不可用 → logger.warning + continue
- **Ponytail 集成**: 并行跑在 try/except 内——失败不影响主流程
- **问题**: 无。改动在设计上是最小侵入——只替换 _code_review() 内部实现，不改变方法签名

### 5. `modes/review/checklist.md`（重写） ✅

- **7 维度覆盖**: 调度器/防幻觉/图谱/沙箱/Agent通信/方案/代码质量，每个 ≥3 条检查项
- **移除**: 财务（金额 Decimal/借贷成对/恒等式/科目余额方向）+ 会计（过账不可逆）
- **Orbit 特有**: orbit.spec hiddenimports / Ponytail 合规 / Goal Judge fail-open
- **问题**: 无

### 6. `testing/orchestrator.py`（修改部分） ✅

- **Ponytail 集成**: asyncio.gather() 并行跑审查+Ponytail——不增加延迟
- **审查强制化**: _ponytail 默认实例化——无 review_service 时自动回退
- **generate_regression_tests()**: severity≥major 筛选 → IntentionDrivenGenerator 生成 → 写入 tests/regression/
- **双向推送**: asyncio.Queue(maxsize=100) + _push_signal_safe() 满时丢弃处理
- **问题**: _push_signal_safe() 满时 get_nowait() 丢最早消息——可能丢重要信号。不阻塞——maxsize=100 在实际场景中很难满（每次 run() 推送 ≤5 条信号）

### 7. `review/models.py`（修改部分） ✅

- **severity 字段**: String(20), default="major" ——合理默认值
- **兼容性**: 旧 ReviewDecision 记录无 severity → migration 设默认值 "major"
- **问题**: 无

### 8. `review/service.py`（修改部分） ✅

- **record_decision()**: severity 作为可选参数（默认 "major"）——向后兼容
- **问题**: 无

### 9. `graph/engines/test_gap_detector.py`（修改部分） ✅

- **rank_by_risk()**: 3 因子加权（centrality 40% / complexity 35% / static_issues 25%）——对标 TestGapRadar 论文
- **归一化**: log2(callers+1)/10 ——保守的归一化方案
- **降级**: code_graph 不可用时 centrality 默认 0.2——合理保守值
- **问题**: 权重默认值基于一篇论文——Orbit 的模块依赖结构与 CQSE 的 Java 系统不同，实际最优权重可能需调。不阻塞——积累 30+ 审查后可按 AC12 自动调参
