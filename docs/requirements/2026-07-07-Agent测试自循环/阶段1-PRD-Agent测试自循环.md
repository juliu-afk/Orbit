## 一、背景与问题

### 1.1 当前状态

Orbit 是多 Agent 软件开发自循环系统。已有 45 个模块、~270 个测试文件、覆盖率 72%（冲刺 80%）。

**但测试全部由人类手动执行**——Agent 生成代码后，人类在阶段 4 手动跑 pytest，手动看结果，手动修 bug。

### 1.2 核心问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | Agent 写完代码不会自己跑测试 | 人类是测试执行的瓶颈 |
| 2 | Agent 不会根据测试失败自己修复 | 每次失败都要人类介入 |
| 3 | Agent 不会为新代码生成测试 | 测试覆盖率依赖人类手写 |
| 4 | 测试失败信息不回流给 Agent | 同样的错误反复犯，不学习 |
| 5 | 每次跑全量测试，不管改动范围 | 45 模块改一行也要全量跑，浪费时间 |
| 6 | 没有 Agent 行为测试 | 不知道 Agent 自身的质量在下降还是提升 |

### 1.3 为什么是现在

- 覆盖率冲刺到 72%——测试基建已有，但 Agent 不会用
- 安全冲刺（PR #224-#226）和 UX 冲刺（PR #227-#228）刚完成——基础设施健康
- 防幻觉 L1-L9、沙箱、合规、审查——验证能力完备，缺编排层

## 二、用户故事

### P0 —— 必须交付

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| US1 | 作为 Orbit 用户，我希望 Agent 生成代码后**自动跑单元测试**，这样我不需要手动执行 pytest | Agent 生成代码 → 立即触发测试执行 → 测试结果出现在聊天流中（摘要卡片） |
| US2 | 作为 Orbit 用户，我希望测试失败时 Agent **自动分析原因并修复**，这样我不需要每次手动 debug | 测试失败 → Agent 读失败信息 → 修复代码 → 重跑，最多 3 轮；3 轮仍失败 → 标记 FAILED 并通知人类 |
| US3 | 作为 Orbit 用户，我希望 Agent 在写代码**之前**先生成测试，这样测试作为代码的规格约束 | Agent 收到编码任务 → 先产出测试代码 → 再产出实现代码（TDD 顺序） |
| US4 | 作为 Orbit 用户，我希望覆盖率不足时 Agent **自动补充测试**，这样不需要我手动追踪未覆盖分支 | 覆盖率 <80% → Agent 分析缺口 → 生成补充测试 → 重跑，直到达标或无法自动补充 |

### P1 —— 应该交付

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| US5 | 作为 Orbit 用户，我希望改动代码后**只跑受影响的测试**，而不是全量跑，这样反馈更快 | 改动 scheduler/ 模块 → 只跑 scheduler/*、checkpoint/*、loop/* 的测试，跳过其他 180+ 无关文件 |
| US6 | 作为 Orbit 用户，我希望 PRD 和技术方案出来后**自动生成测试框架**，这样阶段 3 开始时已有测试骨架可填充 | 阶段1 PRD → 生成 Gherkin 场景骨架；阶段2 技术方案 → 生成契约测试骨架 |
| US7 | 作为 Orbit 用户，我希望 Agent 能**自动选择最优测试策略**（路径敏感 vs 意图驱动），而不是由我指定 | 两种策略都适用时 → Agent 跑 AB 对比 → 记录胜者到 knowledge/ → 后续同类模块直接用胜者 |
| US8 | 作为 Orbit 用户，我希望新增代码**自动检测是否与项目框架冲突**（冗余函数、循环依赖、跨层调用），这样不需要我在审查时逐行比对 | 代码生成后 → code_graph 查同名函数/相似签名/循环依赖/架构越界 → 阻塞项（循环依赖）拦截，警告项标记在摘要卡片 |

## 四、Non-Goals（本次不做）

| # | Non-Goal | 理由 |
|---|---------|------|
| 1 | 不替换现有 CI 管道（GitHub Actions） | CI 是外部门禁，testing/ 是 Agent 内循环，互补不替代 |
| 2 | 不自动合并代码 | 人类审查（阶段 3b）仍然保留 |
| 3 | 不修改现有 45 个模块的行为 | testing/ 是编排层，复用不修改 |
| 4 | 不实现 L3 趋势报告 Dashboard | 已有 Dashboard，不在本次范围 |
| 5 | 不覆盖前端 E2E（Playwright）自动生成 | 复杂度高，Phase 4+ 再做 |
| 6 | 不自动触发混沌实验 | 混沌实验破坏性大，保留人工触发 |

## 五、边缘情况

| # | 场景 | 预期行为 |
|---|------|---------|
| E1 | 沙箱不可用（Docker 未启动） | 回退到本地 pytest 执行 + 警告 |
| E2 | 知识图谱为空（首次运行） | 跳过知识检索，只用代码图谱的 CFG |
| E3 | 代码图谱未构建（目标代码无 Tree-sitter 解析结果） | 回退到纯 LLM 生成（意图驱动），标注"无 CFG 增强" |
| E4 | LLM 生成的测试代码编译失败 | 计数为"测试生成失败" → 重试最多 2 次 → 标记 HUMAN_NEEDED |
| E5 | 测试执行超时（>5 分钟） | 熔断，返回已有结果 + 超时标记 |
| E6 | 并发触发同一个模块的测试 | 加锁（file-level lock），后到排队，不重复跑 |
| E7 | 生成的测试代码包含恶意代码（如 `os.system("rm -rf /")`） | 沙箱执行拦截 + 标记安全告警 |
| E8 | 门禁判定为 PASSED 但人类审查发现 bug | 人类拒绝 → bug 信息入库 knowledge/ → 增强后续门禁 |
| E9 | AB 测试两种策略分数相同 | 选成本更低的（意图驱动 < 路径敏感，因为不需要 CFG 预处理） |
| E10 | coverage.json 不存在（首次生成） | 初始化为空，逐步累积 |
| E11 | 新代码引入循环依赖（A import B, B import A） | 检测到 → 阻塞门禁，不允许通过，返回具体 import 链 |
| E12 | 新代码跨层调用（API 路由直接写 SQL） | 检测到 → ⚠️ 警告标记在摘要卡片，不阻塞，人类审查确认 |

## 六、调度器/防幻觉/图谱影响分析

### 6.1 调度器影响

- `testing/orchestrator.py` 新增异步任务类型 `TestTask`——进入调度器 DAG
- `checkpoint/` 在测试执行前保存检查点（测试失败可回滚到测试前状态）
- 不新增调度器状态——复用现有 `PENDING → RUNNING → DONE/FAILED`

### 6.2 防幻觉层影响

- 测试代码也经过 L1-L9 验证（防 Agent 生成有幻觉的测试）
- L7 沙箱执行：测试代码在 Docker 内跑，宿主机不受影响
- 新增测试特定规则：生成的测试不能 import 被测代码以外的模块（防止测试过度耦合）

### 6.3 图谱影响

- RTS 依赖 `graph/code_graph.py` 的 import 依赖图——不新增图谱，复用
- 测试结果（pass/fail/coverage）存储在 knowledge/ 知识图谱中，关联到对应代码模块
- 不修改 CodeGraph 表结构
