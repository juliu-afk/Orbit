# 阶段1 PRD — 微信集成（iLink Bot API）

> 基于 Marty Cagan 8 章 PRD 模板。通道：腾讯 iLink Bot API + cc-weixin 桥接；交互模型：双向对话，微信是 Orbit 驾驶舱的伴随通道。

## 1. 背景与问题

### 当前状态
Orbit 的用户界面是 Web 驾驶舱（Vue3 + AG-UI），用户必须打开浏览器才能与 Orbit 交互。但实际场景中：
- Agent 任务执行可能需要数分钟到数小时，用户不会一直盯着驾驶舱
- 任务完成/失败/需要审批时，用户希望**即时收到通知**
- 用户在路上/会议中，只有手机，想**快速查看进度或发指令**

### 用户问题
- "任务跑了 2 小时了，不知道完成没有，得打开电脑看？"
- "PR 审查通过了，但我不在电脑前，怎么批准合并？"
- "能不能像 Slack bot 那样，微信里直接指挥 Orbit？"

### 业务价值
- **异步协作**：用户不需要守在驾驶舱前，Orbit 主动推进，关键节点推送微信
- **审批闭环**：HITL（Human-in-the-Loop）审批在微信里完成，不阻塞流水线
- **移动端覆盖**：微信覆盖 Orbit 做不到的移动场景

### 参考报告
`C:\Users\Administrator\Downloads\微信连接.html` —— iLink Bot API + 长消息拆分策略。

## 2. 交互模型

### 模式：双向对话

```
用户 → 微信 → Orbit：命令（启动任务）、审批（同意/拒绝）、问答
Orbit → 微信 → 用户：通知（完成/失败）、审批请求、每日摘要
```

微信**不替代驾驶舱**——复杂操作（看代码 diff、图谱分析、配置管理）仍在驾驶舱完成。微信是**伴随通道**，负责轻量交互。

### 消息方向矩阵

#### A. 用户 → Orbit（微信发指令）

| 场景 | 示例消息 | Orbit 行为 | 优先级 |
|------|---------|-----------|--------|
| 启动任务 | `@Orbit 给 keshen 加导出 PDF 功能` | 创建 Task → 调度 Agent → 回复"已启动 #42" | P0 |
| 查询状态 | `@Orbit #42 怎么样了` | 查 task 状态 → 回复进度（当前阶段/完成百分比/最近日志） | P0 |
| 审批 | `@Orbit 同意 #42 合并` | 触发 HITL 通过 → 继续流水线 → 回复"已批准" | P0 |
| 快速问答 | `@Orbit keshen 试算平衡在哪` | 查代码图谱 → 回复文件路径+行号 | P1 |
| 帮助 | `@Orbit 帮助` | 回复命令列表 | P0 |

#### B. Orbit → 用户（主动推送）

| 触发条件 | 消息内容示例 | 频率 | 优先级 |
|---------|------------|------|--------|
| 任务完成 | `✅ #42 导出 PDF 功能已完成`<br>`PR: github.com/...`<br>`耗时: 12 分钟` | 每任务一次 | P0 |
| 任务失败 | `❌ #42 失败`<br>`原因: 单元测试不通过`<br>`详情: [链接]` | 每失败一次 | P0 |
| 需要审批 | `🔔 #42 审查通过，是否合并？`<br>`回复"同意 #42"或"拒绝 #42"` | 阻塞时推送 | P0 |
| 审批超时提醒 | `⏰ #42 已等待审批 24 小时` | 每天一次 | P1 |
| 每日摘要 | `📊 7月6日 Orbit 日报`<br>`✅ 完成: 3 任务`<br>`🔄 进行中: 2 任务`<br>`❌ 失败: 0` | 每天一次（可配置） | P1 |
| 回滚通知 | `⚠️ #42 触发熔断已回滚到检查点 3` | 即时 | P1 |

#### C. 不推送的内容（太吵）

- 每个 Agent 步骤的中间状态
- 调试日志/token 消耗
- 代码 diff（给链接，驾驶舱看）
- 图谱更新

## 3. 首次连接流程

```
用户（驾驶舱）            Orbit 后端              微信服务器(iLink)
     │                       │                        │
     │ 1. 设置→微信绑定      │                        │
     │──────────────────────→│                        │
     │                       │ 2. 启动 cc-weixin 子进程│
     │                       │   生成 bind_token       │
     │                       │   捕获 QR code          │
     │ 3. 返回 QR code 图片   │                        │
     │←──────────────────────│                        │
     │                       │                        │
     │ 4. 微信小号扫码         │                        │
     │───────────────────────│───────────────────────→│
     │                       │                        │
     │                       │ 5. iLink 回调           │
     │                       │   POST /wechat/callback │
     │                       │←───────────────────────│
     │                       │   (openid + bind_token) │
     │                       │                        │
     │                       │ 6. 验证 token → 写入    │
     │                       │   wechat_bindings 表    │
     │                       │                        │
     │ 7. WS 推送"绑定成功"    │                        │
     │←──────────────────────│                        │
     │                       │ 8. 首条消息推送到微信     │
     │                       │───────────────────────→│
     │                       │   "✅ Orbit 已连接…"    │
```

### 绑定关键设计

| 设计点 | 决策 | 理由 |
|--------|------|------|
| bind_token | uuid4 + secrets.token_urlsafe(16)，一次性，5 分钟过期 | 防截获重用 |
| 绑定关系 | 一个 Orbit 用户 ↔ 一个微信 openid（双向唯一） | 简单，安全 |
| QR 码来源 | MVP：截获 cc-weixin 子进程 stdout → 转为图片；目标：直接调 iLink API | 降低 MVP 集成复杂度 |
| 断开方式 | 驾驶舱点"断开"（主动）；24h 无心跳标记 disconnected（被动） | 覆盖两种场景 |
| 重新绑定 | 驾驶舱点"重新绑定"→清旧记录→生成新 QR 码 | 换号场景 |


## 4. 用户故事

### P0（核心——必须有）

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| P0-1 | 作为用户，我想在驾驶舱扫码绑定微信，以便后续在微信里接收通知 | 设置页面展示 QR 码→微信扫码→驾驶舱显示"已连接"+微信收到确认消息 |
| P0-2 | 作为用户，我想在微信里启动任务，以便不在电脑前也能指挥 Orbit | 发送"@Orbit 启动 [描述]"→Orbit 创建 Task→回复任务 ID |
| P0-3 | 作为用户，我想在微信里查询任务进度，以便了解最新状态 | 发送"@Orbit 查 #42"→回复当前阶段+完成百分比 |
| P0-4 | 作为用户，我想任务完成/失败时微信收到通知，以便不必守着驾驶舱 | 任务状态变为 completed/failed→微信推送通知 |
| P0-5 | 作为用户，我想在微信里审批合并请求，以便不阻塞流水线 | 收到审批请求→回复"同意 #42"→HITL 通过→流水线继续 |

### P1（重要——提升体验）

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| P1-1 | 作为用户，我想在微信里快速问答（查代码位置），以便快速获取信息 | 发送"@Orbit [项目] [符号名] 在哪"→回复文件路径+行号 |
| P1-2 | 作为用户，我想每天收到 Orbit 摘要，以便了解当日工作概况 | 每日定时推送完成/进行中/失败任务数 |
| P1-3 | 作为用户，我想审批超时时收到提醒，以便不遗忘阻塞任务 | 审批等待 >24h → 微信推送提醒 |
| P1-4 | 作为用户，我想在驾驶舱断开微信绑定，以便换号或停用 | 设置页面点"断开"→映射表标记 disconnected→微信端下次发消息收到"未绑定" |

### P2（锦上添花）

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| P2-1 | 作为用户，我想配置推送偏好（开/关每日摘要、静默时段），以便不被非紧急消息打扰 | 设置页推送偏好面板 |
| P2-2 | 作为用户，我想在微信里查看任务最近 5 条日志，以便在不打开驾驶舱的情况下定位问题 | 发送"@Orbit 日志 #42"→回复最近 5 条 |

## 5. 验收标准（汇总）

| # | 标准 | 验证方式 |
|---|------|---------|
| AC-1 | 扫码绑定全流程 < 30 秒（不含微信扫码时间） | 手动测试计时 |
| AC-2 | 微信发指令→Orbit 响应 < 3 秒（不含 Agent 执行时间） | 集成测试 |
| AC-3 | 任务状态变更→微信推送 < 5 秒 | 集成测试 |
| AC-4 | 出站消息频率 ≤ 5 条/分钟（防风控） | 单元测试 token bucket |
| AC-5 | 长消息（>600 汉字）自动按 Unicode 边界拆分，不在中文/Emoji 中间截断 | 单元测试 |
| AC-6 | bind_token 一次性使用 + 5 分钟过期 | 单元测试 |
| AC-7 | Orbit 重启后微信连接自动恢复（无需重新扫码） | 集成测试 |
| AC-8 | 未绑定微信时，微信发消息收到"请先在驾驶舱绑定" | 手动测试 |

## 6. 边缘情况

| # | 场景 | 预期行为 |
|---|------|---------|
| E-1 | 微信扫码但驾驶舱已关闭 | iLink 回调到达→写入绑定→下次驾驶舱打开时显示"已连接"。微信端收到"绑定成功，但驾驶舱当前离线" |
| E-2 | bind_token 已过期（>5 分钟） | 驾驶舱显示"二维码已过期，点击刷新"，自动生成新 token+QR |
| E-3 | 微信取关/删好友 | iLink 无回调。心跳检测 24h 无消息→标记 disconnected→驾驶舱显示"连接已断开" |
| E-4 | cc-weixin 子进程崩溃 | 自动重启子进程（最多 3 次）。3 次均失败→驾驶舱显示"微信服务异常"+微信用户不受影响（原本就连不上） |
| E-5 | 微信发消息但 Orbit 未运行 | iLink 服务器缓存消息。Orbit 启动后拉取离线消息逐条处理 |
| E-6 | 用户发送无法识别的命令 | 回复"未识别的命令，回复'帮助'查看可用命令" |
| E-7 | 并发控制：多个任务同时完成 | 出站队列串行发送，间隔 ≥ 200ms，避免风控 |
| E-8 | 同一个微信号试图绑两个 Orbit 用户 | 拒绝第二次绑定→回复"该微信号已绑定其他 Orbit 账号" |
| E-9 | 用户回复消息超过微信长度限制 | 自动拆分（600 汉字/条），按 UTF-8 字节边界（不是字符边界），不在 Emoji/中文中间截断 |

## 7. 安全考量

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 二维码被他人扫描 | 中 | bind_token 5 分钟过期 + 驾驶舱实时显示扫码状态（谁扫了） |
| 微信号被盗用→操作 Orbit | 高 | 仅限小号使用；驾驶舱可随时断开；敏感操作（合并/删除）需额外确认 |
| bind_token 网络截获 | 低 | HTTPS（iLink API 强制）+ 一次性 token |
| 微信消息内容泄露 | 中 | 不推送敏感信息（API key、密码、Token）；不推送完整代码 diff |
| 微信封号 | 中 | 必须用小号；出站频率控制（token bucket）；遵守 iLink 使用规范 |

## 8. 系统影响分析

### 新增模块
- `src/orbit/integration/wechat/` —— 微信集成模块
  - `channel.py` —— 消息收发主逻辑
  - `bind.py` —— 绑定/解绑管理
  - `router.py` —— 消息意图分类（命令/审批/问答/无效）
  - `outbound.py` —— 出站队列 + token bucket 频率控制

### 现有模块影响

| 模块 | 影响 | 说明 |
|------|------|------|
| `communication/` | 新增 wechat channel 适配器 | 注册到消息总线 |
| `scheduler/` | 挂事件钩子（task_completed/task_failed/hitl_required） | 触发微信推送，不修改调度器状态机逻辑 |
| `api/routes/` | 新增 `/wechat/*` 路由组 | bind/callback/config |
| `core/` | 新增配置项 | wechat_enabled, wechat_daily_summary_time, wechat_quiet_hours |
| `launcher.py` | 启动时判断是否启动 cc-weixin 子进程 | 条件启动，失败不影响主流程 |

### 不影响
- ❌ 防幻觉层（L1-L9）
- ❌ 代码/数据库/配置图谱
- ❌ 沙箱执行
- ❌ 检查点管理

## 10. Non-Goals

- ❌ 不替代驾驶舱（微信只做轻量交互）
- ❌ 不支持微信群聊（仅 1 对 1 私聊）
- ❌ 不支持语音/图片消息
- ❌ 不支持多 Orbit 实例共用一个微信号
- ❌ 不实现企业微信（本期仅个人微信 iLink API）
- ❌ 不在微信里展示代码 diff（给 GitHub 链接）
- ❌ 不通过微信执行沙箱代码
