3.1 为什么需要 9 层 · Why Nine Layers

中文：核心认知——纯静态分析拦截幻觉的理论上限只有 48.5%–77%。要把错误率压到 <3%，必须静态 + 动态结合，用"从快到慢"的分层管道：廉价层拦截大多数简单幻觉，昂贵层（沙箱执行）只处理残余高危幻觉。

English: Static analysis alone caps at 48.5%–77% interception. To reach <3% error, Orbit combines static + dynamic checks in a "cheap-to-expensive" pipeline — cheap layers catch most simple hallucinations; expensive layers (sandbox execution) handle the residual high-risk ones.

分层成本逻辑（docs/开发计划/00-架构总览.md:540-544）：

层组 拦截占比 成本
L1–L2 廉价层~80% 简单幻觉（语法、命名）接近零
L3–L4 中等层~15% 复杂幻觉（语义、逻辑）Token 放大 3–5×，仅关键路径
L5–L7 高成本层~5% 高危幻觉沙箱执行换效果
L8–L9 把关层配置一致性 + 合规时效性零 Token / 异步

3.4 执行管道 · Execution Pipeline

关键：管道执行顺序不等于层编号顺序。实际顺序为"从快到慢"（src/orbit/hallucination/pipeline.py:43-52）：

L1 图谱 → L4 类型 → L3 熵 → L2 追踪 → L6 合约 → L8 配置 → L7 沙箱 → L5 Z3
                                                            ↓
                                                       L9 合规（独立）

两种调用入口：

入口 层组合 用途
validate_quick()L1 + L4 + L3每次代码生成后即时检查
validate_full()L1–L8 全量coding 阶段完成后完整验证

致命层熔断（FATAL_LEVELS）

L1 图谱 和 L7 沙箱 运行时失败时，管道立即停止，不再执行后续层——因为这两层失败意味着代码根本不可信，继续验证无意义（pipeline.py:55-58）。

3.5 设计原理小结 · Design Rationale

  • 顺序服务于早停：便宜且高召回的层放前面，一旦发现致命问题即提前终止，省下昂贵层的开销。
  • 静动互补：L1/L4/L5/L6/L8 静态（确定性、可证明），L2/L3/L7 动态（运行时真值）。静态给出必要条件，动态给出充分验证。
  • L7 沙箱是最后防线：能拦截所有类型幻觉，把错误率从 65% 压到 2%，代价是 Docker 执行开销——因此只在 full 验证跑。
  • L9 独立于 L1–L8：合规是"知识时效"维度，与代码正确性正交，故单独成模块、异步执行、不阻塞主管道。
