因果推理引擎 —— 从相关性监控到反事实根因定位

现状

Monitor 检测症状（目标漂移、重复动作、延迟超标），VIGIL 尝试 heal。但无追问："如果换 Agent B 会怎样？" "导致失败的因果路径是哪条？" 所有告警是相关性信号，不做反事实推理。

必要性

  1. 当前 heal 失效模式：VIGIL 基于规则猜测修复动作（重试 / 降级 / 换模型），无因果模型指导。同一任务失败 3 次后 VIGIL 放弃——因为它不知道"哪个环节错了"。
  2. 审计数据未被充分利用：task_audit_trail 表记录了每个动作、状态转换、验证结果。这是因果发现的理想数据源——当前只用于展示，不用于推理。
  3. GEPA 进化的瓶颈：GEPA 用全局 utility 评分原则好坏，但不区分"原则好但运气差"和"原则差但运气好"。因果推理能解耦这两者。

理论基础

Pearl 的结构因果模型（SCM）+ do-calculus。因果图 $G = (V, E)$ 建模变量关系：

  • $X_1$: Agent 角色选择
  • $X_2$: Prompt 原则集合
  • $X_3$: 工具调用序列
  • $X_4$: 模型层级 (T1/T2/T3)
  • $Y$: 任务成功 (0/1)
  • $Z$: 延迟 (ms)
do(X₁ = architect) → P(Y=1) − P(Y=1 | X₁=developer) ≠ 因果效应（Simpson 悖论未消除前）

在线 Bandit 智能路由 —— 从静态权重到遗憾最小化

现状

router/agent.py 的 RouterAgent 用固定启发式权重选模型层级。权重由 ScoreWeights 预设，不随实际运行结果更新。这是 open-loop 决策。

必要性

  1. 模型性能漂移：LLM provider 的模型行为随时间变化（版本更新、负载波动、限流策略调整）。静态权重在部署时准确，运行一周后不准。
  2. 三层模型路由的浪费：T1（轻量）→ T2（标准）→ T3（最强）的升级逻辑是固定的。当前 "所有任务先 T1，失败升级" 导致失败任务的白白浪费——T1+失败的成本 > T3 直接执行。
  3. CC_SWITCH 是运维手动覆盖——人工判断，不能规模化。

理论基础

多臂 Bandit（Thompson Sampling / UCB）。每个模型层 $k \in \{T1, T2, T3\}$ 是一个"臂"。每任务选择模型 $a_t$，观察奖励 $r_t \in [0,1]$（成功率）和成本 $c_t$（延迟 ms）。目标是最大化期望奖励同时控制遗憾：

Regret(T) = T · μ* − Σₜ μ(aₜ)   （最优模型的总奖励 − 实际总奖励）

Thompson Sampling 保证 $O(\sqrt{kT \ln T})$ 遗憾上界。

class ThompsonRouter:
    """Thompson Sampling 多臂 Bandit 路由器。

    每个模型维护 Beta(α, β) 后验：
      α = 成功次数 + 1（先验）
      β = 失败次数 + 1（先验）
    每次决策：从所有模型的 Beta 后验中采样，选采样值最大者。
    """

    def __init__(self, models: list[str]):
        self._posteriors = {m: {"alpha": 1.0, "beta": 1.0} for m in models}

    def select(self) -> str:
        samples = {m: random.betavariate(p["alpha"], p["beta"])
                   for m, p in self._posteriors.items()}
        return max(samples, key=samples.get)

    def update(self, model: str, success: bool, latency_ms: float):
        p = self._posteriors[model]
        if success:
            p["alpha"] += 1.0
        else:
            p["beta"] += 1.0
        # 成本惩罚：延迟每超 1000ms → 额外 -0.05 reward
        if latency_ms > 3000:
            p["beta"] += 0.5  # soft-penalize slow responses

图谱频谱分析 —— 从语法查询到图的拉普拉斯特征分析

现状

三图谱引擎（代码/数据库/配置）仅做静态查询——符号查找、DDL 解析、配置项 diff。元图谱的 12 种跨图谱关系是离散的点对点边。不做图的连续谱分析。

必要性

  1. 模块边界是主观判断：什么时候一个文件该拆成两个？当前靠人工审查。图拉普拉斯的 Fiedler 向量给出数学上最优的分割。
  2. 变更传播不可预测：改了 payment.py 影响什么？当前用 DEPENDS_ON 边直接遍历。谱半径（邻接矩阵最大特征值）告诉你在最坏情况下改动能传多远。
  3. 跨图谱断裂点不可见：代码改了但配置没跟上——元图谱有 REFERENCES 边但不知道哪些是"本该有但缺失的"。联合谱嵌入能揭示这种跨图谱不一致。

理论基础

L = D − A （拉普拉斯矩阵）
λ₂ = Fiedler 值（代数连通性：模块间耦合强度）
v₂ = Fiedler 向量（节点的自然二分：min-cut 的连续松弛）
ρ(A) = max|λᵢ| （谱半径：变更传播最坏范围）

class SpectralAnalyzer:
    """图拉普拉斯频谱分析器。

    输入: CodeGraph 的邻接矩阵 A (n×n, 稀疏 CSC 格式)
    输出: Fiedler 值、特征向量、谱半径、模块化评分
    """

    def analyze(self, adj_matrix: sp.spmatrix) -> SpectralReport:
        n = adj_matrix.shape[0]
        L = sp.spdiags(adj_matrix.sum(axis=1).A1, 0, n, n) - adj_matrix

        # 用 ARPACK 求 k=5 个最小特征值（运行时 O(kn) 而非 O(n³)）
        eigenvalues, eigenvectors = sp.linalg.eigsh(L, k=5, which='SM')

        # Fiedler 向量: 第二大特征值对应的特征向量
        fiedler_vector = eigenvectors[:, 1]
        # 按符号聚类 → 二分模块
        partition = np.where(fiedler_vector >= 0, 1, 0)

        # 谱半径: 对归一化邻接矩阵求最大特征值
        D_sqrt_inv = sp.spdiags(1.0 / np.sqrt(adj_matrix.sum(axis=1).A1), 0, n, n)
        L_norm = sp.eye(n) - D_sqrt_inv @ adj_matrix @ D_sqrt_inv
        spectral_radius = sp.linalg.eigsh(L_norm, k=1, which='LM', return_eigenvectors=False)[0]

        return SpectralReport(
            algebraic_connectivity=float(eigenvalues[1]),
            fiedler_gap=float(eigenvalues[2] - eigenvalues[1]),
            spectral_radius=float(spectral_radius),
            partition=partition,
            modularity=self._compute_modularity(adj_matrix, partition),
        )

信息瓶颈上下文压缩 —— 从 Token 截断到形式化速率-失真

现状

compression/budget.py 做 Token 计数 + 阈值截断。context/prebuilder.py 按角色选上下文片段。压缩策略是启发式——"保留最重要部分"没有数学定义。

必要性

  1. 当前截断的任意性：128K 窗口到 80% 触发压缩 → 按"最近的上下文"保留。但"最近的" ≠ "最信息量的"。
  2. 5 个 ContextPrebuilder 各自独立截断：无全局最优分配。建模块上下文 20K、建模块上下文 30K → 加起来可能超预算，但各自不知道。
  3. 压缩引入了不可量化的信息损失：如果截断导致 Agent 遗漏了关键依赖文件，失败后无度量归因。

理论基础

Tishby 信息瓶颈（IB）：

min I(X; C) − β I(C; Y)
X = 完整上下文（~128K tokens）
C = 压缩上下文（~32K tokens）
Y = 任务输出质量（成功/失败 + 正确性评分）
β = 压缩 vs 保真度的权衡参数

IB 的最优解是：保留关于输出 Y 的互信息最大化的那部分输入 X。数学上等价于在给定比特预算下最小化失真。

class IBContextCompressor:
    """信息瓶颈上下文压缩器。

    将上下文片段聚类为 K 个"原型"（IB 方法确定 K）。
    给定 Token 预算 B，输出原型子集使得 Σ I(prototype_i; Y) 最大化。
    """

    def compress(self, fragments: list[ContextFragment],
                 budget_tokens: int,
                 task_type: str) -> list[ContextFragment]:
        # 1. 计算每个片段与预期输出的互信息近似
        mi_scores = [self._estimate_mi(f, task_type) for f in fragments]

        # 2. IB 聚类——合并互信息相似的片段
        prototypes = self._ib_cluster(fragments, mi_scores, beta=2.0)

        # 3. 背包问题——在 Token 预算内选最大化 Σ MI 的原型子集
        selected = self._knapsack_select(prototypes, budget_tokens)

        return selected

    def _estimate_mi(self, fragment: ContextFragment, task_type: str) -> float:
        """估算互信息 I(F; Y|task_type)。

        用历史轨迹数据：包含此片段时任务成功率 vs 不包含时的成功率。
        """
        with_frag = self._trajectory_db.success_rate(
            task_type=task_type, context_contains=fragment.hash)
        without_frag = self._trajectory_db.success_rate(
            task_type=task_type, context_contains_not=fragment.hash)
        # MI 近似 = P(success|frag) log ratio
        return with_frag * math.log(max(with_frag, 0.01) / max(without_frag, 0.01))

时序逻辑 Agent 验证 —— 从代码片段检查到全路径形式化

现状

L5 Z3 验证单点代码片段（约束求解）。不验证 Agent 行为随时间推移的正确性。调度器状态机定义了合法转换——但转换序列（路径）是否满足活性/安全性从未被形式化验证。

必要性

  1. 活性 liveness risk：任务是否能最终完成？调度器可能会无限重试（状态循环），这违反了"任务最终结束"的活性属性。
  2. 安全性 safety risk：没有用户确认时，Agent 是否可能静默消耗超过预算？这是一个安全属性——需要在所有执行路径上保证。
  3. 当前测试覆盖不了时序属性：单元测试验证单状态转换，集成测试验证单一 happy path。全路径（含重试/降级/熔断/回滚）的组合爆炸——数学上需要模型检查。

理论基础

线性时序逻辑（LTL）+ 模型检查：

□(budget_overspent → user_confirmed)   （安全性：无确认=不超预算）
◇(task_completed ∨ task_failed)         （活性：任务最终终止）
□(critical_alert → ◇hitl_notified)       （响应性：CRITICAL 最终通知 HITL）

调度器状态机构成 Kripke 结构。用 LTL 模型检查器（Spot / NuSMV）自动验证上述命题在所有可达路径上为真。

class L9TemporalValidator:
    """LTL 时序逻辑验证——Agent 行为路径的形式化检查。

    将调度器状态机编译为 Kripke 结构 →
    用 LTL 公式表达安全/活性属性 →
    Spot 模型检查器验证所有路径。
    """

    SAFETY_PROPERTIES = {
        "no_overspend_without_confirm":
            "G(budget_exhausted -> user_confirmed)",
        "no_unguarded_tool_in_prod":
            "G(destructive_tool_called -> sandbox_active)",
    }
    LIVENESS_PROPERTIES = {
        "task_always_terminates":
            "F(task_completed | task_failed | task_cancelled)",
        "alert_always_responded":
            "G(critical_alert -> F(hitl_notified))",
    }

    async def validate(self, state_machine_spec: dict) -> ValidationResult:
        # 1. 编译 Kripke 结构
        kripke = self._build_kripke(state_machine_spec)

        # 2. 验证每条 LTL 公式
        violations = []
        for name, formula in {**self.SAFETY_PROPERTIES, **self.LIVENESS_PROPERTIES}.items():
            if not self._model_check(kripke, formula):
                # 找到反例路径
                counterexample = self._find_counterexample(kripke, formula)
                violations.append(f"{name}: {formula} violated at: {counterexample}")

        return ValidationResult(
            passed=len(violations) == 0,
            errors=violations,
            level=HallucinationLevel.L9_TEMPORAL,
        )

Shapley 值贡献归因 —— 多 Agent 协作的形式化公平分配

现状

10 个 Agent 角色在任务中协作。GEPA 用全局 utility 评估原则。无从区分"这个 Agent 贡献大"与"任务本来就简单"。SCOPE 的 utility scoring 对所有原则一视同仁——不区分参与 Agent 的贡献差异。

必要性

  1. 团队组成未知最优：GOLDEN_ROUTE 是固定团队模板。"实现新功能" → architect+developer。但对具体任务，architect 可能完全不需要。
  2. 反馈信号分配不清：任务成功 → 所有参与 Agent 的原则 utility 都 +0.1。但一个 Agent 贡献了 80% 的价值应该比贡献 0% 的多得反馈。

理论基础

Shapley 值（Shapley 1953）：

φᵢ(v) = Σ_{S⊆N\\{i\\}} (|S|! · (|N|−|S|−1)!) / |N|! · [v(S∪{i}) − v(S)]
N = 参与任务的 Agent 集合，v(S) = 子集 S 合作完成的价值

φᵢ 是 Agent i 在所有可能联盟中的边际贡献的加权平均。它满足四个公平公理：效率、对称性、空玩家、可加性。

class ShapleyAttribution:
    """Shapley 值归因——计算每个 Agent 对任务成功的边际贡献。

    输入: 已完成任务的审计轨迹（含所有 Agent 子集的结果）
    输出: 每个 Agent 的 Shapley 值及其置信区间
    """

    def attribute(self, task_audit: TaskAuditTrail) -> dict[str, float]:
        agents = list(set(e.agent_role for e in task_audit.entries))
        n = len(agents)

        # 估值函数 v(S): Agent 子集 S 参与时的预期任务质量
        # 用历史轨迹中相同 Agent 组合的平均成功率作为 v(S)
        shapley_values = {}
        for i, agent in enumerate(agents):
            phi = 0.0
            # 遍历所有不含 i 的子集
            for k in range(n):
                for subset in combinations(set(agents) - {agent}, k):
                    s = len(subset)
                    weight = math.factorial(s) * math.factorial(n - s - 1) / math.factorial(n)
                    v_without_i = self._value_of_coalition(subset, task_audit.task_type)
                    v_with_i = self._value_of_coalition(subset | {agent}, task_audit.task_type)
                    phi += weight * (v_with_i - v_without_i)
            shapley_values[agent] = phi

        return shapley_values

PID 自调节控制器 —— 从二元告警到连续反馈

现状

Monitor 输出 CRITICAL → HITL。这是bang-bang 控制——全关或全开。Agent 行为偏差是连续信号（目标漂移缓慢累积），但控制系统是二元的。

必要性

  1. 小的漂移被无视，直到变成 CRITICAL：GoalDriftDetector 检测到漂移但只报 WARNING。Monitor 不处理 WARNING。到 CRITICAL 时已经偏离太远，修复代价大。
  2. 当前反馈链路过长：Agent 执行 → Monitor 观察 → 等到 CRITICAL → HITL → 用户干预 → Agent 调整。对微小的方向偏差，应该在 Agent 内部自纠——不需要 HITL。

理论基础

PID 控制：

u(t) = K_p · e(t) + K_i · ∫₀ᵗ e(s)ds + K_d · de(t)/dt
e(t) = 目标偏离程度（GoalDriftDetector 输出）
u(t) = 矫正信号（注入 Agent 下轮 system prompt 的调整量）
K_p: 对当前偏差的响应 | K_i: 对累积偏差的响应 | K_d: 对趋势的响应

class PIDAgentController:
    """PID 控制器——平滑矫正 Agent 行为偏差。

    Monitor 的连续评分（非二元 CRITICAL）→ PID 计算矫正量 →
    注入 Agent 下轮 system prompt 的 "guidance" 段。
    """

    def __init__(self, Kp: float = 0.5, Ki: float = 0.1, Kd: float = 0.2):
        self.Kp, self.Ki, self.Kd = Kp, Ki, Kd
        self._integral = 0.0
        self._prev_error = 0.0

    def compute_correction(self, drift_score: float,
                           repetition_count: int,
                           latency_ms: float) -> ControlSignal:
        """计算 PID 矫正量。

        Args:
            drift_score: GoalDriftDetector 的漂移分数 (0-1)
            repetition_count: RepetitionDetector 的重复次数
            latency_ms: 当前延迟

        Returns:
            矫正信号——注入 prompt 的 guidance 文本
        """
        # 合成误差信号
        error = 0.5 * drift_score + 0.3 * min(repetition_count / 10, 1.0)
        if latency_ms > 5000:
            error += 0.2

        # PID 计算
        self._integral = max(-3.0, min(3.0, self._integral + error))
        derivative = error - self._prev_error
        correction = self.Kp * error + self.Ki * self._integral + self.Kd * derivative
        self._prev_error = error

        # 映射矫正量到 guidance 文本
        return self._to_guidance(correction, drift_score, repetition_count)

    def _to_guidance(self, correction: float, drift: float, rep: int) -> ControlSignal:
        if correction < 0.3:
            return ControlSignal(level="subtle", text="")
        elif correction < 0.6:
            return ControlSignal(
                level="gentle",
                text=f"[Monitor] Slight deviation detected. Reconfirm alignment with goal."
            )
        elif correction < 0.9:
            return ControlSignal(
                level="firm",
                text=f"[Monitor] Moderate deviation (drift={drift:.2f}, repeats={rep}). "
                     f"Consider fewer tool calls, more thinking."
            )
        else:
            return ControlSignal(
                level="urgent",
                text=f"[Monitor] SEVERE deviation. HALT and reconsider approach. "
                     f"If uncertain, escalate to human."
            )

PAC 泛化保证 —— 原则进化的可证明泛化边界

现状

GEPA 进化 3 代 → utility 分数上升。但不能回答："这个原则泛化到新任务吗？" SCOPE 用 UPGRADE_THRESHOLD=3 控制战术→战略升级——但"3"是拍出来的，无统计基础。

理论基础

PAC 学习（Probably Approximately Correct）：

ε(m, |H|, δ) = √(ln|H| + ln(1/δ)) / (2m)
|H| = 原则库大小 = 候选假设空间大小
m = 评估样本数（历史任务数）
ε = 泛化误差上界
δ = 置信度参数（通常 0.05）

这个公式给出可计算的指导：原则库 100 条 + 评估任务 50 个 → 泛化误差上界约 30%。需要 500 个评估任务 → 上界降为 10%。

落地方案

在 evolution/ 加 pac_bounds.py——GEPA 每代输出泛化误差上界。当界宽时（高方差不可信），自动建议"需要更多评估数据"，停止进化并收集数据。

现状

必要性

  2. GEPA 原则的 utility 不可靠：utility=0.7 是从 30 次评估中估计的。在小样本下这个估计方差极大——但当前无量化。
  3. 无分布假设：PAC 需要 IID 假设。共形预测无分布假设（distribution-free）——不要求 LLM 输出服从任何特定分布。仅要求可交换性（exchangeability）。

理论基础

Γᵅ(x) = {y : p(y | x) > 1 − α}
p(y | x) ≜ |{i: V(xᵢ, yᵢ) ≤ V(x, y)}| / (n+1)
保证：P(Yᵗᵉˢᵗ ∈ Γᵅ(Xᵗᵉˢᵗ)) ≥ 1 − α（有限样本有效）

V 是非一致性评分（nonconformity score）。对代码生成，V = (代码错误数, Token 消耗, 测试失败数) 的加权和。

class ConformalPredictor:
    """Inductive conformal prediction for agent outputs.

    Given a calibration set of (task, code, outcome) triples,
    produce prediction sets with guaranteed coverage.

    Core invariant: P(correct_code ∈ prediction_set) ≥ 1 - α
    """

    def __init__(self, alpha: float = 0.05):
        self.alpha = alpha  # 1 - confidence level
        self._calibration_scores: list[float] = []

    def calibrate(self, calibration_data: list[tuple[str, str, bool]]):
        """Build nonconformity score distribution from calibration set."""
        self._calibration_scores = sorted([
            self._nonconformity(task, code, success)
            for task, code, success in calibration_data
        ])

    def predict(self, task: str, candidates: list[str]) -> list[str]:
        """Return prediction set with ≥ 1-α coverage guarantee."""
        n = len(self._calibration_scores)
        threshold_idx = int((1 - self.alpha) * (n + 1))
        threshold = self._calibration_scores[min(threshold_idx, n - 1)]

        result = []
        for code in candidates:
            score = self._nonconformity(task, code, True)
            if score <= threshold:
                result.append(code)
        return result

    def _nonconformity(self, task: str, code: str, success: bool) -> float:
        """Nonconformity = how unusual this (task, code) pair is."""
        # Lower is better——test failures + lint errors + token count
        base = 0.0 if success else 3.0
        return base + lint_errors(code) * 0.5 + len(code) / 5000

MDP 形式化 —— Agent 循环的最优策略分析

现状

Agent 循环是 ReAct pattern（sense → think → act → observe），固定模式从未被形式化建模。Monitor 监控偏差、RouterAgent 选模型——各自独立。没有统一的数学结构描述：

  • 当前策略离最优有多远？
  • 何时从 T1 升级到 T2 是最优的？
  • 花 Token 做 deeper thinking 的回报率是多少？

必要性

  1. 无最优性基准：无法回答"当前 Agent 表现是差的吗？"因为不知道最优在哪里。MDP 的 Bellman 方程给出理论最优值函数 $V^*(s)$——即使不精确求解，至少给出上界。
  2. Monitor + RouterAgent 是割裂的：Monitor 做状态监控，RouterAgent 做动作选择。MDP 将它们统一为同一公式的不同项。
  3. POMDP 扩展自然包容 Monitor：Agent 不直接观测任务完成状态——Monitor 提供的 drift_score 是部分观测。POMDP 框架给出部分观测下的最优策略形式。

理论基础

Q*(s,a) = R(s,a) + γ Σₛ' P(s'|s,a) · maxₐ' Q*(s',a')
π*(s) = argmaxₐ Q*(s,a) （最优策略）
s = (task_type, graph_snapshot_hash, tool_history[:k], checkpoint_state)
a = (generate_code, call_tool_X, escalate_to_human, wait, backtrack)

POMDP 扩展：Agent 不直接观测状态，而是接收观测 $o \sim O(\cdot|s)$。Monitor 的 drift_score 是观测的一部分。belief state $b(s) = P(s | o_1, a_1, \dots, a_{t-1}, o_t)$ 是充分统计量。

class AgentMDPFormalization:
    """形式化 Agent 循环为有限/无限时域 MDP。

    目的不是精确求解（状态空间太大）——
    是给出最优策略的数学定义，并计算 baseline 与最优的 gap。
    """

    def __init__(self, discount: float = 0.95):
        self.gamma = discount

    def state_features(self, task_ctx: TaskContext) -> np.ndarray:
        """提取状态特征向量——MDP 状态空间的低维投影。"""
        return np.concatenate([
            self._task_type_embedding(task_ctx.task_type),
            self._graph_complexity_features(task_ctx.graph_snapshot),
            self._tool_history_features(task_ctx.tool_history[-10:]),
            self._budget_features(task_ctx.budget_tracker),
        ])

    def reward(self, state: np.ndarray, action: str, next_state: np.ndarray) -> float:
        """奖励函数——任务成功 - 成本。"""
        task_completed = self._is_terminal(next_state)
        if task_completed:
            return 1.0  # 成功完成
        if self._is_failed(next_state):
            return -1.0  # 失败
        # 中间步骤：负成本
        token_cost = self._token_cost(action) * 0.001
        latency_penalty = self._latency(action) * 0.0001
        return -token_cost - latency_penalty

    def compute_bellman_gap(self, policy_fn, value_fn,
                             samples: list[Transition]) -> float:
        """Bellman 误差 = Σ|Q(s,a; θ) − (r + γ maxₐ' Q(s',a'))|。
        小 → 策略接近最优。大 → 策略有提升空间。
        """
        gap = 0.0
        for s, a, r, s_next in samples:
            td_target = r + self.gamma * max(value_fn(s_next, a2)
                                              for a2 in self._actions(s_next))
            td_error = abs(value_fn(s, a) - td_target)
            gap += td_error
        return gap / len(samples)

分离逻辑 —— 生成代码的堆不变式验证

现状

必要性

  1. Agent 常犯别名错误：Agent 生成的代码大量使用可变对象（list/dict/set），经常无意间共享引用。L4 类型系统对此完全盲视。
  2. 沙箱执行发现了运行时错误，但不解释原因：L7 说"输出不对"。分离逻辑说"你违反了第 12 行的 frame 条件——该函数篡改了不应拥有的堆区域"。
  3. 模块化验证：分离逻辑的 frame rule 允许独立验证每个函数——调用者不需要检查被调函数的内部实现。这正是模块化代码生成所需的形式化保证。

理论基础

{P} C {Q}   ⇒   {P * R} C {Q * R}   （Frame Rule：R 未被 C 修改）
P * Q = P 和 Q 拥有不相交的堆区域（分离合取）
emp = 堆为空 | x ↦ v = x 指向值 v
关键：{x ↦ _} x := 5 {x ↦ 5} 且允许平行组合 {x ↦ _ || y ↦ _} x:=5 || y:=6 {x ↦ 5 * y ↦ 6}

变点检测 —— LLM 行为漂移的统计检测

现状

方向 2（Bandit 路由）用 Thompson Sampling 适应慢漂移。但 LLM provider 的变更是急变——一次模型更新从 90% 成功率跌到 50%，Bandit 需要 ~50 次失败才完全适应。这 50 次失败是不可接受的。

必要性

  1. Bandit 的滞后性：Thompson Sampling 适应速率 $\propto 1/t$（后验逐渐压缩）。第 1 次急变后采样仍然偏向旧最优模型，直到 Beta 后验更新足够多次。
  2. 无自动检测：当前靠人工发现——"今天 GPT 4.1 结果不对"——然后手动改权重或 CC_SWITCH。缺乏自动统计检测。
  3. 变点后模型行为完全不同：这不是微调——是 provider 替换了底层模型。成功率从 90% → 50% 在变点前后是两个不同分布。

理论基础

CUSUM: S₀ = 0, Sₜ = max(0, Sₜ₋₁ + log(p₁(xₜ)/p₀(xₜ)))
检测: Sₜ > h（阈值）→ 变点触发
p₀ = 变点前分布（历史基线），p₁ = 备择分布（假设变点后）
ARL₀（虚假告警间隔）≈ eʰ，ARL₁（检测延迟）≈ h/μ₁
典型参数：h = 5 → ARL₀ ≈ 148，对 50% 到 90% 的 shift 检测延迟 ~15 次

class CUSUMDriftDetector:
    """CUSUM 变点检测——监控模型输出分布是否发生急变。

    监控 (latency, success_rate, output_length) 三元组的联合分布。
    每个 metric 独立 CUSUM，任一触发 → 该模型标为 drifted。
    """

    def __init__(self, threshold: float = 5.0, window_size: int = 100):
        self.h = threshold
        self.window = window_size
        self._baselines: dict[str, dict] = {}   # model → {metric → (μ, σ)}
        self._cusum_scores: dict[str, float] = {}

    def update(self, model: str, latency_ms: float, success: bool, output_len: int):
        if model not in self._baselines:
            self._baselines[model] = self._init_baseline(model)
            return None  # 尚未建立基线

        baseline = self._baselines[model]
        # 标准化 anomaly score——成功率的二项 B(1,p)、延迟的对数正态
        metrics = {
            "success": abs(success - baseline["success_rate"]),
            "latency": abs(math.log(max(latency_ms, 1)) - baseline["latency_log_mean"]),
            "output_len": abs(output_len - baseline["output_len_mean"])
                     / max(baseline["output_len_std"], 1),
        }
        # 联合 anomaly score（Fisher 方法：−2 Σ log p-value)
        combined = -2 * sum(math.log(max(1 - m, 0.001)) for m in metrics.values())

        cusum = self._cusum_scores.get(model, 0.0)
        cusum = max(0, cusum + combined - 1.0)  # drift=1 为正常噪声
        self._cusum_scores[model] = cusum

        if cusum > self.h:
            self._cusum_scores[model] = 0.0  # 重置——检测到变点
            return DriftAlert(model=model, score=cusum, metrics=metrics)

        # 滑动窗口更新基线（适应慢漂移）
        self._rolling_update(model, success, latency_ms, output_len)
        return None

信息几何 —— 进化方向的最优度量

现状

GEPA 变异器按 utility 排序最优原则——在欧几里得距离下比较。但原则是概率分布上的点（每条原则改变 Agent 行为分布的参数）——不是欧几里得空间中的点。欧几里得度量在分布空间中没有意义。

必要性

  1. 原则空间的错误度量：两条相似的原则（"先收集信息"和"收集信息后再决策"）在文本编辑距离（Levenshtein）上远，但在语义上极近。

  3. GEPA 交叉算子在错误空间中混合：合并两条高效用原则时，应该在 Fisher-Rao 距离下找中点（KL 散度的对称化版本），而非文本拼接。

理论基础

F(θ) = E[∇ log p(x|θ) · ∇ log p(x|θ)ᵀ] （Fisher 信息矩阵）
θₜ₊₁ = θₜ − η F(θₜ)⁻¹ ∇L(θₜ) （自然梯度下降）
dist_FR(θ₁, θ₂) = ∫ √(dθᵀ F(θ) dθ) （Fisher-Rao 距离）
关键：F(θ) 是参数空间到统计流形的黎曼度量

现状

拜占庭容错 —— 恶意/故障 Agent 的形式化容忍

现状

Ensemble（方向 9）用 3 模型并行投票。但 3 个模型中 1 个故障时，需要多数=2 票才能正确。如果 2 个故障呢？3 个全同意一个错误答案呢？无形式化故障模型。

必要性

  1. LLM 的故障模式不透明：1 个模型可能因幻觉输出破坏性建议（删表、注入后门）。当前 Ensemble 假设多数是对的——但这个假设无依据。
  2. 拜占庭将军问题：$n$ 个 Agent 中 $f$ 个故障/恶意。Lamport (1982) 定理——需要 $n > 3f$ 保证正确共识。3 Agent = 最多容忍 0 个拜占庭故障。当前 Ensemble 没有容忍度。
  3. 关键操作的保护缺口：DROP TABLE、`rm -rf`、`eval()`、`exec()`——这些操作在沙箱之外执行时有真实破坏性。对这类操作应走 BFT 共识。

理论基础

n > 3f 必要且充分（synchronous BFT, oral messages）
n = 3: 容忍 f = 0（只能容忍良性故障，不能容忍拜占庭）
n = 4: 容忍 f = 1
n = 7: 容忍 f = 2
对 Ensemble：n 个独立模型并行执行 → 投票 → 仅当 ≥ ⌊n/2⌋+1 一致时才执行

落地方案

在 goal/ensemble.py 加 bft.py——对关键操作（DROP/TRUNCATE/DELETE/eval/exec/rm）走 BFT 投票。不是简单多数——是 $2f+1$ 出块。
