2.2 逐模块成熟度评估

模块成熟度自评关键缺口
review/service.py 高 状态机完整，去重保护，聚合摘要。对应业界 CRUD 最佳实践。 缺少批量决策接口（一次提交多个 hunk 判定需逐条调 API）
review/models.py 高 SA 2.0 Mapped 风格，三实体设计清晰，独立 Base 隔离。 ReviewDecision 缺少 severity 字段——只能 approved/rejected/comment，无法区分致命/严重/一般
review/ponytail.py 中 5 类检测实用，正则+AST 组合合理，假阳性排除细致。 仅 Python；缺少 JS/TS 支持；dead_code 和 premature_generalization 检测标了 TODO 未实现
modes/review/ 中 交互协议设计完整，checklist 6 维度覆盖广。 checklist.md 含财务/会计维度——这些是恪现项目专用，Orbit 自身不需要；需项目自适应
compose/skills/review.md 高 4 维度加权 + 精确输出格式 + 3 原则。对标 Google Code Review 标准。 缺少与 testing/ 的联动指令——审查 Agent 不知道测试 Agent 的存在
compose/orchestrator.py 中 两道门禁 + 拓扑排序 + 重试。spec-review 和 code-review 分阶段执行。 code_review 只检查错误残留和输出长度——无深度审查集成（不调 review/ 模块，不跑 ponytail）
testing/orchestrator.py 高 已实现 CrossReport + 并行审查调用 + 修复循环 + 框架适配检查。 审查调用是可选的（if self._review）——需强制执行

2.3 关键发现：审查链路存在断点

⚠️ 断点 1 — ComposeOrchestrator._code_review() 从未调用 review/service.py。
它自己做了一个简化版检查（输出长度 + 错误残留），跳过了完整的审查引擎。
影响：审查状态机（pending→approved→merged）空转——没有数据写入 ReviewDecision 表。
⚠️ 断点 2 — modes/review/checklist.md 包含财务/会计维度，但 Orbit 本身不处理金额。
审查清单是恪现项目（Code-Insight-Financial）的模板复制过来的，没有针对 Orbit 的领域特化。
影响：审查 Agent 检查了无关维度，漏掉了 Orbit 特有的调度器/防幻觉/图谱维度。
⚠️ 断点 3 — ponytail.py 的审查结果没有进入 CrossReport。
Ponytail 作为独立的过度工程检测器运行，输出日志但不反馈给 testing/ 的合并报告。
影响：人类看到 CrossReport 的测试+审查结论，但看不到 Ponytail 的"这个类只有一个方法"警告。

3. 测试-审查联动：已有能力 vs 理想状态

3.1 什么是"联动"——定义

定义：测试和审查不是两条独立的串行流水线，而是在同一时刻、同一代码点上同时给出结论，矛盾处自动标记，互补信息双向推送，合并为一份人类只需看分歧点的统一报告。

3.2 联动成熟度模型（5 级）

级别名称描述Orbit 现状已有证据
L0无联动测试和审查完全独立，人读两份报告——
L1并行执行测试和审查同时跑，但不交换信息✅ 已实现orchestrator.py:482 asyncio.gather()
L2结果合并两份结果合并为一份报告，人类只看一张卡片✅ 已实现reporter.py:121 CrossReport
L3交叉验证同一代码点的测试结论 vs 审查结论对比，分歧自动标注⚠️ 部分reporter.py:165 divergent_points；但 Ponytail 未进入 CrossReport
L4双向推送测试→审查推送覆盖率缺口；审查→测试推送缺失测试点；互相增强⚠️ 设计已有PRD 5.5b 信息交换协议设计了但未实现
L5闭环回灌审查发现→自动生成 regression 测试→永久保护；失败模式→知识图谱→未来预防❌ 未实现仅 US9 描述了方向，无显式 AC

3.3 速度与质量收益的量化估算

联动级别预计速度提升预计质量提升估算依据
L0→L1 (并行)~40% 时间节省无变化测试和审查不再串行等待
L1→L2 (合并)~25% 时间节省+5% 缺陷发现率减少上下文切换；人类读一张卡片 vs 两份报告
L2→L3 (交叉验证)~60% 时间节省+15% 缺陷发现率人类只看 ~20% 的分歧点；假绿灯被消除
L3→L4 (双向推送)~10% 时间节省+10% 缺陷发现率测试找到盲区→审查深挖根因（Meta JiT 4× 基准）
L4→L5 (闭环回灌)~5% 时间节省+20% 长期质量同类错误不再重现（Google MuRS 反馈 ML 基准）
L0→L5 总计~75% 人类审查时间节省~50% 缺陷发现率提升
✅ 速度结论：联动提速的主要来源不是"审得快"，而是审得少——共识部分自动通过，人类只处理分歧。预计 L3 级别的交叉验证可节省 ~60% 人类审查时间。
✅ 质量结论：联动的质量增益来自消除假绿灯（测试通过但代码有问题）和消除盲区（审查能看到测试看不到的架构问题）。

3.4 互补盲区分析

核心洞察：测试和审查的盲区不重叠——两者的视角互补，天然适合联动。

测试 Agent 看得到审查 Agent 看得到
擅长 运行时正确性 · 边界覆盖率 · 变异体杀伤 · 循环依赖检测 · 回归保护 架构合理性 · 命名约定 · 重复造轮子 · 过早/过度抽象 · 安全漏洞 · 硬编码密钥 · 方案偏离 PRD
盲区 架构是否合理 · 命名是否好 · 模式是否一致 · 是否与已有代码重复 运行时是否正确 · 边界 case 是否全覆盖 · 变异体是否被杀 · 循环依赖（静态读代码不易发现）

4.2 学术界共识（2024-2026 趋势）

六大共识，全部直接支持 Orbit 的设计方向：

  1. 意图优于代码：告诉 LLM "测什么"比让它看代码猜更有效（IntUT +94% 分支覆盖）
  2. 反馈循环比单轮生成好：编译→测试→变异→回归 的迭代循环显著提升代码+测试质量（LLMLOOP 5 循环）
  3. diff-scoped 而非 full-scope：只分析变更部分可独立于代码库规模扩展（Google MuRS：全量→diff 缩放）
  4. 自动化排序 > 人工筛选：14 指标的风险排序不劣于人类专家（TGA Radar）
  5. 变异测试是测试质量的唯一可靠度量：覆盖率告诉你测试"跑过"哪些行，变异测试告诉你测试"真正测了"什么（GEM / MuRS / Tautest 均基于此）
  6. 人类只在矛盾处介入：测试和审查一致的 80% 自动通过，不一致的 20% 才升级人类（Meta JiT / CrossReport）

7. 渐进式审查：概念验证与可行性

7.1 什么是渐进式审查

核心思想：审查不是代码出现后才开始的——它在 PRD 阶段就开始积累"预期清单"。PRD → 提取验收标准 → ADR → 补充架构约束 → 代码 → 对比预期 vs 实际。

一句话：提前把"代码应该长什么样"写下来，代码出来后只用填空，不用重新理解。

7.2 Orbit 已有的渐进式基础设施

阶段现有能力对应的预期类型成熟度
PRD intention.py:extract_from_prd() — 从验收标准文本提取 Gherkin Scenario 行为预期 — "代码必须能做 X" 已有，待连线
ADR / 技术方案 intention.py:extract_contract_tests() — 从 API 设计文本生成契约测试 接口预期 — "端点必须返回 shape Y" 已有，待连线
Spec Task.signature + behavior + tests (RUNE 启发) — 函数签名 + WHEN/THEN + 验收断言 实现预期 — "函数签名必须是 Z" 已实现
代码 _code_review() + CrossReport 实际 vs 预期对比 已实现

7.3 缺的关键一块：跨阶段对比引擎

现状：四个阶段各有一套"预期提取"机制，但它们不互通。
PRD 阶段提取的 Gherkin 场景保存在 docs/requirements/.../阶段1-测试骨架.feature 文件里。
技术方案阶段提取的契约测试保存在 tests/integration/test_*_skeleton_*.py 文件里。
代码阶段的审查引擎 (review/service.py) 不知道这些文件存在。

缺的：一个 ReviewCheckpoint 模型，在 PRD 阶段建左列、ADR 阶段补中列、代码阶段填右列，三列同框对比。

7.4 推荐模型

class ReviewCheckpoint(BaseModel): """跨阶段审查检查点——从 PRD 到代码的三列对照""" id: str source_phase: str # "prd" | "adr" | "spec" source_ref: str # 出自哪个文档的哪条（可追溯） # 三列——渐进填充 expectation: str # 左列：PRD 阶段填充——代码必须实现什么 adr_constraint: str # 中列：ADR 阶段填充——架构约束 actual: str # 右列：代码阶段填充——实际文件:行号 + 判定 # 判定（代码阶段自动填） verdict: str # "match" | "deviation" | "partial" | "not_found" evidence: str # 证据（测试结果 / 审查发现 / 代码片段）

7.5 与 RUNE 的关系

RUNE (Task.signature + behavior + tests) 是渐进式审查的微观实现。

区别在于范围：RUNE 在一个 Task 内生效（"这个函数的签名必须是什么"），渐进式审查跨阶段生效（"这个 PRD 的验收标准 3 在代码里有没有对应的实现"）。

扩展路径：让 Spec.tasks 自动展开为 ReviewCheckpoint 列表——每个 Task 的 signature/behavior/tests 成为一条检查点，然后 PRD 和 ADR 的预期也追加到同一张表。代码阶段一次性对照。

9. 结论与建议

9.1 核心结论

  1. Orbit 的代码审查架构是 4 层体系，设计完整但有 3 个断点。审查引擎 (review/) 和编排器 (compose/) 之间有连线缺口，审查清单未针对 Orbit 领域特化，Ponytail 过度工程检测的结果未进入 CrossReport。
  2. 测试和审查联动能同时提速（~60% 人类审查时间节省）和提质（~50% 缺陷发现率提升）。核心机制不是"审得快"而是"审得少"——共识自动通过，人类只看分歧点。学术界和工业界均验证了这个方向。
  3. Agent 测试自循环 PRD 与联动研究的重叠度 ~85%。12 条 AC 中 9 条有直接学术界或工业界支持。AB 自动选优（AC11）和契约骨架生成（AC10）是真正的差异化。唯一缺口是审查→测试回灌闭环（建议新增 AC14）。
  4. 渐进式审查（PRD→ADR→代码预填充）在 Orbit 有现成基础设施。RUNE 模式 (Task.signature + behavior + tests) 是其微观实现。缺的是一张跨阶段对照表——ReviewCheckpoint 模型（约 30 行代码）。
  5. 6 篇学术论文和 4 家工业公司的实践全部支持 Orbit 的设计方向。没有发现任何证据表明当前方向有误。AB 自动选优是真正的无人区。

P0 执行 S1-S4 短期优化——修复 3 个断点 当前系统有功能但链路不完整——修断点比建新功能优先 2-3 天
P0 在 Agent 测试自循环 PRD 中加入 AC14（审查→测试回灌闭环） 这是唯一一个有强学术界和工业界支持但 PRD 里没有显式验收标准的能力 30 分钟（加一段文字）

9.3 一句话

Orbit 的测试-审查联动设计在学术上和工业上都有充分依据。
当前最该做的事不是设计新能力——是把已经设计和已经验证的能力连起来。3 个断点 + 1 个 AC14 + 1 张 ReviewCheckpoint 表。
做完这些，Orbit 的测试审查体系将达到业界领先水平。AB 自动选优是真正的无人区，值得长期投入。
