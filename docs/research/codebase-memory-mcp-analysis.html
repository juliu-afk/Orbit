📐 定位修正

Orbit 代码图谱当前实现是 1,882 行多引擎图谱体系——不是"~300 行 ast"。

但与设计文档中的完全体（Tree-sitter 多语言 + Def-Use + 运行时追踪 + 统一查询接口）仍有差距。单语言 Python ast 解析是当前最大的技术债务——这一点设计文档 Step 3.1 的 ADR 已经明确指出："若需支持多语言，需切换至 Tree-sitter。但 MVP 阶段仅 Python，已规划后续扩展。"

下面的 CBM 分析基于"Orbit 完全体 vs CBM 当前"的对比，而非"Orbit MVP vs CBM"。

Part A — codebase-memory-mcp 调研

A.1 项目概览

纯 C 静态二进制，零依赖。Tree-sitter AST 解析 158 种语言 → SQLite 知识图谱 → 14 个 MCP 工具。单一命令安装，覆盖 11 个编程 Agent。MIT 许可证。

🏗️ 核心架构

语言纯 C（33M 代码），零依赖
解析Tree-sitter 158 语言（grammar 编译进二进制）

可视化内置 3D 图谱 UI（localhost:9749）

A.3 维度对决（Orbit 完全体 vs CBM）

🏆 CBM 胜

  • 语言覆盖：158 vs 1（差距最大）
  • 解析引擎：C 实现 Tree-sitter vs 文档规划
  • 查询语言：openCypher vs 命令式 Python
  • 语义搜索：Nomic 768d 代码嵌入
  • 类型推断：混合 LSP 11 语言
  • 索引速度：C 内存管道 vs Python async
  • MCP 工具数：14 vs 4
  • 团队共享：zstd 图谱产物

🏆 Orbit 胜

  • 数据库图谱：DDL+MVCC 反射，CBM 无
  • 配置图谱：.env/Nginx/docker-compose
  • 元图谱：12 种跨图关系，CBM 无
  • 知识图谱：财务会计领域本体+BGE
  • 审计图谱：第四维度决策追溯
  • 防幻觉集成：L1-L9 纵深防御
  • 调度器联动：图谱变更→任务失效→重调度
  • 三级分级：preview/full/oversized

🔄 各有所长

  • IaC：CBM 有 K8s，Orbit 有 Nginx/docker-compose——不重叠
  • 分发：CBM 多平台二进制，Orbit PyInstaller+Tauri
  • 边丰富度：CBM 15+ 代码边，Orbit 统一边表+12 跨图关系——不同维度

A.4 CBM 核心判断

🔀 代码图谱引擎：CBM 是最佳实现参考，但不是替代方案

Orbit 的差异化在于多图谱融合（代码+DB+配置+知识+审计+元图谱 12 种跨图关系）+ 防幻觉纵深防御 + 调度器联动。CBM 只做代码图谱——但它做到了极致。

策略：接 CBM 做 MCP sidecar 获得即时 158 语言代码智能。Orbit 按设计文档路线图推进 Tree-sitter 迁移 + Def-Use + 运行时追踪——完成完全体。

Part B — Open Knowledge Format (OKF) 调研

B.1 是什么

Open Knowledge Format (OKF) 是 Google Cloud 于 2026-06-13 发布的厂商中立、开放标准（v0.1）。定义了一套用纯 Markdown + YAML frontmatter表示组织知识的格式。零数据库、零专有 SDK、零锁定。

B.3 规范核心（v0.1）

要素规范
Bundle 一个目录树。可发布为 git repo / tarball / zip / 大仓库子目录。
Concept 一个 .md 文件 = 一个知识单元。Concept ID = 文件路径去掉 .md。
Frontmatter YAML，仅 type 必需。推荐：title / description / resource / tags / timestamp。可任意扩展。
Body 标准 Markdown。约定段：# Schema / # Examples / # Citations。
链接 = 边 Markdown 链接自动成为有向边。推荐绝对路径 /tables/orders.md——文件移动后仍稳定。断链是警告，不是错误。
index.md 目录清单——支持渐进披露（Agent 先看 index，再决定打开哪个文件）。无 frontmatter。
log.md 更新历史——ISO 8601 日期分组，最新在前。
符合性 宽松消费：未知 type → 不拒绝；缺可选字段 → 不拒绝；断链 → 不拒绝。故意设计——部分生成的 bundle 仍可用。

B.6 OKF vs MCP vs llms.txt

llms.txt

站点根目录路标文件——指向 AI 应优先阅读的页面。轻量入口。

OKF

一捆 Markdown 知识文件——交付完整内容库。多为内部/团队知识。

MCP

连接协议——让 Agent 调用工具获取实时数据。相当于"图书管理员"。

三个标准叠加使用，不互斥。

Part C — 三方对照

C.1 定位三角

🧠 CBM

代码智能引擎

高性能、深解析、广覆盖。158 语言 AST → SQLite 知识图谱 → MCP 工具。只管代码。

核心价值：速度 + 精度 + Token 节省

不做的：领域知识、跨图谱融合、Agent 编排

📋 OKF

知识交换标准

厂商中立、人机双读。Markdown + YAML → 目录 bundle → 链接图。只管知识表示和交换。

核心价值：可移植性 + 可版本化 + 免锁定

不做的：代码解析、实时查询、Agent 工具

🔄 Orbit

多 Agent 自循环系统

五图谱 + 9 层防幻觉 + 调度器 + 多 Agent 编排。代码+DB+配置+知识+元图谱全覆盖。

核心价值：图谱融合 + 防幻觉纵深 + Agent 闭环

不做的：通用代码智能引擎、知识交换标准

C.4 Orbit 如何用 OKF

场景 1：领域知识图谱用 OKF 格式存储

Orbit 当前的财务会计领域知识（knowledge/engine.py + BGE 向量搜索）可以输出为 OKF bundle。好处：

  • 人类会计可以直接用 Obsidian/VS Code 编辑知识——不需要学 SQL
  • 版本控制：git diff 看知识变更，blame 看谁改的
  • 可移植：知识 bundle 可以独立于 Orbit 分发——给审计师、给客户

场景 2：用 okf-skills 连接器生成 DB 图谱 OKF 视图

Orbit 已有 db_graph.py，但生成的是内部 SQLite 格式。用 okf-skills 连接器（SQLite/MySQL/PostgreSQL）可自动生成 OKF 格式的数据库文档：

  • 每张表 → 一个 OKF concept（frontmatter: type=Table, resource=DDL 位置）
  • Schema 段 → 自动从 DDL 反射生成
  • 外键 → Markdown 链接到被引用表
  • 非技术人员可读——不学 SQLAlchemy 也能理解数据模型

场景 3：OKF 作为 Orbit 的"知识交换 API"

Orbit 内部图谱 → 导出为 OKF bundle → 给外部 Agent（Claude Code / Codex）直接消费。OKF bundle 放在项目根目录 .orbit/knowledge/ 下，任何 MCP 客户端都能通过 okf-mcp 读取——不依赖 Orbit 进程。

  • 不用 Python 重写 CBM：CBM 33M C 代码的实现深度（混合 LSP / Nomic 嵌入 / openCypher 引擎）用 Python 复制不划算。接 MCP 用即可。
  • 不替换 Orbit 内部图谱存储为 OKF：OKF 是交换格式，不是查询引擎。Orbit 图谱查询需要毫秒级性能——SQLite 是正确选择。OKF 是导出/导入层。
  • 不引入 Neo4j：设计文档早期考虑过 Neo4j（元图谱 ADR），但最终落地为 SQLite。保持 SQLite——CBM 也验证了这个选择的正确性。

核心判断

CBM = 完成了 Orbit 代码图谱引擎的 10 倍版本但用 C 语言编写。接它，不要重写它。
OKF = Google 刚发布的"知识的 Markdown"。用它做交换层，不要替换内部存储。
Orbit = 唯一一个把代码+DB+配置+知识+审计五图谱融合 + 9 层防幻觉 + Agent 闭环的系统。这是 CBM 和 OKF 都做不到的。
一句话：CBM 做代码大脑，OKF 做知识语言，Orbit 做中枢神经——三个各自最强，接起来才是完整系统记忆。
