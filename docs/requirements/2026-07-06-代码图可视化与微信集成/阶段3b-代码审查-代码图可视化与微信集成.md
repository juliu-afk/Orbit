## 2. 发现问题

### 2.1 ⚠️ 一般：outbound.py `_split_message` surrogate pair 检测死代码

**文件**：`src/orbit/integration/wechat/outbound.py:148-153`  
**问题**：Python 3 `str` 以 Unicode codepoint 为单位，`len()` 和索引均基于 codepoint，内置的 surrogate pair 不会出现在 Python `str` 中（那是 UTF-16/JavaScript 的概念）。检测 `0xD800 <= codepoint <= 0xDBFF` 的代码**永远不会为 True**，是死代码。

**影响**：无功能影响——`len(content) <= max_chars` 判断已正确按 codepoint 计算。死代码不产生错误，仅误导维护者。

**修复**：删除死代码，简化 split 逻辑为直接 `[content[i:i+max_chars] for i in range(0, len(content), max_chars)]`。

**严重程度**：一般——不修也不影响功能，但降低代码可读性。

### 2.2 ⚠️ 一般：channel.py QR 码生成为占位实现

**文件**：`src/orbit/integration/wechat/channel.py:94-100`  
**问题**：`get_qrcode_data_url()` 当前生成的是占位 QR 码（包含 `https://orbit-bind.example.com/...`），非 cc-weixin 真实登录二维码。实际 QR 码需等 cc-weixin 安装实测后接入。

**影响**：扫码后无法完成 iLink 绑定——但 block 在 cc-weixin 未安装/未启动时已返回 WECHAT_002。

**严重程度**：一般——MVP 预期行为。标记为 P0 后续任务。

### 2.3 ⚠️ 一般：router.py 路由模式「查」存在二义性

**文件**：`src/orbit/integration/wechat/router.py:22,28`  
**问题**：`查` 同时出现在 query_task（第 22 行）和 qa（第 28 行）模式中。依赖路由表顺序区分：query_task 要求 `#?(\d+)`，qa 不要求。当用户输入 `@Orbit 查 某种功能` 时，query_task 因 `\d+` 不匹配而跳过，qa 命中。这种行为依赖顺序，不够显式。

**建议**：qa 模式中移除 `查`，仅保留 `找|where is`。`查` 在 query_task 已覆盖数字 ID 场景。

**严重程度**：一般——当前行为正确但脆弱，加新模式时可能无意破坏顺序。
