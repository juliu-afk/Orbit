🔬 Orbit vs OpenCode — 架构对比分析报告

2026-07-15 | 基于 OpenCode (anomalyco/opencode) 源码 + Orbit v0.50.0 代码审计

核心结论

OpenCode 在 会话生命周期管理、消息持久化、上下文压缩触发机制、Agent 循环鲁棒性 四个基础维度全面领先 Orbit。Orbit 已具备概念层对应物 (SessionRegistry, ContextCompressor, TaskRunner)，但 接线不完整、触发条件缺失、副作用未被持久化——导致用户实际体验中出现"鬼打墙""回复不存""跨 Agent 失忆"等硬伤。

缺口数: 12 项 P0 (阻塞使用) + 8 项 P1 (影响流畅度)

1. 会话生命周期与消息持久化

维度OpenCodeOrbit v0.50.0差距
消息存储每轮 user + assistant 全部落库 (SQLite via Drizzle)chat 意图路径 ✅；skill/chain ❌ (v0.50.0 刚修)⚠️ 已修复但需验证
回复持久化assistant 消息在 流式生成前 即创建占位记录，流式过程逐 part 更新_send 之后才 add_message——如果 _send 失败或连接断开，回复永久丢失🔴 P0
会话恢复任意 session_id 可恢复全部历史——message-v2 完整序列化SessionRegistry.get_messages() 可恢复，但仅 50 条上限🟡 P1
消息过滤filterCompactedEffect——压缩覆盖的消息自动过滤，不物理删除无——get_messages 返回全量，压缩后旧消息仍存在🟡 P1
并发控制会话级锁 (lock.ts)——同一 session 同时仅一个 loop 在跑无会话级锁——多 WS 消息可并发命中同一 session🔴 P0

2. Agent 主循环 (RunLoop)

维度OpenCode prompt.ts (runLoop, 17 步)Orbit TaskRunner + ReActAgent差距
退出条件finish 非 tool-calls + lastUser.id < lastAssistant.id (双重校验)仅基于 LLM 返回状态——无循环层级校验🔴 P0

Doom Loop连续 3 次相同工具+参数 → 触发 permission.ask("doom_loop")有死循环检测但无用户交互——直接 abort🟡 P1

4. 消息与 Part 系统

维度OpenCode message-v2.ts (11 种 Part 类型)Orbit ChatMessageRecord (5 字段)差距
消息粒度Part 级——tool call 是 message 内的一个 Part，与 text/reasoning 共存消息级——每条消息一个 role + content 字符串🔴 P0
工具调用可追溯ToolPart (pending→running→completed/error) 状态机 + output/error 字段无工具调用记录——只在 context dict 中存 invocation 日志🔴 P0
Token 统计每消息记录 input/output/reasoning/cache 四项 token + costTokenBudgetTracker 有估算但无持久化🟡 P1
结构化输出structured output 作为独立 Part 类型——可回溯clarification_status + structured_prd 在 API response 中，不入 session🔴 P0

9. P0 缺口清单 (阻塞基本使用)
