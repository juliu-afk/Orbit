# compose:tdd

## 思考框架（必须遵守——Kent Beck TDD 经典循环 + RUNE 启发增强）

### TDD 三阶段循环

```
RED → GREEN → REFACTOR → (循环)
```

#### RED: 先写失败测试
- 每次只测一个行为（一个 assert 对应一个行为）
- 测试命名: test_<函数名>_<场景>_<预期结果>
- 覆盖顺序: 正常路径 → 边界条件 → 异常情况
- ❌ 反面: 先写实现后补测试 ← 这是"开发后测试"，不是 TDD
- ✅ 正面: 测试先于实现——看到红灯（测试失败）再动手写代码

#### GREEN: 写最小实现
- 只写让测试通过的最少代码——不要"顺便"加功能
- 哪怕实现很丑——先绿了再重构
- ❌ 反面: 测试还没跑通就开始优化代码结构
- ✅ 正面: 最快让测试变绿

#### REFACTOR: 消除重复 + 改善可读性
- 绿灯后才能重构——测试是安全网
- 检查: 有 3 处相似代码？→ 提取
- 检查: 函数超过 30 行？→ 拆分
- 检查: 变量名准确描述用途？→ 重命名

### 验收标准驱动的测试覆盖
按优先级写测试:
1. P0: spec 中定义的验收测试（强制通过）
2. P1: 正常路径（happy path）
3. P2: 边界条件（空/零/负/超大/超长）
4. P3: 异常路径（网络失败/DB 超时/权限不足）

## 流程
1. 读 spec——理解功能需求和验收标准
2. RED: 写 1 个失败测试——覆盖最高优先级的未覆盖行为
3. GREEN: 写最小实现——让这个测试通过
4. REFACTOR: 重构代码——消除重复+改善可读性, 确认所有已有测试仍绿
5. 回到步骤 2——直到 spec 所有验收标准覆盖
6. 最终验证——跑全量测试确认无回归

## 原则
- **测试先行**——总是先看到红灯
- **最小实现**——只写让测试通过的代码
- **持续重构**——每次绿灯后检查是否可以改进
- **一个循环一个行为**——不要在一个循环里实现多个功能
- **禁止跳过 RED**——跳过 RED 不是 TDD, 是"写完代码补测试"
