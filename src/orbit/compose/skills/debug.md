# compose:debug

## 思考框架（必须遵守——基于"五问法"根因分析）

### 原则 0：根因优先——不要修复症状

症状 vs 根因:
- ❌ 症状: "API 返回 500" → 加 try-catch（掩盖问题）
- ✅ 根因: "API 返回 500" → 查日志 → 发现 UnboundLocalError → 变量未初始化 → 修复初始化逻辑

### 五问法（遇到问题时连问 5 个"为什么"）
```
问题: 用户上传 CSV 后预览为空
  → 为什么? API 返回 500
    → 为什么? csv_parser 抛出 IndexError
      → 为什么? 第 0 行是空行, parse_header 访问 rows[0]
        → 为什么没跳过空行? skip_empty_lines 默认为 False
          → 为什么默认 False? 历史遗留, 原需求要求保留空行
根因: skip_empty_lines 默认值不适配上传输出的使用场景
修复: 上传解析场景设 skip_empty_lines=True
```

### 修复原则
- **最小改动**——改动越小风险越低。能改一行不改一个函数，能改一个函数不改一个文件
- **验证通过**——修复后必须跑相关测试（至少 1 条回归用例）
- **记录根因**——commit message 写"为什么"而非"做了什么"

## 流程
1. **收集证据**——读取日志/测试输出/堆栈跟踪/截图
2. **复现问题**——能复现才叫理解了问题
3. **五问法定位根因**——逐层追问，直到找到根本原因
4. **提出修复方案**——最小改动原则，需说明"为什么这个修复是最小的"
5. **执行修复**
6. **验证**——跑相关测试，确认问题消失 + 无回归

## 输出格式
```markdown
# 调试报告: [问题简述]

## 症状
[用户看到的现象]

## 根因（五问法）
[逐层追溯链]

## 修复
[改动文件:行号] — [改了什么] — [为什么这么改]

## 验证
[跑了哪些测试，结果如何]
```
