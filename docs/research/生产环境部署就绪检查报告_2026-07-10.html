B2. 单元测试失败（1 条）

FAILED tests/unit/test_llm_client.py::test_error_rate_below_min_calls_no_trip

模块：gateway/llm_client —— LLM 调用入口。

影响：网关错误率统计逻辑可能已在变更中损坏。

修复方向：先复现确认是否为本次基线问题，再定位根因。

B3. .env 文件可能已被 Git 追踪

.gitignore 声明忽略 .env，但 .env 文件（907 字节）存在于工作树。

风险：若已提交，任何克隆仓库的人都能看到 API key 占位符或真实密钥。

验证：git ls-files .env

修复：git rm --cached .env + 轮换已暴露的密钥。

3. 建议项 （推荐完成，不阻塞发布）

S1. CI poetry.lock 不同步（历史遗留）

CI lint/typecheck job 报全红，根因 poetry.lock 与 pyproject.toml 版本不一致。

修复：poetry lock --no-update → commit new lock file。

S2. gitignore 遗漏（5 项）

缺失：.mypy_cache/、.ruff_cache/、.semgrep/、*.log、coverage.xml。

S3. Helm chart 缺失生产必要配置

  • 无 secrets 模板（LiteLLM key、DB 密码需通过 K8s Secret 注入）
  • 无 ConfigMap 模板（环境变量直接写在 values.yaml）
  • Ingress 默认禁用，生产需启用并配置 TLS
  • limits.memory: 2Gi 可能不够（168MB exe + LLM 调用叠加）

S4. Dockerfile 改为多阶段构建

当前单阶段 python:3.11-slim。建议 build stage 安装 poetry，run stage 仅复制 .venv。

S5. 前端 CI 接入

JS/TS lint、vitest、TypeScript 类型检查均不在 CI（前端仅作为 exe 构建步骤出现）。
