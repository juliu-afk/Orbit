## 2. P0-1：WebSocket 聊天端点认证

### 2.1 根因分析

```
AuthMiddleware (BaseHTTPMiddleware)
  → dispatch() 拦截 HTTP 请求 ✓
  → WebSocket 升级请求 ✗ — Starlette 在中间件栈中单独处理 WS 升级，
    不经过 BaseHTTPMiddleware.dispatch()
```

[chat.py:89-99](src/orbit/api/routes/chat.py#L89-L99) — `chat_endpoint` 直接 `await ws.accept()` 无任何 token 校验。

### 2.2 修复方案

**在 `chat_endpoint()` 内部 accept 前校验 token。**

token 传递方式：**WebSocket 查询参数 `?token=...`**（与现有 `verify_stream_token` 的 SSE 查询参数方案一致，快速止血）。

```python
# chat.py — 修改 chat_endpoint 函数开头
from orbit.core.config import settings  # 新增 import

@router.websocket("")
async def chat_endpoint(ws: WebSocket) -> None:
    # P0-1: WS 认证——accept 前校验 token
    # WHY gated on AUTH_ENABLED: 桌面应用默认不启用认证（无 env var），
    #   仅生产 k8s 部署（ORBIT_AUTH_TOKEN 已设）时强制校验，
    #   与 AuthMiddleware 行为一致
    if settings.AUTH_ENABLED:
        token = ws.query_params.get("token", "")
        if not token or token != settings.ORBIT_AUTH_TOKEN:
            await ws.close(code=4001, reason="未授权访问")
            return
    
    await ws.accept()
    # ... 后续逻辑不变
```

### 2.3 改动文件

| 文件 | 改动 | 行数 |
|------|------|------|
| `src/orbit/api/routes/chat.py` | `chat_endpoint`：accept 前加 token 校验（gate on AUTH_ENABLED）| +8 行 |
| `frontend/src/stores/chat.ts` | WS URL 附加 `?token=` 参数，读自 localStorage | +3 行 |

### 2.4 数据流

```
客户端 WS 连接 ws://host:18888/api/v1/chat?token=xxx
  → chat_endpoint(ws)
  → AUTH_ENABLED=False: 跳过校验 → accept（桌面应用，向后兼容）
  → AUTH_ENABLED=True: ws.query_params.get("token") → 校验
    → 无效: ws.close(4001) → 拒绝
    → 有效: ws.accept() → 正常聊天流程
```

### 2.5 客户端影响

前端 `chat.ts:69` — WS URL 构造改为：
```typescript
const token = localStorage.getItem('orbitAuthToken') || ''
const url = `${protocol}//${location.host}/api/v1/chat${token ? `?token=${token}` : ''}`
```
桌面应用无 token → 空 query string → 后端 AUTH_ENABLED=False 跳过校验，向后兼容。

## 3. P0-3：Shell Exec 模式切换

### 3.1 根因分析

[shell.py:263](src/orbit/tools/shell.py#L263)：
```python
proc = await asyncio.wait_for(
    asyncio.create_subprocess_shell(cmd, ...),  # shell=True
    timeout=5,
)
```

`create_subprocess_shell` 将 `cmd` 字符串传给 `/bin/sh -c`。即使 `validate_command()` 做了白名单校验，shell 解释器本身仍存在变量展开、命令替换等攻击面。`shlex.split` 的解析结果（`parts`）已可用但未复用。

### 3.2 修复方案

**改用 `create_subprocess_exec(*parts, ...)`—直接 exec 可执行文件，不经过 shell。**

```python
# shell.py:263 — 修改 exec_command 中的子进程调用
# 旧代码:
# proc = await asyncio.wait_for(
#     asyncio.create_subprocess_shell(cmd, stdout=..., stderr=..., cwd=...),
#     timeout=5,
# )

# 新代码: 直接 exec，不经过 shell 解释器
parts = shlex.split(cmd)  # 复用已有解析逻辑，与 validate_command 一致
proc = await asyncio.wait_for(
    asyncio.create_subprocess_exec(*parts, stdout=..., stderr=..., cwd=...),
    timeout=5,
)
```

### 3.3 管道/重定向影响

`create_subprocess_exec` 不支持 `|`、`>`、`<` 等 shell 运算符。这些符号会被当作字面参数传递，导致命令失败。

影响评估：
- 白名单中无任何需要管道的合法用例（`| sh` 已被拒绝）
- `ls | head` 等组合从来不是 Agent 的实际使用场景
- **安全收益（消除 shell 注入）>> 功能损失（0 个实际使用场景）**

### 3.4 改动文件

| 文件 | 改动 | 行数 |
|------|------|------|
| `src/orbit/tools/shell.py` | `exec_command()`: `create_subprocess_shell` → `create_subprocess_exec` | ~3 行改 |

### 3.5 测试影响

| 测试 | 影响 | 处理 |
|------|------|------|
| `test_exec_command_success` | mock `create_subprocess_shell` → 改为 mock `create_subprocess_exec` | 更新 mock 路径 |
| `test_exec_command_with_warnings` | 同上 | 更新 mock 路径 |
| 其余 `validate_command` 测试 | 无影响 | 不动 |

## 5. 风险与缓解

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| WS 认证破坏现有前端连接 | 中 | 前端同步追加 `?token=` 参数，阶段 4 E2E 验证 |
| `create_subprocess_exec` 路径查找差异 | 低 | `exec` 使用 `PATH` 查找，与 `shell` 行为一致 |
| 管道/重定向功能退化为错误输出 | 低 | 白名单从未允许合法管道，影响为零 |
| `shlex.split` 在 shell.py 重复解析 | 低 | `validate_command` 已解析一次，`exec_command` 再解析一次——性能可忽略 |
