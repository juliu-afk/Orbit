## 1. 战略定调与设计哲学

V14.1在V14.0“自研可控、极致性能”的基础上，正式补齐了四个尚未覆盖的极致优化维度，目标是实现**从代码语法到业务语义、从静态结构到动态行为、从应用逻辑到运行环境**的全方位覆盖。

### 1.1 设计哲学

- **极致性能：**单任务Token≤35，调度层延迟≤1.5s（不含验证层）。
- **透明可控：**所有逻辑可审计、可修改，零黑盒依赖。
- **全维度覆盖：**代码 + 数据库 + 配置 = 三图谱完整记忆。
- **动态+静态结合：**确定性分析（Tree-sitter/Z3）+ 运行时反馈（追踪/熵监控）。
- **渐进式复杂：**先跑通核心闭环，再逐步优化，不追求一次性完美。
- **生产就绪：**从设计之初即考虑状态持久化、灰度发布、降级策略、可观测性和混沌工程。
- **决策可追溯：**每次Agent动作、每个状态转换、每次验证结果均可追溯至原始上下文。

### 1.2 放弃CrewAI与Understand Anything的量化依据

| 对比维度 | CrewAI | 自研调度器 | Understand Anything | CodeGraph |
| --- | --- | --- | --- | --- |
| 单任务Token消耗 | ~4.5K（基准测试） | ≤35（目标） | 多Agent流水线调用消耗高 | 零Token（本地查询） |
| 单次运行耗时 | ~32秒（含验证） | ~1.5秒（目标） | 启动分析耗时数分钟 | <50ms查询 |
| 内存占用 | 框架开销~200MB | ~50MB（自研） | 1.2GB（实测峰值） | 50-200MB（SQLite） |
| 可控性 | 黑盒调度，难审计 | 完全可控 | 多Agent内部逻辑不可干预 | 完全可控 |
| Token节省 | 基准线 | 可优化至1/128 | 基准线 | 省51% Token（实测） |

数据来源：2026年多Agent框架基准测试报告；CodeGraph官方论文（42.8k Stars项目实测数据）。

## 2. 三图谱架构全景

V14.1首次提出**“三图谱”**体系：代码图谱 + 数据库图谱 + 配置图谱。三者统一存储在CodeGraph的SQLite中，通过统一的查询接口供Agent调用。

### 2.1 三图谱对比

| 图谱类型 | 数据来源 | 存储方式 | 核心价值 |
| --- | --- | --- | --- |
| **代码图谱** | 源码文件（Tree-sitter解析） | CodeGraph SQLite | 符号定义、调用关系、文件依赖、数据流（Def-Use链） |
| **数据库图谱** | DDL + 存储过程 + 触发器 + 有条件MVCC快照 | CodeGraph SQLite + 独立快照目录 | 表结构、字段类型、主外键、索引、历史Schema版本（核心表） |
| **配置图谱**新增 | .env + Nginx/Apache + PHP.ini + docker-compose.yml | CodeGraph SQLite + 黄金配置基线 | 环境变量依赖、运行时参数兼容性、容器资源限制 |

### 2.2 三图谱协作机制

- **代码修改 → 数据库影响分析：**Agent修改代码时，代码图谱告知“哪些函数操作了哪些表”，数据库图谱告知“这些表的Schema是否兼容变更”。
- **代码修改 → 配置影响分析：**代码图谱告知“哪些配置项被引用”，配置图谱告知“这些配置项在当前环境中的实际值”。
- **数据库修改 → 代码影响分析：**数据库图谱变更时，代码图谱反向查询“哪些函数/方法依赖了被修改的表/字段”。
- **配置修改 → 代码/数据库影响分析：**配置图谱变更时，检查代码图谱中对该配置项的引用，以及数据库连接池参数是否匹配。

### 2.3 运行时技术栈

- **主语言：**Python 3.11 + asyncio（调度器、验证器、图谱查询）
- **图谱存储：**SQLite（CodeGraph）
- **审计存储：**PostgreSQL + Redis（热数据缓存）
- **沙箱执行：**Docker Engine
- **前端：**Vue3 + AG-UI协议（WebSocket实时通信）
- **成本网关：**LiteLLM
- **形式化验证：**Z3 4.15.0+（通过rotalabs-verity调用）
- **依赖管理：**无重型框架依赖，仅标准库 + 少量轻量第三方库（httpx、pyyaml、python-dotenv）

## 3. 组件选型最终结论

V14.1在V14.0基础上新增配置图谱相关工具，整体选型如下：

| 候选项目 | 原属模块 | 最终决策 | 核心理由 |
| --- | --- | --- | --- |
| CrewAI | 调度编排 | ❌ 放弃 | Token消耗为自研预估的2-3倍，耗时超3倍（单任务~32秒），黑盒调度不可审计 |
| Understand Anything | 代码图谱 | ❌ 放弃 | 多Agent流水线黑盒，内存峰值1.2GB，启动分析耗时数分钟，不可控 |
| CodeGraph | 代码图谱 | ✅ 采纳 | 确定性、轻量（万行代码索引仅数十MB），省51% Token，零LLM调用 |
| rotalabs-verity | 形式化验证 | ✅ 有条件采纳 | 仅核心算法（代码量占比<10%），避免Z3全量验证导致超时；端到端1.87-3.73秒 |
| LiteLLM | 成本网关 | ✅ 采纳 | 性能充足（4核8G支撑1000+ QPS），实测成本降低70% |
| SwarmX | 调度基底 | ⚡ 借鉴 | 单文件500行极简设计参考其“Agent交接”原语，但不直接依赖 |
| Python标准库 | 配置图谱 | ✅ 采纳 | configparser + pyyaml + python-dotenv解析各类配置文件 |
| CopilotKit | 前端驾驶舱 | ❌ 放弃 | React+商业许可，改为自研Vue3+AG-UI协议 |
| Maat | 验证 | ❌ 坚决弃用 | 二进制分析框架，与V14的源代码操作目标严重冲突 |
| a3-python | 验证 | ❌ 放弃 | 与rotalabs-verity功能重叠，避免重复引入 |
| llm-gate | 成本控制 | ⚡ 借鉴 | 自研熔断器时参考其状态机设计，但不直接依赖 |
| Hermes Agent | 教训库 | ⚡ 参考 | 借鉴其跨会话经验积累的设计思路，自研教训库模块 |
| GitNexus | 代码图谱 | ❌ 弃用 | 与CodeGraph功能高度重叠，避免重复 |

## 6. 自研流水线详细设计

由于放弃CrewAI，调度器完全自研，核心基于**Python asyncio + 有限状态机（FSM）**实现。

### 6.1 调度器状态机设计

| 状态 | 描述 | 转换条件 | 审计记录 |
| --- | --- | --- | --- |
| `IDLE` | 空闲，等待新任务 | 接收到用户需求 | 任务创建事件 |
| `PARSING` | 解析用户意图 | 意图解析完成 | 解析结果摘要 |
| `WAITING_CLARIFICATION` | 等待用户补表需求（模糊需求触发） | 用户回复/[越时/[取消 |用户回复记录+越时事件 |
| `PLANNING` | 生成tasks.json（DAG） | 规划完成 | DAG结构摘要 |
| `PREFLIGHT` | 成本预检+环境预检+配置基线检查 | 通过预检 | 预检结果 |
| `CODING` | Agent循环编码（含熵监控） | 生成Diff | LLM响应摘要+熵值 |
| `VALIDATING` | 8层防幻觉验证（含L8配置漂移检测） | 全部通过 | 各层验证结果 |
| `COMMITTING` | Git提交+三图谱更新 | 提交成功 | Commit SHA |
| `DONE` | 任务完成 | — | 任务完成事件 |

| `FAILED` | 挂走等待人工（系统失败或澄清越时） | 重试3次失败/[澄清越时 | 失败原因+堆栈 |


### 6.2 Agent循环设计（参考SwarmX理念）

每个Agent执行单元遵循**“思考→行动→观察”**三阶段循环：

- **思考（Think）：**构建Prompt（含三图谱上下文、任务描述、历史教训），调用LLM生成结构化输出（JSON Schema约束 + 熵监控）。
- **行动（Act）：**解析LLM输出，执行工具调用（读文件、写Diff、运行测试）。
- **观察（Observe）：**收集行动结果（测试输出、验证结果、L1-L8反馈），决定是否继续循环或终止。

与CrewAI的关键区别：**循环次数由任务复杂度动态决定**，而非固定Agent数量。简单任务1次循环完成，复杂任务最多5次循环。每个循环步骤都保存检查点，支持故障恢复。

### 6.3 完整流水线时序

1. **T+0ms：**用户提交需求 → 状态机进入PARSING，保存检查点，写入审计记录。
2. **T+50ms：**MiniLM意图解析 + ClarificationEngine需求清晰度檢查 → 模糊则进入WAITING_CLARIFICATION（不进入PLANNING） → 生成任务摘要。
3. **T+150ms：****三图谱查询**（代码+数据库+配置）→ 获取完整上下文（零Token）。
4. **T+250ms：**LiteLLM成本预检 +**L8配置漂移预检**（比对黄金基线）。
5. **T+350ms：**Agent循环启动 → 调用LLM生成Diff（**含熵监控**，高熵则提前熔断）。
6. **T+350ms-2.5s：**LLM流式响应，实时写入审计日志。
7. **T+3s：****8层防幻觉验证**（含L3熵监控/采样一致性、L4 Z3、L5功能聚类、L6双向合约验证、L8配置漂移检测），每条验证结果写入审计表。
8. **T+3.5-8s：**验证通过 → Git提交 →**三图谱增量更新**（含运行时追踪数据反馈）。
9. **T+8s：**状态机进入DONE，保存最终检查点，写入任务完成审计记录。

### 6.4 增强型审计追踪（轻量级审计链）

**背景：**V14.1用“按需快照+Git溯源”替代全量MVCC，虽然节省了存储成本，但**丢失了Agent的推理链追溯能力**。Git只能记录最终提交，无法还原“为什么这么改（对应哪条PRD）”、“验证失败了几次”、“每次错误的类型是什么”。

**修正方案：**在检查点机制的基础上，新增**轻量级审计表（task_audit_trail）**，仅存元数据和摘要（不存全量快照），实现推理链追溯。

## 7. 企业级特性实现（自研版）

- **环境指纹自举：**自研脚本维护“依赖版本→Docker镜像Tag”映射表，启动即校验，不符则自动拉取预构建镜像。
- **不可变审计日志：**自研中间件以JSON Lines格式写入本地文件，每行带递增序列号与HMAC-SHA256签名，防止篡改。同时双写审计表（PostgreSQL）供结构化查询。
- **数据生命周期(TTL)：**基于三图谱的血缘标签（业务域标记），自动归档超过3个月的快照至冷存储；MVCC快照保留30天；审计表保留90天。
- **回滚后烟雾验证：**自研冒烟任务自动执行（编译检查 + 核心接口200验证 + 数据库写入测试 + 配置加载验证），失败则告警。
- **架构约束引擎：**自研DSL解析器（支持`@constraint`注解语法），嵌入自研前端。Agent生成的代码若违反约束则被验证层驳回重写。
- **实时熔断：**自研熔断器（参考llm-gate状态机设计），配合LiteLLM实现毫秒级超预算拦截。
- **配置漂移自动修复：**检测到配置偏离黄金基线时，自动执行修复脚本（如调整memory_limit），安全敏感项则告警转人工。
- **动态调用运行时追踪：**生产环境异步记录__call/eval等动态调用，反哺图谱。
- **LLM熵监控驾驶舱：**自研前端展示实时熵值曲线、高熵告警、Token流速。
- **教训库：**自研模块存储失败案例的因果链+修复方案，供后续任务参考（借鉴Hermes设计思路）。
- **推理链审计可视化：**驾驶舱支持按任务查看完整决策链（通过审计表的parent_step_id递归展示）。

## 9. 8层防幻觉体系（完整技术细节）

本方案系统化整合了2025-2026年学术界与工业界对抗代码幻觉的最新研究成果，构建了从“输入约束”到“提交拦截”的8层纵深防御体系。该体系与底层框架无关，完全基于自研逻辑和确定性工具实现。

### 9.1 幻觉分类与精准打击策略

基于**CodeHalu**框架，将代码幻觉分为四大类，分别映射到不同防御层：

| 幻觉类型 | 描述 | 示例 | 对应防御层 | 检测方法 |
| --- | --- | --- | --- | --- |
| **映射幻觉** | 错误地将需求映射到错误的API或数据结构 | 需求“排序” → 生成`array_reverse()` | L3采样一致性 + L6合约检查 | 语义相似度聚类 + 业务断言 |
| **命名幻觉** | 凭空捏造不存在的函数名、参数、库 | `pd.read_exel()` | **L2动态知识库** | AST确定性匹配 + API签名对照 |
| **资源幻觉** | 引用不存在的文件、路径、网络资源 | `include 'config/missing.php'` | L5沙箱执行 + L7 VibeGuard | 实际文件系统检查 + 沙箱运行时验证 |
| **逻辑幻觉** | 算法逻辑错误、边界条件遗漏、并发问题 | 循环边界`i <= count`导致数组越界 | L4 Z3验证 + L5功能聚类 | 形式化约束求解 + I/O行为聚类 |

#### 关键洞察

没有任何单一技术能解决所有幻觉问题。学术界已形成共识——**多层防御是唯一可行路径**。本方案的8层体系从L1到L8形成“宽进严出”的漏斗结构：
• L1-L2（廉价层）：拦截80%的简单幻觉（语法、命名），成本接近于零；
• L3-L4（中等成本层）：拦截15%的复杂幻觉（语义、逻辑），Token放大3-5倍但仅关键路径启用；
• L5（高成本层）：拦截剩余的5%高危幻觉，通过沙箱执行换取“错误率65%→2%”的效果；
• L6-L8（最后把关）：业务语义验证 + 配置一致性 + 提交拦截，确保无幻觉代码进入版本库。

### 9.3 各层详细技术实现

#### L1: 受约束解码

使用JSON Schema约束LLM输出结构。确保输出的代码变更、Commit Message、解释文本严格符合预定格式。这是成本最低、最有效的语法级防幻觉手段。

- **技术实现：**调用DeepSeek时使用`response_format={"type": "json_schema"}`；本地Qwen使用Ollama的Grammar约束。
- **Schema定义：**`file_path`、`action`（create/update/delete）、`content`、`explanation`。
- **Token开销：**极低（+50-100 tokens）。

#### L2: 动态知识库AST确定性匹配（增强：运行时追踪反馈）

将CodeGraph的图谱能力延伸为“动态知识库”，通过库内省获取准确API签名，并通过运行时追踪补充动态调用信息。

- **静态匹配：**Python`import module; dir(module)`；PHP`get_extension_funcs()`+ Reflection API。
- **动态追踪增强：**注入AOP探针记录`__call()`、`eval()`、`call_user_func()`等动态调用，异步写入日志，批量反哺图谱。
- **准确率：**检测100%，召回87.6%（F1=0.934），修复77%。
- **Token开销：**零Token。

#### L3: 采样一致性验证（增强：LLM概率熵监控）

- **采样一致性：**对关键任务生成3-5个候选代码，计算MiniLM语义嵌入相似度，低于阈值（如0.85）触发重新生成。
- **熵监控增强：**流式响应中实时计算Shannon熵H = -Σ(p_i × log₂(p_i))，滑动窗口均值超过2.5 bits时判定为“高熵状态”，触发快速重新生成或熔断。
- **准确率：**AUROC=0.76，比基线LYNX高15.2%。
- **Token开销：**3-5×（仅关键路径）。

#### L4: Z3形式化验证 + 自动推理检查

- **CEGIS循环：**LLM生成代码 → Z3验证 → CE2P修复建议 → LLM重新生成。
- **适用范围：**仅核心算法（支付状态机、并发锁、边界条件），设置5秒超时。
- **扩展：**自动推理检查——验证LLM输出解释与代码逻辑的一致性。
- **准确率：**99%（可证明保证）。
- **Token开销：**零Token（本地求解）。

#### L5: 功能聚类 + 沙箱执行

- **方法（MIT团队）：**生成多个候选变体，在Docker沙箱中执行，将I/O行为相同的候选归为同一簇。最大簇占比≥70%时采纳。
- **效果：**LiveCodeBench错误率65%→2%，保守阈值下可降至0%。
- **Token开销：**执行开销（仅对Z3验证通过后的高危代码启用）。
- **优化：**预构建镜像池+预热技术，沙箱启动从~200ms降至~50ms。

#### L6: 合约满意度评估（增强：合约-代码双向验证）

- **正向验证：**PRD→可执行合约（PHP Assertions），检查代码是否满足合约。
- **反向验证（增强）：**分析代码所有执行分支，检查每个分支是否至少覆盖一条合约。未覆盖分支判定为“合约漏洞”。
- **Token开销：**零Token。

#### L7: VibeGuard Git Hook提交拦截

- **实现：**`pre-commit`钩子中集成自研扫描器，检测不存在的函数、硬编码密钥、SQL拼接、危险函数引入。
- **Token开销：**极低（仅提交时触发）。

#### L8: 配置漂移检测与自动修复（新增）

- **黄金配置基线：**开发期建立.env/Nginx/PHP.ini/docker-compose.yml基线。
- **检测：**运行时比对当前环境与基线，发现差异。
- **自动修复：**memory_limit、upload_max_filesize等非敏感参数自动调整；敏感项告警转人工。
- **Token开销：**零Token。

### 9.4 静态分析的量化边界

**重要认知：**静态分析（L2的AST匹配、L4的Z3）能检测**16%-70%**的错误，以及**14%-85%**的库幻觉。其**理论上限为48.5%-77%**（来源：2026年ICSE静态分析工具量化研究论文）。这意味着**静态分析永远无法单独解决幻觉问题**——必须与动态验证（L5功能聚类）和人工审核（L6/L7/L8）结合。

本方案的8层体系正是对这一认知的回应：**每层都有明确的失效边界，层与层之间形成交叉验证**，静态分析解决确定性可判定问题，动态验证解决运行时行为问题，配置检查解决环境一致性问题。

## 10. 三图谱全覆盖

### 10.1 代码图谱

- **技术：**CodeGraph（Tree-sitter + SQLite）
- **内容：**符号定义、调用关系、文件依赖、数据流（Def-Use链）
- **更新：**Git post-commit钩子增量更新
- **增强：**通过运行时追踪反馈动态调用信息
- **Token开销：**零Token，查询<50ms

### 10.2 数据库图谱

- **静态解析：**自研预处理脚本解析DDL（表结构、字段类型、主外键、索引）
- **存储过程/触发器：**LLM辅助反解析生成结构化注释，存入CodeGraph扩展节点
- **MVCC快照（有条件恢复）：****适用范围：**仅核心业务表（支付、订单、用户、库存）——由架构师标记。**触发时机：**每次Agent提交涉及核心表的DDL变更时，自动保存完整Schema快照。**保留策略：**保留最近30天的快照，超过自动清理。**恢复方式：**`mysql < schema_snapshots/orders_2026-06-21.sql`
- **Token开销：**零Token

### 10.3 配置图谱（新增）

- **.env解析：**`python-dotenv`提取变量名、默认值、注释约束
- **Nginx/Apache解析：**自研轻量解析器提取fastcgi_pass、root、index、location
- **PHP.ini解析：**php --ini + ini_get()导出，关注memory_limit、upload_max_filesize、max_execution_time、opcache
- **docker-compose.yml解析：**pyyaml解析mem_limit、cpus、environment、volumes
- **存储：**存入CodeGraph扩展表`config_nodes`和`config_edges`
- **查询接口：**`get_config(key, environment)`、`find_dependencies(config_key)`
- **Token开销：**零Token

### 10.4 三图谱统一查询接口

- Agent通过统一`query_graph()`接口查询：`query_graph(type="code", symbol="PaymentService")`、`query_graph(type="database", table="orders")`、`query_graph(type="config", key="MEMORY_LIMIT")`。
- 全部本地执行，零Token开销，延迟<50ms（SQLite索引优化）。

### 10.5 审计图谱（第四维度）

**三图谱 → 四维闭环：**在代码、数据库、配置三个静态图谱的基础上，新增**审计图谱（Audit Graph）**作为动态维度。审计图谱记录了Agent执行的**完整推理链**（每个状态转换、每次LLM调用、每条验证结果），通过`task_audit_trail`表实现。

四者关系：**代码图谱**描述“是什么”（结构），**数据库图谱**描述“存什么”（数据），**配置图谱**描述“怎么跑”（环境），**审计图谱**描述“为什么这么做”（决策链）。

#### 10.5.1 审计图谱核心查询

- **按任务追溯：**`get_audit_trail(task_id)`→ 返回该任务完整执行序列。
- **按决策链追溯：**`get_step_chain(task_id)`→ 通过parent_step_id递归，还原推理树。
- **失败模式分析：**`get_failures_by_type(error_type, since)`→ 统计各类验证失败的频率。
- **Token消耗分析：**`get_token_distribution(task_id)`→ 各步骤Token消耗分布。
- **跨任务关联：**通过metadata中的project_id或requirement_id，关联同一需求的多次迭代。

#### 10.5.2 审计图谱与其他图谱的关联

- **审计 → 代码：**审计记录中的`output_summary`（代码摘要）与代码图谱的符号定义关联。
- **审计 → 数据库：**审计记录中涉及的SQL变更与数据库图谱的DDL节点关联。
- **审计 → 配置：**审计记录中的配置修改与配置图谱节点关联（通过L8层触发）。
- **代码 → 审计反向追溯：**给定一个代码文件，可追溯“是谁（哪个任务）生成的”、“经历了哪些验证回合”。

#### 10.5.3 存储与性能

- **存储引擎：**PostgreSQL（与检查点共用PG实例）
- **索引策略：**按(task_id, created_at)复合索引 + (task_id, parent_step_id)递归查询索引
- **分区策略：**按月分区（按created_at），便于归档和清理
- **写入性能：**异步批量写入（每100条或每1秒刷盘），不阻塞主流程
- **查询性能：**单任务完整链查询<100ms（索引优化后）

## 11. Token监控与毫秒级熔断

### 11.1 理论基础

- **权威数据来源：**API响应`usage`字段（100%准确）。
- **速度检测：**记录TTFT和总耗时，计算TPS = tokens/秒。
- **动态定价：**外部JSON配置高峰/非高峰单价，支持热更新。
- **熔断逻辑：**每次请求前基于当前时段、模型、预估Token数快速判断，毫秒级决策。

### 11.2 实践方案

- **自研熔断器（参考llm-gate）：**令牌桶/滑动窗口计数器，`allow_request()`延迟≤12ms。三种状态：CLOSED、OPEN、HALF_OPEN。
- **LiteLLM：**上层策略网关，按项目/用户设置每日/每月上限及RPM/TPM限流，单实例4核8G支撑1000+ QPS，降本70%。
- **集成方式：**自研调度器中包装`cost_guard`装饰器，预检通过后发起HTTP请求。
- **熵监控集成：**流式响应中实时计算熵值，高熵触发熔断。
- **前端可视化：**自研驾驶舱通过WebSocket实时推送成本、流速、熵值、熔断事件。
- **审计集成：**每次熔断事件和Token消耗记录写入审计表，供事后分析。

### 11.3 准确率汇总

| 数据类型 | 准确率 | 原理 |
| --- | --- | --- |
| Token用量（API） | 100% | 官方计费数据 |
| 输入预估（三图谱） | ≥99% | 精确统计+分词器估算 |
| 输出预估（滑动平均） | ~95% | 历史数据预测 |
| 熔断决策延迟 | ≤12ms | 纯内存判断 |
| 熵监控响应 | 实时 | 流式Token级计算 |

### 11.4 熔断恢复机制

- 熔断触发后进入“降级模式”：所有LLM请求拦截，改用本地Qwen或挂起人工。
- 半开状态：每5分钟允许试探请求，成功则逐步恢复。
- 所有熔断事件记录审计日志并推送告警。

### 13.3 关键教训与策略调整

- **“集成优先”的陷阱：**V13-OSS虽然快速集成，但CrewAI和UA的黑盒本质导致无法进行深度性能优化。转自研虽延长周期，但换来长期可控性和极致性能。
- **“完美主义”的代价：**V13原计划支持全量MVCC和时序图谱查询，但实际工程中此类低频需求不足以支撑其高昂的资源成本。V14改用“按需快照+Git溯源”的务实策略。
- **“运行环境”的重要性：**早期方案聚焦代码和数据库，忽视了配置漂移这一生产级故障的主要来源。V14.1补全配置图谱，形成“代码+数据+配置”三位一体的闭环。
- **“生成过程”的可观测性：**仅检查输出代码不足以防御所有幻觉，必须监控LLM生成过程中的概率熵和动态调用行为，这是V14.1四维增强的核心动因。
- **“生产就绪”的前置思考：**从V14.0到V14.1的演进证明，设计阶段就必须考虑状态持久化、灰度、降级、可观测性等生产级能力，而非事后补丁。
- **“审计追溯”不可少：**用Git替代MVCC虽然节省了存储，但丢失了Agent推理链的可追溯性。轻量级审计表以极低成本填补了这一空白。

### 13.4 关键决策的深度复盘与修正

**决策复盘：**“完美主义的代价：V13原计划支持全量MVCC和时序图谱查询，但实际工程中此类低频需求不足以支撑其高昂的资源成本。V14改用‘按需快照+Git溯源’的务实策略。”

**这个决策真的对吗？**—— 需要分为两个层面回答：

#### 层面一：决策在当时约束下的合理性 ✅

| 约束条件 | 具体数值/事实 |
| --- | --- |
| 开发周期 | 13周（含MVP+强化+环境+生产），资源极度紧张 |
| 全量MVCC成本 | 千表级项目，每日快照存储膨胀约**10-20倍**（PostgreSQL的pg_dump + 索引重建） |
| 实际使用场景 | 多Agent代码生成场景，需要回滚的频次极低（<5%的任务），主要用途是**审计溯源**而非**频繁回滚** |
| Git溯源替代性 | 代码本身存于Git仓库，配置/数据库Schema可以通过`alembic`迁移历史追溯 |

**结论：**在这些约束下，“砍掉全量MVCC”是**合理的工程权衡**——用20%的成本去覆盖5%的场景，在MVP阶段确实不应该做。

#### 层面二：决策的3个致命缺陷 ❌（过度简化）

| 缺陷编号 | 缺陷描述 | 真实场景证明 |
| --- | --- | --- |
| **缺陷1** | 审计溯源 ≠ Git历史 | Git是Commit级别（人为提交），无法记录Agent的Token级决策链：
• 原始PRD是什么？
• Planner生成了什么方案？
• 为什么选择了方案A而不是方案B？
• 验证失败了几次？每次的错误是什么？
**这些信息Git完全无法提供**——但恰恰是生产环境故障复盘最需要的数据。 |
| **缺陷2** | 数据库Schema变更 ≠ 应用层快照 | `alembic`只能看到“删除了字段”，但**无法还原被删除的数据**。
真实故障：Agent生成了一段删除字段的迁移代码并执行，导致数据丢失。全量MVCC可以恢复，alembic不能。 |
| **缺陷3** | 配置图谱 ≠ 环境完整快照 | 配置图谱只检测配置文件的指纹，不检测**运行时环境状态**（如`pip freeze`、内核参数、文件权限）。
真实场景：某次Agent执行时自动升级了某个依赖包，导致系统行为变化。配置图谱不会告警，但全量环境快照会。 |

#### 修正方案：轻量级审计链（已纳入V14.1）

- **不需要回到“全量MVCC”的极端**，而是增加**最小化审计链**——仅存元数据和摘要（前200字符）。
- **新增表`task_audit_trail`：**记录每个状态转换、每次LLM调用摘要、每条验证结果。
- **通过`parent_step_id`形成链式结构：**可递归追溯完整推理树。
- **存储成本：**千次任务仅~50MB（相比全量MVCC的50GB，节约1000倍）。
- **保留能力：**完整推理链追溯（高频审计需求）+ 故障根因分析 + Token消耗分布分析。
- **放弃能力：**完整状态回滚（低频需求，<1%场景）。

**修正结论：**砍掉“全量MVCC和时序图谱”是对的，但**砍掉“时序追溯能力”是错的**。修正方案是在不增加资源成本的前提下，恢复**推理链审计能力**。V14.1已通过新增审计图谱完成此修正，形成了“代码+数据库+配置+审计”的四维闭环。

**实施影响：**此修正不增加开发周期（审计表DDL已在第1周纳入），不增加主流程延迟（异步写入），仅增加约50MB/千次任务的存储开销。

## 14. 从设计到生产：关键鸿沟的工程化弥合

V14.1的设计蓝图已达到工业级标准，但要真正投入生产，还需在以下五个关键领域进行工程化落地。本节针对每个鸿沟提出具体的实施方法和融入V14.1架构的路径。

### 14.1 状态持久化与恢复

**鸿沟：**生产环境要求Agent具备韧性，能从故障中无缝恢复，避免重复执行和状态丢失。

**工程化方法：**

- **检查点（Checkpoint）机制：**在调度器状态机的每个关键状态转换点（PARSING→PLANNING，PLANNING→CODING，CODING→VALIDATING等）保存完整执行上下文（当前任务DAG、已执行的Agent循环历史、中间产物、三图谱查询结果缓存）。
- **存储设计：**热数据使用Redis（TTL=1小时，支持快速恢复），持久化数据存入PostgreSQL或S3，定期归档至冷存储。
- **恢复流程：**调度器重启后，读取最后一个检查点，从该状态继续执行，跳过已完成步骤。检查点包含`state_id`、`timestamp`、`dag_state`、`agent_history`、`validation_results`。
- **与V14.1集成：**在自研调度器的状态机中嵌入`save_checkpoint()`和`load_checkpoint()`方法，使用`pickle`或`json`序列化状态对象，支持版本兼容。
- **容灾演练：**定期模拟调度器宕机，验证检查点恢复的完整性和时效性（目标RTO≤30秒）。
- **审计集成：**每次检查点保存时同步写入审计记录，形成恢复操作的可追溯日志。

### 14.2 灰度发布与金丝雀部署

**鸿沟：**直接全量更新生产环境Agent风险极高，需要渐进式放量和快速回滚能力。

**工程化方法：**

- **版本化Agent服务：**将调度器、验证器、图谱服务等核心组件容器化，每个版本构建独立镜像，通过Kubernetes Deployment管理，打上版本标签（如`v14.1-canary`）。
- **流量路由策略：**在API网关（如APISIX/Traefik/Envoy）层面配置按比例路由（1%→5%→20%→50%→100%）或按内容路由（特定User-ID、请求头、项目标签）。
- **自动回滚条件：**定义关键SLO指标（错误率>5%、P99延迟>基线×1.5、Token消耗异常>基线×2、幻觉告警率>阈值），超过阈值时自动回滚至上一稳定版本。
- **蓝绿环境：**保留旧版本（蓝环境）作为“备份”，新版本（绿环境）验证通过后切换流量，实现秒级回滚。
- **与V14.1集成：**在LiteLLM网关层增加路由规则，支持按模型版本或Agent版本分流；构建独立的canary命名空间，新旧版本并行运行。
- **审计集成：**每次灰度切换记录审计事件，便于回滚决策分析。

### 14.3 降级与容错策略

**鸿沟：**下游API（LLM、数据库、图谱服务）或工具可能不可用，系统需优雅降级而非崩溃。

**工程化方法：**

- **熔断器模式：**在调用LLM和外部工具的客户端集成熔断器（自研或`circuitbreaker`库），设置失败阈值（连续5次失败或错误率>30%），触发熔断后快速失败返回降级响应。
- **多级降级链路：****L1降级：**主模型DeepSeek不可用 → 切换备用模型Qwen-Coder。**L2降级：**所有云端模型不可用 → 降级为本地规则引擎（简单任务）或直接挂起转人工。**L3降级：**图谱服务不可用 → 使用最后一次缓存的三图谱数据（标记为“陈旧”）。**L4降级：**Z3求解器超时 → 跳过形式化验证，仅依赖L2/L3/L5层。
- **智能重试：**对可恢复错误（HTTP 429/5xx）实施指数退避重试（最多3次，退避基数2，抖动±20%）。
- **超时控制：**每个外部调用设置明确超时（LLM调用30秒、Z3求解5秒、沙箱执行10秒），超时后进入降级路径。
- **与V14.1集成：**在自研调度器的Agent循环中包装`call_llm_with_retry`方法，集成熔断器状态检查；LiteLLM网关层配置模型优先级和降级策略；Z3调用设置5秒硬超时。
- **审计集成：**每次降级触发记录审计事件，便于分析系统韧性。

### 14.4 AgentOps体系

**鸿沟：**生产级Agent需要全链路可观测性、持续监控和自动化运维闭环。

**工程化方法：**

- **可观测性采集：**在调度器、验证器、沙箱、图谱查询等关键组件中埋点，记录每个步骤的输入、输出、耗时、Token消耗、熵值、决策日志、验证结果。
- **数据存储与可视化：****指标（Metrics）：**Prometheus采集（任务成功率、延迟分布、Token消耗速率、熔断次数、各层拦截率），Grafana构建仪表盘。**日志（Logs）：**ELK Stack（Elasticsearch + Logstash + Kibana）存储审计日志和调试日志。**追踪（Traces）：**Jaeger或Tempo收集分布式调用链。**审计（Audit）：**PostgreSQL存储结构化审计记录，供合规和根因分析。
- **自动化告警与修复：**设置阈值告警（单任务Token超过50、Z3超时率>5%、配置漂移检测、熵值持续>2.5 bits）并触发自动修复脚本（切换模型、重启沙箱、回滚配置）。
- **持续反馈闭环：**将生产环境中的成功案例和失败案例自动反馈到教训库，用于后续任务的参考和模型的持续优化。
- **与V14.1集成：**自研调度器中的审计日志（JSON Lines）直接作为观测数据源；通过OpenTelemetry SDK导出指标；前端驾驶舱集成Grafana面板，统一展示运维数据；审计表提供“按任务ID查看完整决策链”的可视化界面。

### 14.5 性能基准与混沌工程

**鸿沟：**生产环境的极端条件（高并发、网络抖动、组件故障）无法在开发环境中模拟，系统韧性未经实战考验。

**工程化方法：**

- **性能基准测试：**使用Locust或JMeter模拟并发请求（从1 QPS逐步加压至100 QPS），测试调度器的最大吞吐量、P99延迟、资源消耗，建立性能基线和扩容阈值。
- **混沌工程实验：**主动注入故障（如模拟LLM API超时、数据库连接断开、磁盘写满、网络延迟），验证系统的熔断、降级、重试和恢复机制是否符合预期。
- **实验场景设计：****延迟注入：**LLM API增加5-30秒延迟，验证超时控制和降级策略。**错误注入：**LLM API返回500/429，验证重试和熔断。**资源限制：**容器内存限制降至50%，验证系统在资源紧张时的行为。**网络分区：**隔离图谱服务，验证缓存降级。**数据库故障：**断开PostgreSQL连接，验证检查点降级和审计写入容错。
- **持续混沌：**在预发布环境中常态化运行混沌实验，作为CI/CD流程的一部分，确保每次迭代不引入韧性退化。
- **与V14.1集成：**在测试环境中部署Chaos Mesh或Gremlin，设定实验场景，验证V14.1的8层防御、熔断器、检查点恢复、审计链写入是否按预期工作。

**🎯 生产化实施路线图：**

- **第1-2周（生产准备）：**实现检查点机制、灰度发布配置（Kubernetes + APISIX）、熔断器集成、审计表DDL。
- **第3-4周（可观测性）：**部署Prometheus+Grafana+ELK，建立告警规则，集成OpenTelemetry，完成审计链可视化原型。
- **第5-6周（韧性验证）：**执行性能压测（建立基线）+ 混沌实验（验证降级策略），调优参数，验证审计链在故障下的完整性。
- **第7-8周（生产就绪）：**根据压测结果优化资源分配，编写运维SOP（标准操作流程），完成生产发布。

此路线图与V14.1的10周开发计划可并行或顺延，总体交付周期约14-16周。

#### 14.6.2 输出物测试能力

```
   /_________\
```

- PR必须通过：`pytest --cov=src`（覆盖率≥80%）+ `pytest tests/smoke/`（冒烟全过）

## 1. V14.1 五层上下文架构回顾

 
V14.1将上下文视为稀缺的战略资源，设计了五层分级架构：

 | 层级 | 名称 | 内容 | 访问方式 | Token策略 |

| L1 | 全局不可变上下文 | System Prompt、硬性规则、Agent角色定义 | 每次LLM调用自动注入 | 固定开销（~2K tokens/Agent） |

## 2. 开源项目对比与优化机会

 

### 2.1 Agentic Context Engineering (ACE) —— 上下文增量学习

 
核心思想：斯坦福与SambaNova提出将Prompt视为“可演化的剧本”。三个角色：Generator执行任务、Reflector诊断轨迹提取教训、Curator合并教训为增量子弹。关键洞察：传统Prompt优化器存在简洁性偏差（迭代优化趋于泛化）和上下文崩溃（整体重写会缩减小心的上下文）。

 | 维度 | V14.1现有设计 | ACE可借鉴点 |
| --- | --- | --- |
| L5长期记忆 | 仅检索相似历史教训 | 可演化的“子弹”式增量知识（带使用次数、时间戳、来源工具） |
| 经验更新 | 静态存储，不随时间演化 | Reflector + Curator形成闭环，经验持续精炼 |
| 去重机制 | 无 | FAISS语义去重 |

 

### 2.2 Architext —— 上下文工程框架

 
核心思想：将上下文构建从“零散的字符串拼接”提升为“可操作、可组合、可演化的工程实体”。核心特性：Provider驱动的动态内容生成、列表式增删改查操作、可见性开关控制、智能缓存。

 | 维度 | V14.1现有设计 | Architext可借鉴点 |
| --- | --- | --- |
| 上下文构建 | Prompt模板+字符串拼接 | 声明式Provider对象，动态按需渲染 |
| 上下文操作 | 静态组装 | 运行时pop/insert/append/slice操作 |
| 可见性控制 | 无 | Provider.visible开关，保留但不渲染 |
| 缓存机制 | 无 | 内容变更时才刷新，否则复用缓存 |

 

### 2.3 Observational Memory（Mastra）—— 双Agent背景压缩

 
核心思想：使用Observer和Reflector两个背景Agent将对话历史压缩为带时间戳的观察日志。Observer在消息达30K tokens时压缩历史，Reflector在观察达40K tokens时重组浓缩。关键优势：稳定的上下文窗口可充分利用Prompt缓存，Token成本降低最高10倍。

 | 维度 | V14.1现有设计 | Observational Memory可借鉴点 |
| --- | --- | --- |
| L3压缩策略 | 滑动窗口+二次摘要 | 双Agent按阈值触发压缩，格式为文本观察 |
| 缓存利用 | 无 | 稳定上下文窗口可最大化Prompt缓存命中 |
| 压缩触发 | 固定步数 | 基于Token阈值（可配置） |

 

### 2.4 LangChain Deep Agents —— 三层压缩技术

 
核心思想：Deep Agents SDK实现三层压缩：① Offloading大工具结果（>20K tokens时替换为文件路径引用+前10行预览）；② Offloading大工具输入（上下文超85%时替换为文件指针）；③ Summarization（生成结构化摘要，包含session_intent、artifacts、next_steps）。主动压缩可在上下文仅达10-20%时就触发。

 | 维度 | V14.1现有设计 | Deep Agents可借鉴点 |
| --- | --- | --- |
| 大工具结果 | 全量保留在上下文 | 超阈值自动offload到文件系统，仅留引用+预览 |
| 压缩触发时机 | 接近窗口满时 | 可配置阈值（10-20%即触发） |
| 摘要结构化 | 纯文本摘要 | 包含session_intent、artifacts、next_steps的字段化摘要 |

 

### 2.5 OpenViking —— 三层分级上下文加载

 
核心思想：将上下文管理建模为“文件系统”，采用L0/L1/L2三层分级结构按需加载。L0热数据（当前会话）、L1温数据（近期历史）、L2冷数据（长期存储），通过目录递归检索组合定位与语义搜索。支持可视化检索轨迹，便于调试。

 | 维度 | V14.1现有设计 | OpenViking可借鉴点 |
| --- | --- | --- |
| 上下文分级 | 五层但未明确冷热 | L0/L1/L2明确定义，按需逐级加载 |
| 检索可视化 | 仅审计日志 | 可视化检索轨迹，便于调试 |
| 自动压缩 | 手动触发 | 自动压缩对话内容、提取长期记忆 |

 

### 2.6 ContextPilot —— 上下文缓存复用

 
核心思想：上下文工程层的优化系统。智能上下文复用提升前缀缓存命中率、支持多种RAG库和推理引擎。测试数据：MultihopRAG上F1=64.68%（对比SGLang的64.15%），缓存命中率提升4-13倍，预填充延迟降低1.5-3.5倍，GPT-5.2测试中输入Token成本降低约36%。

 | 维度 | V14.1现有设计 | ContextPilot可借鉴点 |
| --- | --- | --- |
| 上下文缓存 | 无 | 智能前缀缓存复用 |
| 跨请求共享 | 无 | 多请求共享重叠上下文时最大化复用 |
| 成本优化 | 无 | 输入Token成本降低36% |

```python
 from typing import List, Optional
 from datetime import datetime
 import faiss
 import numpy as np

 class ACEKnowledgeEngine:
 def __init__(self, llm_client, vector_store):
 self.llm = llm_client
 self.vector_store = vector_store

 async def reflect(self, task_id: str, trajectory: Dict) -> Optional[KnowledgeBullet]:
 """Reflector: 分析执行轨迹，提取教训"""
 prompt = f"""
 分析以下任务执行轨迹，提取可复用的经验：
 任务：{trajectory['prd']}
 执行步骤：{trajectory['steps']}
 最终结果：{trajectory['result']}

 如果成功，提取"成功模式"（做了什么导致成功）。
 如果失败，提取"失败教训"（哪里出了问题）。
 输出格式：{{"type": "success|failure", "content": "...", "confidence": 0.0-1.0}}
 """
 response = await self.llm.generate(prompt)
 bullet = self._parse_bullet(response, task_id)
 return bullet

 async def curate(self, bullets: List[KnowledgeBullet]) -> List[KnowledgeBullet]:
 """Curator: 合并相似子弹，去重"""
 # 1. 语义去重（FAISS）
 vectors = [b.semantic_vector for b in bullets]
 if len(vectors) > 1:
 index = faiss.IndexFlatL2(len(vectors[0]))
 index.add(np.array(vectors))
 # 找到相似度>0.95的重复项
 duplicates = self._find_duplicates(index, vectors)

 # 2. 合并重复项
 merged = []
 for bullet in bullets:
 if bullet.id not in [d.id for d in duplicates]:
 merged.append(bullet)
 return merged
```

## 2. Fable 5 System Prompt 逆向工程剖析

 
Anthropic 为 Claude Fable 5 设计的 System Prompt（约12万字符、1585行、72个命名章节）是目前公开领域最完整的AI Agent“操作系统配置”。以下提炼其核心设计原则及对V14.1的启发。

 

### 2.1 七大核心设计原则

 | 原则 | Fable 5 实现方式 | V14.1 借鉴落地 |
| --- | --- | --- |
| 1. 结构先行 | 使用类似XML的闭合标签构建绝对语义边界；ANTML语法调用工具 | Agent Prompt 使用 &lt;instructions&gt;、&lt;rules&gt;、&lt;tools&gt; 标签分隔；工具调用使用 {{invoke:name}} |
| 2. 防御性编程 | 开篇即最高优先级热修复指令；明确红线清单；拒绝危险请求话术 | 每个Agent Prompt顶部设紧急指令区；L1-L8验证层作为“红线引擎” |
| 3. 模块化设计 | 外部只读技能文档（SKILL.md），按需加载；容器化操作环境 | 五Agent共享底层框架，差异化管理角色定义和工具列表；与Docker沙箱（Step MVP-03）理念一致 |
| 4. 自适应思考 | effort参数（max/high/medium）动态控制思考深度 | Agent System Prompt 加入Effort控制指令；简单任务用low/medium，复杂任务用high |
| 5. 降级路由 | 无法处理或高风险请求自动改由其他模型回答；“感觉有风险则少说” | L3高熵触发后Agent进入“沉默模式”；调度器状态机增加降级路径 |
| 6. 防蒸馏机制 | 内置推理提取分类器，检测到白嫖推理逻辑则拒绝或提供垃圾代码 | V14.1审计链核心推理步骤可标记为confidential，输出时打码或脱敏 |

## 1. 总览：双模架构设计理念

 

> 核心洞察：“图谱化之后仍然需要RAG，不是因为图谱不够好，而是因为两者解决的是不同的问题。”

 | 维度 | 精确查询（图谱） | 语义检索（RAG） | 适用场景 |
| --- | --- | --- | --- |
| 查询方式 | Cypher/SPARQL 精确匹配 | 向量相似度检索 | — |
| 返回结果 | 确定性的实体和关系 | 相关性排序的文本片段 | — |
| 准确率 | 100%（数据存在时） | 60-85%（依赖embedding质量） | — |
| 适用问题 | “流动比率的公式是什么？” | “最近关于流动比率的监管要求变化” | 精确 vs 探索 |
| 未知处理 | 查不到就返回空 | 总能返回“最像”的结果 | 确定 vs 模糊 |
| Token开销 | 零Token | 低（向量检索+LLM摘要） | 成本敏感 vs 探索优先 |

 

> 核心原则：图谱负责“事实锚定” RAG负责“上下文扩展”
 两者通过MCP协议（Model Context Protocol）统一暴露给Agent，Agent根据问题类型自动选择或组合使用。

### 7.2 ADR：知识图谱双模架构

 | ADR · 知识图谱采用“精确查询 + 语义检索”双模架构 |
| --- |

## 1. Agent间通信协议与契约

    

### 1.1 设计原则

    
        
- 异步优先：Agent间调用以异步消息为主，避免同步阻塞导致调度器卡死。
        
- 超时必设：每次跨Agent调用必须设置超时，默认30秒，可配置。
        
- 幂等性保障：消息可重放，下游必须支持幂等处理（通过request_id去重）。
        
- 熔断传播：下游Agent熔断时，上游Agent收到标准化错误码，可执行降级策略。
        
- 审计完备：所有通信记录写入task_audit_trail，支持全链路追踪。
    

    

### 1.2 通信模式

    | 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| Request-Response | Agent A发送请求，阻塞等待Agent B返回结果 | DeveloperAgent → QAAgent（验证代码） |
| Fire-and-Forget | Agent A发送消息，不等待响应 | 审计日志记录、指标上报 |
| Streaming | Agent B分块返回结果 | L3熵监控（流式Token分析） |
| Callback | Agent A提供回调URL，Agent B完成后异步通知 | 长时间运行的验证任务（Z3求解、沙箱执行） |

### 1.5 通信层实现

    

```python
# /src/communication/message_bus.py
    import asyncio
    from typing import Dict, Optional, List
    from src.communication.protocol import Request, Response, Notification

    class AgentMessageBus:
        """Agent间异步消息总线"""
        def __init__(self, checkpoint_manager):
            self._agents: Dict[str, 'Agent'] = {}  # 注册的Agent
            self._pending: Dict[str, asyncio.Future] = {}  # 等待响应的请求
            self._checkpoint = checkpoint_manager

        def register(self, agent_name: str, agent_instance: 'Agent'):
            """注册Agent到消息总线"""
            self._agents[agent_name] = agent_instance

        def unregister(self, agent_name: str):
            if agent_name in self._agents:
                del self._agents[agent_name]

        async def request(self, request: Request) -> Response:
            """发送同步请求，等待响应"""
            # 检查目标Agent是否可用
            if request.target_agent not in self._agents:
                raise AgentUnavailableError(request.target_agent)

            # 检查目标Agent熔断器状态
            if self._agents[request.target_agent].circuit_breaker.is_open():
                raise AgentCircuitOpenError(request.target_agent)

            # 创建Future等待响应
            future = asyncio.Future()
            self._pending[request.id] = future

            try:
                # 异步转发请求
                asyncio.create_task(
                    self._deliver(request)
                )

                # 等待响应（带超时）
                response = await asyncio.wait_for(
                    future,
                    timeout=request.timeout_seconds
                )
                return response

            except asyncio.TimeoutError:
                self._pending.pop(request.id, None)
                raise AgentTimeoutError(request.target_agent, request.timeout_seconds)
            finally:
                self._pending.pop(request.id, None)

        async def _deliver(self, request: Request):
            """实际投递请求到目标Agent"""
            try:
                target = self._agents[request.target_agent]
                response = await target.handle_request(request)

                # 记录通信审计
                await self._checkpoint.record_communication(request, response)

                # 唤醒等待的Future
                if request.id in self._pending:
                    self._pending[request.id].set_result(response)

            except Exception as e:
                # 异常情况：返回错误响应
                error_response = Response(
                    id=str(uuid4()),
                    correlation_id=request.id,
                    source_agent=request.target_agent,
                    target_agent=request.source_agent,
                    status="error",
                    error_code=ErrorCode.INTERNAL_ERROR,
                    error_message=str(e),
                    duration_ms=0
                )
                if request.id in self._pending:
                    self._pending[request.id].set_result(error_response)

        def notify(self, notification: Notification):
            """发送单向通知（Fire-and-Forget）"""
            asyncio.create_task(self._deliver_notification(notification))

        async def _deliver_notification(self, notification: Notification):
            if notification.target_agent in self._agents:
                target = self._agents[notification.target_agent]
                await target.handle_notification(notification)

        def get_agent_status(self, agent_name: str) -> Dict:
            """查询Agent状态"""
            if agent_name not in self._agents:
                return {"status": "unavailable"}
            return self._agents[agent_name].get_status()

        # === 幂等性去重 ===
        _processed_requests: set = set()

        async def is_duplicate(self, request_id: str) -> bool:
            """检查请求是否已处理（幂等性保障）"""
            if request_id in self._processed_requests:
                return True
            self._processed_requests.add(request_id)
            # 定期清理（生产环境用Redis TTL）
            return False
```

    

### 1.6 ADR：通信模式选型

    | ADR · Agent间通信模式 |
| --- |
| 决策 | 采用异步消息总线 + 同步Future等待的混合模式： ① Agent间调用通过MessageBus转发，调用方使用await bus.request()同步等待。 ② 底层使用asyncio实现非阻塞，支持超时和熔断传播。 ③ 长耗时操作（Z3、沙箱）使用Callback模式，避免长时间占用连接。 |
| 理由 | ① 简化调用方代码（同步写法，异步执行）。② 超时和熔断可在单一位置统一管理。③ 与V14.1的asyncio调度器自然兼容。 |
| 备选方案 | ① 纯异步回调（代码复杂度高，调试困难）→ 放弃。② gRPC流式通信（过重，不适合Agent间轻量通信）→ 放弃。 |

## 2. 工具调用标准化与注册机制

    

### 2.1 设计原则

    
        
- 声明式注册：工具通过装饰器或配置文件声明，支持动态加载。
        
- 权限隔离：每个工具定义allowed_agents，只有授权Agent可调用。
        
- 版本兼容：工具支持语义化版本，Agent调用时指定版本范围。

- 优雅降级：工具不可用时，系统自动降级（如返回缓存结果或跳过）。

### 2.3 工具注册中心

    

```python
# /src/tools/registry.py (续)
    import asyncio
    from typing import Dict, Optional, List
    from collections import defaultdict
    import time

    class ToolRegistry:
        """工具注册中心 - 管理所有工具的生命周期"""
        def __init__(self):
            self._tools: Dict[str, Dict[str, ToolSchema]] = {}  # name -> version -> schema
            self._handlers: Dict[str, Dict[str, Callable]] = {}  # name -> version -> handler
            self._rate_limiter: Dict[str, Dict[str, List[float]]] = defaultdict(dict)

        def register(
            self,
            schema: ToolSchema,
            handler: Callable
        ) -> None:
            """注册工具"""
            if schema.name not in self._tools:
                self._tools[schema.name] = {}
                self._handlers[schema.name] = {}

            self._tools[schema.name][schema.version] = schema
            self._handlers[schema.name][schema.version] = handler

        def get_tool(self, name: str, version: str) -> Optional[ToolSchema]:
            """获取工具元数据"""
            if name in self._tools and version in self._tools[name]:
                return self._tools[name][version]
            return None

        def get_latest_version(self, name: str) -> Optional[str]:
            """获取最新版本"""
            if name in self._tools:
                return max(self._tools[name].keys())
            return None

        async def invoke(
            self,
            name: str,
            params: Dict[str, Any],
            agent_name: str,
            version: Optional[str] = None
        ) -> ToolInvocation:
            """调用工具"""
            # 1. 解析版本（默认最新）
            if not version:
                version = self.get_latest_version(name)
                if not version:
                    raise ValueError(f"Tool '{name}' not found")

            # 2. 获取工具定义
            schema = self.get_tool(name, version)
            if not schema:
                raise ValueError(f"Tool '{name}:{version}' not found")

            # 3. 权限检查
            if agent_name not in schema.allowed_agents:
                raise PermissionError(f"Agent '{agent_name}' not allowed to call '{name}'")

            # 4. 限流检查
            if schema.rate_limit > 0:
                if not self._check_rate_limit(name, version, agent_name, schema.rate_limit):
                    raise RateLimitError(f"Rate limit exceeded for '{name}'")

            # 5. 执行工具
            start_time = time.time()
            try:
                handler = self._handlers[name][version]
                if schema.is_async:
                    result = await handler(params)
                else:
                    result = handler(params)

                status = "success"
                error = None

            except asyncio.TimeoutError:
                status = "timeout"
                result = None
                error = f"Tool '{name}' timed out after {schema.timeout_seconds}s"

            except Exception as e:
                status = "error"
                result = None
                error = str(e)

            duration_ms = (time.time() - start_time) * 1000

            # 6. 返回调用记录
            return ToolInvocation(
                id=str(uuid4()),
                tool_name=name,
                tool_version=version,
                agent_name=agent_name,
                parameters=params,
                result=result,
                error=error,
                status=status,
                duration_ms=duration_ms
            )

        def _check_rate_limit(self, name: str, version: str, agent: str, limit: int) -> bool:
            """检查限流"""
            key = f"{name}:{version}:{agent}"
            now = time.time()
            if key not in self._rate_limiter:
                self._rate_limiter[key] = []

            # 过滤1分钟内的记录
            self._rate_limiter[key] = [
                t for t in self._rate_limiter[key] if now - t = limit:
                return False

            self._rate_limiter[key].append(now)
            return True

        def is_deprecated(self, name: str, version: str) -> bool:
            """检查工具是否已废弃"""
            schema = self.get_tool(name, version)
            return schema.deprecated if schema else False

        def get_deprecation_message(self, name: str, version: str) -> Optional[str]:
            schema = self.get_tool(name, version)
            return schema.deprecated_message if schema else None
```

## 3. 多任务并发与资源调度策略

    

### 3.1 设计原则

    
        
- 优先级分层：任务分为CRITICAL/HIGH/NORMAL/LOW四级，高优先级抢占资源。
        
- 资源配额：按任务/团队/全局三级设置资源配额（LLM调用、沙箱实例、Token预算）。
        
- 公平调度：防止单个大任务独占资源，引入时间片轮转。
        
- 背压控制：资源不足时，新任务排队或拒绝，而非崩溃。

```python
            if task_id not in self._running:
                return False

                return True

        async def release(self, task_id: str):
            """释放任务资源"""
            async with self._lock:
                if task_id in self._running:
                    self._quota_usage["concurrent_tasks"] -= 1
                    # 释放其他资源...
                    del self._running[task_id]

                # 从队列中取出下一个任务（抢占式）

            for priority in [TaskPriority.CRITICAL, TaskPriority.HIGH,
                            TaskPriority.NORMAL, TaskPriority.LOW]:
                if self._queues[priority]:
                    next_task = self._queues[priority].popleft()
                    # 重新提交（递归）
                    await self.submit(next_task, priority)

        def get_queue_status(self) -> Dict:
            """获取队列状态"""
            return {
                "critical": len(self._queues[TaskPriority.CRITICAL]),
                "high": len(self._queues[TaskPriority.HIGH]),
                "normal": len(self._queues[TaskPriority.NORMAL]),
                "low": len(self._queues[TaskPriority.LOW]),
```

```python
    class TaskOrchestrator:
        def __init__(self, scheduler: ResourceScheduler):
            self.resource_scheduler = scheduler

        async def execute(self, task: Task):
            # 1. 提交任务到调度器
            priority = self._determine_priority(task)
            if not await self.resource_scheduler.submit(task.id, priority):
                # 排队等待
                task.state = TaskState.QUEUED
                return

            try:
                # 2. 执行主流程（每个LLM调用前检查资源）
                while task.state != TaskState.DONE:
                    # 资源预检
                    if not await self.resource_scheduler.can_proceed(task.id):
                        # 资源不足，挂起等待
                        task.state = TaskState.PAUSED_RESOURCE
                        await asyncio.sleep(5)
                        continue

                    # 执行下一步（如CODING）
                    await self._execute_step(task)

            finally:
                # 3. 释放资源
                await self.resource_scheduler.release(task.id)

        def _determine_priority(self, task: Task) -> TaskPriority:
            """根据任务类型和上下文确定优先级"""
            if task.context.get("is_hotfix"):
                return TaskPriority.CRITICAL
            if task.context.get("is_production_issue"):

            if task.context.get("is_feature"):
                return TaskPriority.HIGH
            if task.context.get("is_exploration"):
```

### 3.5 ADR：抢占式调度 vs 公平调度

    | ADR · 多任务调度策略 |
| --- |
| 决策 | 采用优先级抢占式调度 + 公平时间片混合策略： ① 高优先级任务（CRITICAL/HIGH）可抢占低优先级任务的资源。 ② 同优先级任务采用时间片轮转（每个任务最多连续运行30秒）。 ③ 长运行任务（>5分钟）自动降级为LOW优先级。 |
| 理由 | ① 生产故障修复必须优先保障（CRITICAL）。② 防止单个任务无限占用资源。③ 保证低优先级任务也能获得执行机会。 |
| 风险与缓解 | 风险：频繁抢占导致低优先级任务饥饿。缓解：CRITICAL任务每天不超过5个，超过后降级为HIGH。 |

## 23.1 架构锚定：编排层 vs 执行层

    

> 核心声明：V14.1 系统在设计上明确锚定在 “多智能体软件开发操作系统” 这一层级（编排层），而非单智能体执行工具（如 Claude Code、Codex）的层级。这一锚定决定了所有 Prompt 和 Context Engineering 的设计必须服务于 协作流程的编排与治理，而非单次代码生成的质量。

    

### 23.1.1 编排层 vs 执行层

    | 维度 | 执行层（Claude Code / Codex） | 编排层（V14.1） |
| --- | --- | --- |
| System Prompt 锚点 | "你是一个编程助手" + 工具定义 | "你是多智能体协作网络中的角色" + 协作契约 |
| 上下文来源 | 当前代码库的实时快照 | 四图谱（代码+数据库+配置+知识）+ 协作状态 + 审计链 |
| 信息组织方式 | 用户指令 + 文件内容 + 工具输出 | 结构化图谱查询 + Agent 产出摘要 + 状态机上下文 |
| 决策单位 | 单次 LLM 调用的输出 | 跨 Agent 协作流程的步骤转换 |
| 约束来源 | 用户指令 + 工具权限 | 协作契约 + 验证门禁 + 合规规则 |
| 状态管理 | 无状态（任务即生命周期） | 有状态（检查点贯穿全流程） |
| 通信格式 | 自然语言 | 结构化、跨 Agent 可解析 |
| 成功标准 | 生成正确的代码 | 交付经过验证、符合合约、合规检查的完整产物 |

    

### 23.1.2 为什么锚定编排层？

    
        
- V14.1 不是“另一个 Claude Code”：Claude Code、Codex 是单智能体执行工具，解决的是“如何让一个 AI 帮你写代码”。V14.1 解决的是“如何让一群 AI 协作完成软件开发全流程”。两者的目标不同，处于不同的抽象层级。
        
- 编排层的核心价值在于“治理”：单智能体工具依赖模型本身的推理能力和权限控制；V14.1 通过多 Agent 协作、状态机调度、四图谱、9层验证门禁、审计链，实现了对开发流程的系统性治理。
        
- Claude Code/Codex 可以是 V14.1 的执行引擎：V14.1 不排斥在底层调用 Claude Code 或 Codex 作为执行单元，但 V14.1 的核心价值在上层——编排、治理、验证、追溯。

## 23.2 五条核心设计原则

    基于“编排层锚定”，Prompt 和 Context Engineering 的设计必须遵循以下五条核心原则：

    

> 原则一：System Prompt 定义“协作角色”而非“个人能力”
        执行层写法（不应采用）："你是一个 Python 专家，擅长编写高质量的代码。"
        编排层写法（应该采用）："你是 V14.1 多智能体协作网络中的 DeveloperAgent，在 ArchitectAgent 确定的技术方案范围内，生成符合四图谱事实的代码，输出必须通过 L1-L9 验证。"
        核心差异：Prompt 描述的是 Agent 在协作流程中的位置和契约，而非个人技术能力。

    

> 原则二：上下文是“协作状态”而非“代码库快照”
        执行层写法（不应采用）：直接塞入 payment/service.py 的完整内容。
        编排层写法（应该采用）：注入上游 Agent 的产出摘要、当前 DAG 进度、四图谱查询结果、验证门禁状态。
        核心差异：上下文描述的是“协作进展到哪一步、上下游产出了什么、事实是什么”，而非“代码库长什么样”。

    

> 原则三：指令与约束来自“协作契约”而非“用户指令”
        执行层写法（不应采用）：“用户说：请修改 timeout 为 60”。
        编排层写法（应该采用）：协作契约约束（来自 PLANNING 阶段）+ 验证门禁规则（L1-L9）+ 合规要求（领域知识）。
        核心差异：约束来自整个协作流程的累积契约，而非当前用户的“一句话指令”。

    

> 原则四：上下文组织遵循“渐进式披露”而非“一次性加载”
        执行层写法（不应采用）：把所有文件内容塞进上下文窗口。
        编排层写法（应该采用）：五层分级（L1-L5），按需加载，每层有明确的 Token 预算和加载策略。
        核心差异：上下文是分层、按需、渐进式披露的，而非一次性装满。

    

> 原则五：通信格式服务于“跨 Agent 可解析”而非“人类可读”
        执行层写法（不应采用）：“我觉得这个函数应该改成异步的”。
        编排层写法（应该采用）：结构化 JSON（含 proposal_id、change、reasoning、evidence、contract_assertions）。
        核心差异：通信格式是结构化的、可被其他 Agent 自动解析的，而非自由文本。

## 23.3 五层上下文的编排层视角

    原 V14.1 的五层上下文架构（L1-L5）在编排层视角下应重新诠释如下：

    | 层级 | 原有定义 | 编排层重新诠释 | 承载的内容 |
| --- | --- | --- | --- |
| L1 | 全局不可变上下文 | 协作宪法：所有 Agent 必须遵守的全局规则 | System Prompt（协作角色定义）+ 协作契约 + 验证规则 + 红线清单 |
| L2 | 确定性事实上下文 | 协作事实库：当前任务依赖的确定性事实 | 四图谱查询结果（代码+数据库+配置+知识） |
| L3 | 任务动态上下文 | 协作状态：当前任务的执行状态 | PRD 摘要、上游 Agent 产出摘要、当前 DAG 进度、检查点摘要 |
| L4 | Agent 局部工作记忆 | 个体工作台：Agent 私有上下文 | 思考→行动→观察循环中的中间产物（不跨 Agent 共享） |
| L5 | 跨任务长期记忆 | 协作经验库：跨任务的模式复用 | 教训库、成功模式库（通过 RAG 检索注入） |

    

> 关键洞察：五层上下文的本质是 协作流程的信息分层——从全局宪法（L1）到具体执行（L4），从当前状态（L3）到长期积累（L5）。每一层都有明确的语义边界和加载策略，共同构成 Agent 感知的“协作全景图”。

## 23.4 验证门禁：协作契约的守卫

    在编排层视角下，L1-L9 验证门禁不再是“代码质量检查工具”，而是协作契约的守卫（Guardians of the Collaboration Contract）。

    

### 23.4.1 验证门禁的定位

    
        
- 执行层视角：验证门禁是“代码是否正确”的检查。
        
- 编排层视角：验证门禁是“Agent 的产出是否满足协作契约”的门槛。只有通过所有门禁，产出才能进入共享上下文，成为下游 Agent 的输入。
    

    

### 23.4.2 验证门禁与协作契约的映射

    | 层 | 验证内容 | 对应的协作契约条款 | 失败时的协作行为 |
| --- | --- | --- | --- |
| L1 | 输出格式合规 | 契约条款：输出必须符合 JSON Schema | 驳回，要求 Agent 重写 |
| L2 | 符号存在于四图谱 | 契约条款：不得引用不存在的符号 | 驳回，要求 Agent 修正引用 |
| L3 | 生成确定性（熵监控） | 契约条款：高熵产出视为无效 | 熔断，终止生成 |
| L4 | 类型正确性 | 契约条款：类型必须匹配 | 驳回，要求 Agent 修正 |
| L5 | 执行时正确性（沙箱） | 契约条款：代码必须可执行 | 驳回，触发重试 |
| L6 | 业务语义验证 | 契约条款：必须满足所有合约断言 | 驳回，标记为 NEEDS_HUMAN |
| L7 | 提交拦截 | 契约条款：不得包含危险代码 | 拦截，禁止进入 Git |
| L8 | 配置一致性 | 契约条款：配置不得漂移 | 触发自动修复或告警 |
| L9 | 合规性验证 | 契约条款：必须符合法规/标准 | 阻断，标记为 NEEDS_COMPLIANCE_REVIEW |

    

> 关键洞察：验证门禁不是“质检员”，而是协作契约的执行者。它们确保每个 Agent 的产出在进入共享上下文之前，已经满足所有契约条款，从而防止幻觉在 Agent 间传播。

## 24.1. 核心概念澄清

    

> 核心声明：在 V14.1 中，Agent 不是独立微服务、不是容器、不是进程，而是 调度器进程内的一个 Python 异步协程（asyncio Task）。

    

### 24.1.1 “调用 Agent”的两种语义

    | 语义 | 含义 | V14.1 的实现 |
| --- | --- | --- |
| A2A 通信 | Agent A 请求 Agent B 执行某任务或提供信息 | 通过 A2A 协议（结构化消息）实现 Agent 间通信 |
| 系统拉起 Agent | 调度器将 Agent 实例化并开始执行 | 调度器的状态机通过 asyncio.create_task() 拉起 Agent 协程 |

    用户的问题“怎么调用 Agent”属于第二种语义：系统如何将 Agent 实例化并启动执行。

    


    
    
    
    

    

## 24.2. Agent 的实现形态

    

```python
# Agent 的本质是一个异步协程类
        def __init__(self, llm_client, graph_repo, sandbox, checkpoint_mgr):
            self.llm = llm_client
            self.graph = graph_repo
            self.sandbox = sandbox
            self.checkpoint = checkpoint_mgr

        async def run(self, context: TaskContext) -> AgentResult:
            # 思考 → 行动 → 观察 循环
            while not self.should_stop():
                thought = await self.think(context)        # 调用 LLM
                action = await self.act(thought)           # 执行工具（通过 MCP）
                observation = await self.observe(action)   # 收集结果
                context = self.update_context(context, observation)
            return AgentResult(...)
```

    

### 24.2.1 关键特征

    
        
- 进程内执行：所有 Agent 运行在同一个进程中，无进程间通信开销。
        
- 协程隔离：每个 Agent 是一个独立的 asyncio Task，由事件循环调度。
        
- 状态隔离：Agent 间不直接共享内存，通过调度器的 Session/Checkpoint 传递经过验证的数据。
        
- 私有工作记忆（L4）：每个 Agent 有独立的局部工作记忆，不跨 Agent 共享。
        
- 拉起速度：协程创建为微秒级，远快于容器启动（秒级）。

```python
                    result = await self._run_agent(ParserAgent, task)

                elif task.state == TaskState.PARSING:
                    task.state = TaskState.PLANNING
                    result = await self._run_agent(ArchitectAgent, task)

                elif task.state == TaskState.PLANNING:
                    task.state = TaskState.CODING
                    # 🔴 关键：拉起 DeveloperAgent
                    result = await self._run_agent(DeveloperAgent, task)

                # ... 依次类推
```

    

### 24.3.3 第3步：拉起 Agent 协程（核心）

    

```python
# 实际拉起 Agent 的方法
    async def _run_agent(self, agent_class, task: Task) -> AgentResult:
        # 1. 构建 Agent 的上下文（注入 L1-L5）
            l1=SYSTEM_PROMPTS[agent_class.__name__],       # 协作宪法
            l2=await self._query_graphs(task),             # 四图谱事实
            l3=self._build_task_context(task),             # 任务状态
            l4={},                                         # 私有工作记忆（空）
            l5=await self._retrieve_memories(task)         # 长期记忆

        # 2. 实例化 Agent（依赖注入）
            llm_client=self.llm_client,
            graph_repo=self.graph_repo,
            sandbox=self.sandbox,
            checkpoint_mgr=self.checkpoint_mgr

        # 3. 🔴 拉起 Agent 协程（本质是异步函数调用）
        result = await agent.run(context)

        # 4. 将结果写入检查点（供下游 Agent 使用）
        await self.checkpoint_mgr.save(task.id, result)

```

```python
        async def run(self, context: TaskContext) -> AgentResult:
            for iteration in range(self.max_iterations):

                context.l4["last_observation"] = observation

                # 终止条件判断
                if self.should_stop(observation):

            return AgentResult(success=True, output=context.l4["final_output"])
```

## 24.4. 与 MCP/A2A 的抽象分层

    在 V14.1 中，MCP 和 A2A 服务于不同的抽象层级，而“系统拉起 Agent”是另一个独立的层级。

    

> ┌─────────────────────────────────────────────────────────────┐
        │              🌐 API 网关层（Step 1.1）                     │
        │  • 状态机驱动（Step 5.1）                                 │
        │  • 🔴 通过 asyncio.create_task() 拉起 Agent 协程          │
        │  • 通过 Checkpoint 管理状态（Step 2.2）                   │
        │  • 通过 Audit 记录决策链（Step 1.2 补丁）                 │
                       │ await agent.run()
        │  • 思考→行动→观察 循环（Step 5.1）                        │

    

### 24.4.1 三层调用关系

    | 层级 | 用途 | 协议/方式 | 调用方向 |
| --- | --- | --- | --- |
| 系统 → Agent | 调度器实例化并驱动 Agent 协程 | asyncio.create_task() + await agent.run() | 向上驱动 |
| Agent → 工具 | Agent 调用外部工具/资源（四图谱、沙箱、数据库） | MCP（Model Context Protocol） | 向下调用 |
| Agent → Agent | Agent 之间协作通信（任务分发、审查、仲裁） | A2A（Agent-to-Agent Protocol） | 水平通信 |

    

### 24.4.2 与现有协议的定位

    
        
- MCP 不在“拉起 Agent”层：MCP 是 Agent 已经运行后用来调用工具的协议。它不负责 Agent 的启动或生命周期管理。
        
- A2A 不在“拉起 Agent”层：A2A 是 Agent 之间已经运行后用来相互通信的协议。它也不负责 Agent 的启动。
        
- “拉起 Agent”是调度器的职责：通过 asyncio 协程直接实例化和驱动，是 V14.1 自研调度器的核心能力。
    

    


    
    
    
    

    

## 24.5. PRD/ADR 规格

    

### 24.5.1 PRD · Agent 拉起机制

    | PRD · Agent 拉起机制 |
| --- |
| 背景 | 外部系统需要通过调度器驱动多智能体协作。Agent 不是预先部署的微服务，而是由调度器按需实例化的异步协程。 |
| 用户故事 | 作为调度器，我根据状态机的进度，在合适的时机await agent.run()拉起对应的 Agent 协程，执行完成后将结果写入检查点。 |
| 需求描述 | ① Agent 定义为异步协程类，实现 run(context: TaskContext) -> AgentResult 方法。 ② 调度器通过 asyncio.create_task() 拉起 Agent。 ③ 拉起时注入 L1-L5 上下文（L1 协作宪法、L2 四图谱事实、L3 任务状态、L4 私有记忆、L5 长期记忆）。 ④ Agent 执行完成后，结果写入检查点（Checkpoint），供下游 Agent 使用。 ⑤ 外部系统通过 API 调用调度器，不直接调用 Agent。 |
| 范围 | Do：进程内协程拉起；依赖注入；上下文构建；结果持久化。 Don't：不通过 HTTP/gRPC 远程调用 Agent；不将 Agent 部署为独立容器。 |
| 数据契约 | ```python class TaskContext(BaseModel): task_id: str l1: str # System Prompt（协作宪法） l2: Dict[str, Any] # 四图谱查询结果 l3: Dict[str, Any] # 任务状态（PRD摘要、DAG进度） l4: Dict[str, Any] # Agent私有工作记忆 l5: List[Dict[str, Any]] # 长期记忆（检索结果） class AgentResult(BaseModel): success: bool output: Any error: Optional[str] = None duration_ms: float ``` |
| SC→AC | SC1: Agent 拉起成功 → AC1: 状态机触发 CODING 状态时，调用 _run_agent(DeveloperAgent, task) 返回 AgentResult。 SC2: 上下文注入完整 → AC2: Agent 的 run() 方法中 context.l1、context.l2 非空且包含预期内容。 SC3: 结果持久化 → AC3: Agent 执行完成后，检查点中存在对应 task_id 的记录。 |
| 待定决策 | Q: Agent 执行超时如何处理？ → 决议：在 _run_agent() 中包装 asyncio.wait_for(agent.run(), timeout=300)，超时后取消协程并标记 FAILED。 |

    

### 24.5.2 ADR · Agent 实现形态

    | ADR · Agent 实现形态 |
| --- |
| 决策 | Agent 实现为进程内异步协程，而非独立微服务或容器。 ① 所有 Agent 运行在调度器同一进程中。 ② 通过 asyncio 事件循环调度。 ③ 通过依赖注入传递外部依赖（LLM 客户端、图谱仓库、沙箱）。 ④ 通过 Checkpoint 实现状态跨 Agent 传递。 |
| 理由 | ① 无网络开销：协程间通信为零延迟。② 极速拉起：协程创建为微秒级。③ 共享内存：Checkpoint 直接读取，无需序列化传递。④ 简化运维：无需部署多个微服务。⑤ 与 V14.1 的 asyncio 调度器自然兼容。 |
| 备选方案 | ① 独立微服务（HTTP/gRPC 拉起）→ 网络延迟高，资源消耗大，不适合密集 Agent 协作。② 独立容器（Docker 拉起）→ 启动慢（秒级），资源隔离过度。 |

    1. 设计原则：替代 vs 补充

    
        **核心决策原则：**

        
            **替代条件：**开源方案在核心能力上**全面覆盖**自研实现，且社区成熟度足够时，执行替代。

    3.2 PRD · PR-Agent替代方案

    PRD · PR-Agent替代自研ReviewerAgent

    **背景**

        现有自研ReviewerAgent审查能力有限，不支持大规模PR、平台覆盖面窄。PR-Agent作为11k+ stars的开源项目，在专业性和效率上全面领先。

    

    **SC→AC**

        **SC1:** PR-Agent调用成功 → **AC1:** ReviewAgent调用PR-Agent的/review接口，30秒内返回审查结果。

        **SC2:** 接口兼容 → **AC2:** 调度器调用ReviewerAgent的方式不变，AgentResult格式不变。

        **SC3:** 审查质量提升 → **AC3:** 对比测试中，PR-Agent的审查意见覆盖率比自研提高≥30%。

    

    

    ADR · PR-Agent替代决策

    **决策**

        用PR-Agent替换自研ReviewerAgent的底层审查引擎，上层接口保持不变。

    

    **理由**

        ① 审查专业性、覆盖面、平台支持全面优于自研。
② 单次LLM调用（~30秒）远优于自研的多轮交互。
③ 作为成熟开源项目，无需从零维护。
④ 替换风险可控：上层接口不变，调度器无感知。

    7.1 PRD · 代码质量优化方案

    PRD · 代码质量优化

    **背景**

        V14.1现有验证体系对代码质量的保障主要集中在“防幻觉”和“基础安全”层面。引入专业化开源工具可系统性提升代码质量——从架构健康、AI代码专项检测到Agent行为评估。

    

    **用户故事**

        作为技术负责人，我期望AI生成的代码不仅没有幻觉，还能在架构健康、安全性、可维护性、逻辑正确性等维度达到专业级水平，且Agent自身的行为也是可评估、可改进的。

    

    **需求描述**

        ① 引入7个开源工具，覆盖架构健康、代码质量、AI代码检测、安全内生、PR审查、Agent行为评估。
② PR-Agent替代自研ReviewerAgent。
③ 其余6工具以非侵入方式融入L1/L7/事后评估层。
④ 所有工具通过MCP协议或CLI调用，保持系统解耦。

    

    **范围**

        **Do：**集成7工具；PR-Agent替代；6工具补充；事后评估闭环。
**Don't：**不替换V14.1核心调度逻辑；不改变Agent间通信协议。

        **SC1:** 架构门禁生效 → **AC1:** sentrux检测到架构分数下降>200时，L7拒绝提交。

        **SC3:** Agent行为可评估 → **AC3:** Agent-EvalKit定期输出行为质量报告，含忠实度等指标。

    

    

    7.2 ADR · 工具集成决策

    ADR · 工具集成决策

    **决策**

        采用“**替代 + 补充**”混合策略：

        ① PR-Agent替代自研ReviewerAgent（底层引擎替换）。

        ② sentrux、hefesto-ai、ai-slop-detector、ai-trust-validator、Project CodeGuard、Agent-EvalKit以非侵入方式补充融入L1/L7/事后评估层。

    

    **理由**

        ① 避免重复造轮子，利用成熟开源生态。
② 替代降低维护成本，补充填补能力空白。
③ 非侵入式融入保障系统稳定性。

        风险：工具版本更新可能导致兼容性问题。缓解：锁定工具版本，在CI中集成版本兼容性检查。

    # /src/validation/l7_vibe_guard.py（增强版）
    import subprocess
    import json
    from typing import List, Dict, Any
    from src.validation.base import Validator

    class L7VibeGuard(Validator):
        """增强版L7 VibeGuard - 集成多个质量工具"""
        def __init__(self, config: Dict[str, Any]):
            self.config = config
            self.tools = {
                "sentrux": self._check_architecture,
                "hefesto": self._check_quality,
                "slop": self._check_slop,
                "trust": self._validate_trust,
                "codeguard": self._security_review,
            }

        async def validate(self, code: str, context: Dict[str, Any]) -> ValidationResult:
            """执行所有L7验证"""
            results = []
            for tool_name, check_func in self.tools.items():
                result = await check_func(code, context)
                results.append(result)
                if not result.passed:
                    # 阻断传播：任何一项失败则整体失败
                    return ValidationResult(
                        passed=False,
                        errors=[f"[{tool_name}] {err}" for err in result.errors],
                        metadata={"check_results": results}
                    )
            return ValidationResult(
                passed=True,
                metadata={"check_results": results}
            )

        async def _check_architecture(self, code: str, context: dict) -> ValidationResult:

            cmd = ["sentrux", "gate", "--path", context["workspace"], "--format", "json"]
            result = subprocess.run(cmd, capture_output=True, text=True)
            data = json.loads(result.stdout)
            if data.get("score_delta", 0)  ValidationResult:
            """hefesto-ai代码质量"""
            cmd = ["hefesto", "analyze", "--code", code, "--format", "json"]
            result = subprocess.run(cmd, capture_output=True, text=True)
            issues = json.loads(result.stdout)
            critical = [i for i in issues if i["severity"] == "CRITICAL"]
            if critical:
                return ValidationResult(
                    passed=False,
                    errors=[f"CRITICAL: {i['description']}" for i in critical]
