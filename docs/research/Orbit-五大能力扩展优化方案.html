0. 执行摘要

背景

Orbit 当前是纯文本多 Agent 编排系统——操作对象仅限于代码和文本文件。对标 Claude Code（43 工具含图片/Computer Use）、Cursor（多模态上下文）、Copilot（Office 集成）、Open Interpreter（OS 级操作），Orbit 在五个维度存在能力缺口。本文档为每个缺口提供 3-4 个备选方案、选型理由和分阶段实施路线。

#能力当前状态优先级推荐方案预计工期
①文件类型扩展 仅 22 种文本扩展名白名单 P1 关键 python-magic + 插件式解析器注册中心 2 周
②视频/图片多模态 不支持——LLMRequest 无 image 字段 P1 关键 Claude Vision（主）+ Gemini Flash（免费降级）双路由 3 周
③多模态 LLM 集成 四模型全文本，Gateway 无 vision 管线 P0 致命 LLMRequest 扩展 content array + MultimodalAdapter 2 周（与②并行）
④Computer Use 不支持——操作对象是代码，非 GUI P1 关键 Claude Computer Use（主）+ Playwright 结构化工具（兜底） 4 周
⑤Office 文件读写 无 openpyxl/python-docx/python-pptx 依赖 P2 增强 openpyxl + python-docx + python-pptx 三件套 2 周
关键决策点

能力 ②+③ 必须并行实施。 多模态 LLM 网关扩展是图片/视频支持的先决条件——没有 vision-capable 的 LLM 管线，图片输入无法被 Agent 理解。能力 ④（Computer Use）依赖 ②+③ 的截图理解能力。能力 ① 和 ⑤ 相对独立，可并行推进。

① 文件类型扩展

1.1 现状

Orbit FileService（src/orbit/files/service.py）通过硬编码扩展名白名单过滤文件：

  • TEXT_EXT 白名单（22 种）：.py .ts .tsx .js .jsx .vue .css .scss .html .md .json .yaml .yml .toml .sql .txt .cfg .ini .env .sh .svg
  • EXCLUDE_EXT 黑名单（8 种）：.pyc .pyo .exe .dll .so .o .db
  • EXCLUDE_DIRS（13 个目录）：__pycache__ node_modules .git .venv ...
  • 文件读取上限 1MB，UTF-8 编码
  • 无 MIME 类型检测，无二进制文件解析能力

1.2 备选方案

方案 A
扩展现有白名单
在 TEXT_EXT 中追加常见扩展名（.xml .csv .log .rst .tex .go .rs .java .kt 等），其他逻辑不变。
✅ 改动最小，5 分钟完成
✅ 零新依赖
❌ 仍是硬编码，维护成本线性增长
❌ 无法处理二进制格式（PDF/图片）
❌ 新语言/格式每次需改代码
推荐
方案 B ★
python-magic + 插件式解析器注册中心
用 python-magic（libmagic 绑定）做 MIME 检测，建立 FileParserRegistry 插件体系——每种格式一个 parser，按 MIME 自动路由。
✅ MIME 自动检测，无需维护扩展名列表
✅ 插件架构——新格式注册即用
✅ 兼容二进制（PDF/图片/压缩包）
✅ 与现有 ToolRegistry 模式一致
❌ 需安装 libmagic 系统依赖（Windows 需 DLL）
❌ 改动 FileService 架构
❌ 约 500 行新代码
方案 C
Apache Tika 集成
通过 Tika Server（Java）或 tika-python 做全格式自动检测+内容提取。支持 1200+ 格式。
✅ 格式覆盖最广（1200+）
✅ 自动语言检测+元数据提取
❌ 需运行 Java 进程（Tika Server 500MB+ 内存）
❌ 与 Orbit 轻量定位冲突
❌ 引入外部服务依赖
方案 D
mimetypes 标准库 + 按需解析
用 Python 标准库 mimetypes 推断类型，仅对已知文本格式做读取。无外部依赖。
✅ 零外部依赖
✅ 标准库稳定
❌ mimetypes 基于扩展名猜测（不准确）
❌ 无法区分 UTF-8 vs GBK vs 二进制
❌ 无插件扩展能力

1.3 选型理由

选择方案 B — python-magic + 插件式解析器注册中心

核心理由：Orbit 已有成熟的 ToolRegistry 插件注册模式（src/orbit/tools/registry/），方案 B 的设计与之完全一致——开发者只需在 parser 文件底部 registry.register_parser() 即可注册新格式。python-magic 的 MIME 检测准确率 95%+（基于 libmagic 数十年的文件签名数据库），远超扩展名猜测。Windows 下的 libmagic DLL 已在 PyInstaller 构建链中验证过（pillow 同理）。

② 视频/图片多模态感知

2.1 现状

Orbit 的操作对象是代码和文本，不支持任何视觉输入：

  • LLMRequest 仅含 prompt: str + messages: list[dict]，无 image/video 字段
  • FileService 排除所有非文本格式
  • 无截图/屏幕捕获模块
  • 已安装 pillow（仅用于微信 QR 码生成）

2.2 图片支持：备选方案

推荐

Claude Vision（主）+ Gemini Flash（降级）
经 LiteLLM 网关调用 vision-capable 模型。Claude Sonnet/Opus 做精确理解，Gemini 2.0 Flash 做免费降级。图片以 base64 data URL 或 URL 传入 content array。
✅ Claude Vision 行业最强图表/截图理解
✅ Gemini Flash 免费 tier（1500 req/day）
✅ 经 LiteLLM 统一网关，现有熔断/降级可用

❌ 大图需压缩（Claude max 5MB/image）

2.3 视频支持：策略

视频处理策略：关键帧抽取 + 音频转录

当前没有 LLM 支持视频流式理解。所有 vision API 只接受图片。视频处理流程：

  1. 抽帧：用 opencv-python-headless 按场景变化检测抽取关键帧（每视频 ≤20 帧）
  2. 音频转录：用 openai-whisper（本地）或 Gemini Audio 提取语音文字
  3. 多模态理解：关键帧 → Claude Vision 描述画面 + 转录文本 → 融合理解

适用场景：PR 审查视频、产品演示录制、bug 复现录屏。

2.4 选型理由

选择方案 A — Claude Vision + Gemini Flash 双路由

核心理由：

  1. 质量优先：Claude Vision 在图表理解、UI 截图分析、代码截图 OCR 上三项 benchmarks 均领先 GPT-4V（参考 VQAv2/TextVQA/MMMU）。Orbit 主要场景是代码截图和技术图表，Claude 优势明显。
  2. 成本兜底：Gemini 2.0 Flash 每天 1500 次免费调用，处理缩略图/简单图标识别等低价值场景够用。
  3. 架构一致性：两个 provider 都经 LiteLLM 网关，复用现有熔断器、降级策略（主力失败 → 自动切 Gemini Flash）、Token 追踪。
  4. 渐进式：先上 Claude Vision（1 周），验证后加 Gemini 降级（1 周），最后加国产模型支持（可选）。

③ 多模态 LLM 集成

3.1 现状

Orbit 的 LLMRequest schema（src/orbit/gateway/schemas.py）仅支持纯文本：

class LLMRequest(BaseModel): prompt: str # 纯文本 prompt system_prompt: str # 纯文本 system messages: list[dict] | None # 仅 text content tools: list[dict] | None # ❌ 无 images / content_parts / vision_config

这是 ② 的前置依赖——没有多模态 LLM 管线，图片输入到不了模型。

3.2 备选方案

推荐
方案 A ★
LLMRequest 扩展 content array（OpenAI Vision API 兼容）
在 messages 中支持 content 为 list 格式——对标 OpenAI vision API。Adapter 层负责转换 provider 特定格式。
✅ 行业标准格式（OpenAI/Anthropic/Google 均兼容）
✅ 最小侵入——只改 schema + adapter
✅ 文本调用不受影响（向后兼容）
✅ 新 provider 只需写 adapter
❌ 需处理 base64 图片编码
❌ 大图片需预处理（压缩/缩放）
方案 B
独立 MultimodalGateway（新模块）
新建 src/orbit/gateway/vision/，与文本 Gateway 并行。各自独立路由。
✅ 隔离性好——文本/视觉互不干扰
✅ 独立熔断/降级
❌ 代码重复——_call_with_circuit/_stream_completion 等需重写
❌ 两套网关增加维护成本
❌ 调度器需知道何时用哪个网关
方案 C
统一 content array + 自动检测路由
在方案 A 基础上，Gateway 自动检测 content 中是否含 image——自动选 vision model 路由。
✅ 对调用方透明——同一个 generate()
✅ 无额外 API
❌ 自动检测可能误判
❌ 需额外模型路由逻辑

3.3 选型理由

选择方案 A + C 组合——content array 扩展 + 自动检测路由

核心理由：OpenAI vision content array 格式已是事实标准——Claude API、Gemini API、LiteLLM 均原生支持。方案 A 改动范围可控（schema + 2 个 adapter + client.py），非破坏性。方案 C 的自动检测路由与 Orbit 现有的 TaskModelRouter（按 task_type 路由）模式一致——只是多一个 "content_has_image → vision_model" 规则。

3.4 架构变更

④ Computer Use

4.1 现状

Orbit 操作对象是代码和文本，不是 GUI。现有研究文档明确结论：

「CUA（Computer Use Agent）核心能力是像素级 GUI 操作：截图 → 识别 → 点击(x,y) → 键盘输入 → 再截图。Orbit Agent 操作对象是代码和文本，不是 GUI。」—— docs/research/CUA项目源码解构——Orbit可借鉴模式分析.html

Orbit 已具备的接近能力：

  • Docker 沙箱执行代码（src/orbit/sandbox/）
  • 白名单 Shell 命令（src/orbit/tools/shell.py）
  • 文件系统读写（src/orbit/tools/filesystem.py）
  • Playwright 测试基础设施（tests/e2e/）

4.2 备选方案

推荐
方案 A ★
Claude Computer Use（主）+ 结构化工具（兜底）
集成 Anthropic Computer Use API——截图→理解→操作→再截图循环。同时提供结构化工具（Playwright/clicks）作为确定性操作的兜底。
✅ Claude 原生 Computer Use——最成熟的通用方案
✅ 无需训练——自然语言指令驱动
✅ 支持任何 GUI 应用（浏览器/桌面/IDE）
✅ 结构化工具兜底——精确操作不依赖像素
❌ 成本高（每步 $0.05-0.15）
❌ 延迟高（截图+推理×N 步）
❌ 需 Docker/KVM 隔离运行环境
❌ 依赖能力②+③（vision 管线）
方案 B
Playwright + 自研像素识别
用 Playwright 控制浏览器，自研截图对比+元素定位。不依赖外部 Computer Use API。
✅ Playwright 已集成（E2E 测试）
✅ 零 API 费用
✅ 浏览器操作精确可靠
❌ 仅限浏览器——不能操作桌面应用
❌ 自研识别远不如 Claude Vision 准确
❌ 维护成本高——每个网站需适配
方案 C
OpenAdapt / Skyvern（开源录制回放）
用开源 Computer Use 框架——录制人类操作→回放+AI 适配变化。
✅ 开源，可自部署
✅ 录制回放减少 AI 调用
❌ 项目成熟度低（Skyvern 仍在快速迭代）
❌ 需额外部署维护
❌ 不如 Claude CU 灵活
方案 D
纯结构化工具——不实现像素级 GUI
扩展 Shell 白名单 + 新增 Playwright 结构化工具（navigate/click/fill/screenshot）——Agent 通过工具调用操作，不依赖像素识别。
✅ 安全——白名单模式，无任意点击
✅ 快速——不经过截图→推理循环
✅ Orbit 已有全部基础设施
❌ 无法操作非浏览器应用
❌ 无法处理未知 UI
❌ 需为每个操作场景写工具

4.3 选型理由

选择方案 A（Claude Computer Use 主）+ 方案 D（结构化工具兜底）

核心理由：

  1. 分层策略：确定性操作（打开 URL/填表单/点按钮）走结构化工具——快、安全、零 AI 成本。不确定性操作（"找到发布按钮"、"检查这个表格是否正确渲染"）走 Claude Computer Use——灵活、通用。
  2. 渐进式实施：Phase 1 先上结构化工具（2 周）→ Phase 2 加 Claude CU（2 周）。Phase 1 产出已可用。
  3. 安全边界：结构化工具保持白名单模式（只允许 navigate/click/fill/screenshot/screenshot_element），Claude CU 在 Docker 隔离环境中运行——对标 Orbit 现有沙箱架构。
  4. 与现有架构一致：结构化工具走 ToolRegistry 注册；Claude CU 走 AnthropicAdapter（已在 gateway 中）；截图理解走能力②+③管线。

⑤ Office 文件读写（Excel / Word / PPT）

5.1 现状

Orbit 完全无法处理 Office 文件：

  • 无 openpyxl、python-docx、python-pptx 依赖
  • FDE 深度分析将此列为 🚫 缺失——数据集成管道（连接 CSV/Excel/PDF）未实现
  • 仅支持 CSV 通过文本读取（无结构化解析）

5.2 备选方案

推荐
方案 A ★
openpyxl + python-docx + python-pptx 三件套
Python 生态三大标准库——纯 Python 实现，无需系统依赖。支持读写 .xlsx/.docx/.pptx。
✅ 纯 Python——无系统依赖，PyInstaller 友好
✅ 社区成熟（openpyxl 7K+ stars）
✅ 支持样式/图表/公式（openpyxl）
✅ 与 FileParserRegistry 插件架构一致
❌ 不支持旧格式 .xls/.doc/.ppt（需 LibreOffice）
❌ python-pptx 功能较弱（模板支持有限）
❌ 大文件（>50MB Excel）性能差
方案 B
LibreOffice headless 转换
用 LibreOffice headless 模式将任意 Office 格式转为 PDF/CSV/纯文本。覆盖 .xls/.doc/.ppt 等旧格式。
✅ 全格式覆盖（含旧版 .xls/.doc）
✅ 转换质量高（原生渲染）
❌ 需安装 LibreOffice（800MB+）
❌ 启动慢（每次转换 2-5s）
❌ 并发受限（单实例）
❌ 不适合打包进 exe
方案 C
Polars（Excel/CSV）+ python-docx + 自研 PPT
Excel 用 Polars 高性能引擎（已有依赖），CSV/TSV/Parquet 原生支持，Word 用 python-docx，PPT 暂不实现。
✅ Polars 读 Excel 极快（比 openpyxl 快 5-10x）
✅ CSV/Parquet 已有基础设施
✅ 最小新依赖
❌ Polars 写 Excel 功能有限（无样式/图表）
❌ 放弃 PPT 能力
方案 D
OnlyOffice / Collabora Online API
通过 Docker 部署 OnlyOffice Document Server，API 调用转换/编辑。
✅ 全格式+协同编辑
✅ 渲染质量最高
❌ 需 Docker 运行额外服务（2GB+ 内存）
❌ 引入外部服务依赖
❌ 过于重量级——超出 Agent 工具范畴

5.3 选型理由

选择方案 A — openpyxl + python-docx + python-pptx 三件套

核心理由：

  1. 纯 Python 栈：三个库均纯 Python 实现，无 C 扩展（python-docx 有 lxml 但已广泛兼容），PyInstaller 打包无障碍。
  2. 覆盖主流场景：95%+ 的 Office 文件是 .xlsx/.docx/.pptx 格式。旧格式 .xls 可通过 openpyxl 的 read-only 模式（有限支持）或提示用户转换。
  3. Python 生态标准：openpyxl 是 pandas 的 Excel 引擎、python-docx 是 LangChain 文档加载器的基础、python-pptx 是 ReportLab 的替代——三个库都有大量社区验证。
  4. 与方案 A（能力①）互补：Excel/Word 文件通过 python-magic 识别 MIME → FileParserRegistry 路由到对应 parser → openpyxl/python-docx 解析 → Agent 获得结构化内容。
  5. 大文件兜底：>10MB Excel → Polars 高性能引擎（方案 C 的读部分），>10MB Word → 截断警告。

5.4 Office Parser 设计

# src/orbit/files/parsers/excel_parser.py class ExcelParser(FileParser): mime_types = [ "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet", "application/vnd.ms-excel", ] async def parse(self, path: Path) -> ParsedContent: """ 策略: - ≤10MB → openpyxl（完整功能——样式/公式/图表元数据） - >10MB → Polars read_excel（仅数据——速度优先） """ if path.stat().st_size > 10_000_000: return self._parse_large(path) return self._parse_full(path) def _parse_full(self, path): wb = openpyxl.load_workbook(path, data_only=True) sheets = {} for name in wb.sheetnames: ws = wb[name] sheets[name] = { "rows": ws.max_row, "cols": ws.max_column, "headers": [cell.value for cell in ws[1]], "sample": [[cell.value for cell in row] for row in ws.iter_rows(min_row=1, max_row=20)], } return ParsedContent( text=self._format_for_llm(sheets), metadata={"sheets": list(sheets.keys()), "size_mb": ...}, ) # src/orbit/files/parsers/word_parser.py class WordParser(FileParser): mime_types = ["application/vnd.openxmlformats-officedocument.wordprocessingml.document"] async def parse(self, path: Path) -> ParsedContent: doc = Document(str(path)) paragraphs = [p.text for p in doc.paragraphs if p.text.strip()] tables = [[[cell.text for cell in row.cells] for row in t.rows] for t in doc.tables] return ParsedContent( text="\n\n".join(paragraphs), metadata={"paragraphs": len(paragraphs), "tables": len(tables)}, ) # src/orbit/files/parsers/ppt_parser.py class PPTParser(FileParser): mime_types = ["application/vnd.openxmlformats-officedocument.presentationml.presentation"] async def parse(self, path: Path) -> ParsedContent: prs = Presentation(str(path)) slides = [] for i, slide in enumerate(prs.slides): texts = [shape.text for shape in slide.shapes if hasattr(shape, "text") and shape.text.strip()] slides.append({"slide": i+1, "content": texts}) return ParsedContent( text=self._format_slides(slides), metadata={"slides": len(slides)}, )
