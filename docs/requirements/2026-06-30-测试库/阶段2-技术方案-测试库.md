## 4. 数据流

### 4.1 开发者使用流程

```
开发者写测试
  │
  ├─ 1. 创建 Mock（选择预设故障/延迟）
  │     mocks = {"llm": MockLLMClient(fail_count=3), ...}
  │
  ├─ 2. 选择 Builder 或手写编排
  │     chain = TaskChain(mocks=mocks)
  │
  ├─ 3. 配置链路（链式调用）
  │     chain.start(prd).fast_lane().with_sandbox(MockSandbox(oom=True))
  │
  ├─ 4. 执行
  │     result = await chain.run_to_completion()
  │
  ├─ 5. 断言
  │     chain.assert_failed_at("CODING")
  │     assert "OOM" in chain.checkpoints[-1]["error"]
  │
  └─ 6. (可选) 用专用断言增强
        assert_circuit_state(mocks["circuit_breaker"], "OPEN")
```

### 4.2 Builder 内部数据流

```
TaskChain.start(prd)
  │
  ├─ 1. 创建 AgentInput（用 factories.agent_input()）
  │
  ├─ 2. 进入状态机循环（模拟 TaskRunner._agent_cycle）:
  │     for state in [IDLE, PARSING, PLANNING, CODING, VERIFYING, DONE]:
  │       │
  │       ├─ 2a. Role lookup（ROLE_MAP[state] → agent role）
  │       │
  │       ├─ 2b. 调用 MockLLMClient.generate() → LLMResponse
  │       │       │
  │       │       ├─ fail_count > 0 → 抛异常 → 熔断逻辑
  │       │       └─ 正常 → 返回预设 tool_calls/content
  │       │
  │       ├─ 2c. 工具调度（通过 MockToolRegistry）
  │       │       │
  │       │       ├─ tool_calls → dispatch → MockSandbox.run()
  │       │       └─ 无 tool_calls → GoalJudge 评估 → 决定是否继续
  │       │
  │       ├─ 2d. 状态转换 + 检查点保存（MockCheckpointManager.save()）
  │       │
  │       └─ 2e. 事件发布（MockEventBus.publish()）
  │
  └─ 3. 返回 AgentOutput(status="ok")
```

### 4.3 Scenario 运行流程

```
pytest 发现 tests/lib/scenarios/
  │
  ├─ conftest.py: scenario_mocks fixture → 创建所有 Mock
  │
  ├─ test_normal_task_flow(scenario_mocks):
```

## 5. 调度器状态变更

**N/A** — 本方案为测试基础设施，不新增或修改调度器状态。

理由：`TaskChain` 内部模拟 `TaskRunner` 的状态机循环，但不修改 `scheduler/orchestrator.py` 或 `scheduler/task_runner.py` 中的生产代码。测试库是对生产组件行为的**观察和断言**，不改变行为本身。

---

## 6. 防幻觉层影响

**N/A** — 本方案不修改 L1-L8 防幻觉判定逻辑。

理由：`scenarios/hallucination.py` 覆盖各层拦截场景，但 Mock 注入点在生产代码的调用方（LLMClient/Sandbox），不是防幻觉层内部。防幻觉层自身无改动。

---

## 7. 图谱 Schema 变更

**N/A** — 本方案不修改 CodeGraph SQLite 结构。

理由：测试库纯内存运行（`:memory:` SQLite / Mock），不读写真实图谱数据，不新增表或索引。

## 8. 边界 Case 清单（硬性）

| 类别 | 场景 | 预期行为 | 对应实现 |
|------|------|---------|---------|
| **LLM 网关** | `fail_count=0`, `fixed_response` 含 tool_call | MockLLMClient 正常返回 tool_calls 列表 | `mocks/llm_client.py:generate()` |

## 9. 风险与缓解

| # | 风险 | 概率 | 影响 | 缓解措施 |
|---|------|------|------|---------|
| 1 | **Mock 接口漂移**：生产代码接口变更，Mock 未同步更新，测试假阳性 | 中 | 高 | 每个 Mock 的 docstring 标注对应生产文件:行号；CI 门禁中加入"Mock 接口一致性"检查（`isinstance(mock, OriginalClass)` 或 Protocol runtime check） |
| 2 | **Builder 行为与真实 TaskRunner 偏差**：TaskChain 简化版状态机可能漏掉边界条件 | 中 | 高 | TaskChain 不重写状态机——它调用 `AgentFactory.create()` + `agent.execute()`，即真实 Agent 逻辑。仅 LLM/沙箱被 Mock。E2E 测试保留真实 LLM 调用路径 |
| 3 | **测试库自身无测试**：Phase 5 才补，Phase 1-4 期间库可能含 bug | 中 | 中 | Phase 1-4 期间，测试库的消费者（scenarios + 新模块测试）也是库的间接测试。bug 在开发阶段暴露 |
| 4 | **`tests/lib/` 被 pytest 自动发现**：库代码被当作测试运行 | 低 | 低 | `tests/lib/` 下不放 `test_*.py`（自身测试放 `tests/unit/test_lib_*.py` 或 `tests/lib/__tests__/`）；`conftest.py` 空文件不注册 fixture |
| 5 | **内存泄漏**：Mock 长时间运行积累 `calls` 列表过大 | 低 | 低 | Mock 提供 `reset()` 方法；scenario conftest 中 fixture 为 function scope，每次测试自动重建 |
| 6 | **测试隔离**：多测试并发访问同一 Mock 实例 | 低 | 中 | 所有 Mock 无全局状态；fixture scope=function 确保隔离；不实现 singleton Mock |
