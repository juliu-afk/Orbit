一、Orbit 基线能力速览

Orbit 是轻量级多 Agent 软件开发自循环系统。280 个 Python 源文件，~50 个功能模块。以下按架构层分类。

🔑 Orbit 的三大独特资产
① L1-L9 防幻觉体系——从静态图谱到 Z3 形式化验证，市面上没有同类产品做到 9 层
② Goal→SubTask→PR 自主流水线——从模糊需求到合并 PR，全程无人干预
③ 自蒸馏进化引擎——从执行轨迹中提炼可复用策略原则，效用驱动剪枝
⚠️ Orbit 的三大结构性问题
① 技能设计哲学缺失——技能是"做什么步骤"的清单，不是"按什么框架思考"的方法论注入
② 工作流是管道不是对话——spec 进去结果出来，中间无决策点、无推荐、无用户干预
③ 假设验证能力为零——L1-L9 验证"代码对不对"，不验证"方向对不对""假设会不会崩"

2.3 核心创新：方法论注入 vs 步骤编排

这是 pm-skills 与 Orbit 最根本的差异。两个系统都有"技能"，但技能的内容完全不同。

Orbit compose/skills/plan.md（步骤清单） 
1. 读取 spec 文件
2. 分析现有代码结构
3. 输出架构设计方案

告诉 Agent 做什么步骤。

pm-skills create-prd（方法论注入） 
你必须按以下 8 章结构思考：

1. Problem Statement — 用户问题，非功能描述
2. Objectives & Success Metrics — 可量化
3. User Segments — 按 JTBD 分，非按人口
4. Value Proposition — 差异化，非功能列表
5. Solution Overview — MVP 边界+Non-Goals
6. Release Plan — P0/P1/P2 优先级矩阵
7. Risks & Assumptions — 显式标注待验证假设
8. Appendix — 术语表+引用

每章有具体格式要求和反面示例。

告诉 Agent 按什么框架思考。

2.5 三个最独特的技能

① intended-vs-implemented（独有能力）
不只是在代码里找 bug。对比 PRD 文档和实际代码，找产品需求级差距：
"PRD 要求 5 秒内预览 CSV → 代码实测 45 秒（10MB 文件）→ GAP FOUND"
Orbit 的 L6 合约验证只检查技术接口（函数签名/类型），不检查产品需求（性能标准/UX/业务规则）。
② strategy-red-team（对抗性假设验证）
对 PRD 每个 load-bearing assumption：Steelman(最强重构) → Attack(最可能失败方式) → Rank(Impact×Likelihood×Cheapness)
输出 cheapest test + kill criteria。
"假设: 小企业主愿为财税合规付费 → 失败模式: 已用代账公司 500 元/月 → 最便宜验证: 访谈 5 人 → 杀死标准: ≥3/5 说'代账够用了'"
Orbit 的 CritiqueAgent 验证"代码对不对"，不验证"假设会不会崩"。
③ 插件领域打包架构
9 个插件各自是自包含领域包（上下文+术语+输出格式+技能间引用）。Orbit 6 个技能平铺在一个目录——扩展到 68 个时直接崩。

2.6 Orbit 可借鉴清单

#借鉴什么怎么实现优先级
1 技能从"步骤清单"升级为"方法论注入" 改造 compose/skills/plan.md——注入框架结构（问题定义→约束→方案对比→推荐→风险→Non-Goals），不只列出步骤 P0
2 链式推荐——工作流中间检查点 ComposeOrchestrator.run_spec() 每个 phase 后返回 {result, recommendations}，Skill 模型加 next_phases 字段 P0
3 插件/领域打包架构 compose/domains/{backend,frontend,infra,review}/ 替代平铺 skills/。每个 domain 有独立上下文+术语 P1
4 intended-vs-implemented 差距检测 扩展 hallucination/l6_contract.py——加产品需求级合约维度（性能标准/UX 规则/业务约束），不仅技术接口 P1
5 对抗性假设验证（red-team） 新 compose/skills/red-team.md + 在 _spec_review() 后自动调用。提取方案假设→逐个攻击→输出 kill criteria P1
6 测试场景自动生成 Goal 模块收到任务后自动生成 test scenarios（正常路径/边界/错误/并发），注入到 SubtaskSession P2

3.3 三个最值得借鉴的设计

① STRATEGY.md 锚点机制——Orbit 最缺的东西
不是普通 README，是持续更新的产品战略文档。所有技能执行时自动读取，输出自动对齐产品目标。
效果: 每个 Agent 都理解"我们在解决什么问题、谁是用户、怎么衡量成功"——不需要每次手动说明。
Orbit 有 goal/models.py 定义 GoalSession（含 description/constraints），但 Agent 的 system prompt 不自动注入项目级战略上下文。需要新建 context/prebuilders/strategy_prebuilder.py。
② 12+ Agent 并行审查——从串行到并行
每个审查维度独立 Agent，并行运行。全部完成后按严重程度聚合。
Orbit 的 CritiqueAgent 是单一维度（正确性/安全/性能/可维护性各 .40/.25/.15/.20 权重），一个 prompt 同时审 4 个维度——遗漏率高。
改造路径: scheduler/dag_runner.py 已支持 DAG 并行 → 新增 review/parallel_reviewer.py 扇出 6+ Agent → 聚合。
③ 复利闭环——蒸馏引擎就差最后一步连线
Compound 的 /ce-compound 和 Orbit 的 evolution/distill.py::DistillationEngine 做的是同一件事：从经验中提炼可复用知识。
差距不在引擎本身——在连线。 Compound 把经验注入到下一轮 brainstorm/plan/review 的 system prompt。Orbit 的蒸馏引擎有 search()/top_principles() 但没有人调用它们。
改造: 1 个新文件 context/builders/principles_builder.py + 改 prompt/builder.py 加一行调用。

3.4 Orbit 可借鉴清单

#借鉴什么怎么实现优先级
1 STRATEGY.md 锚点机制 新建 context/prebuilders/strategy_prebuilder.py——读取项目 STRATEGY.md 或 GoalSession，注入 Agent system prompt 顶部 P0
2 蒸馏引擎接入 Agent 上下文管线 新建 context/builders/principles_builder.py——调用 DistillationEngine.top_principles()，注入 system prompt P0
3 并行审查 Agent 新建 review/parallel_reviewer.py——用 dag_runner 扇出 6+ 维度审查 Agent，聚合结果 P1
4 LLM 增强蒸馏（Phase D） evolution/distill.py 加 distill_with_llm()——多条轨迹喂给 LLM，总结共同模式。当前纯规则驱动的蒸馏不支持跨轨迹模式发现 P2

四、RyanCodrai/turbovec — 零训练向量量化

📍 项目: github.com/RyanCodrai/turbovec  |  ⭐ 6.7K  |  MIT  |  论文: TurboQuant (ICLR 2026)  |  Rust 44.3% + Python 55.7% (maturin 绑定)
一句话: 零训练 × 16x 压缩 × 速度超 FAISS 的向量索引

4.1 核心算法：TurboQuant 六步管线

原始向量 → [1.归一化] → [2.随机旋转] → [3.逐坐标校准(TQ+)]
       → [4.Lloyd-Max量化] → [5.位打包] → [6.长度重归一化] → 压缩向量

核心洞察：高维超球面上的向量经随机旋转后，每个坐标服从已知分布——与输入数据无关。这是"零训练"的数学基础。

4.2 关键技术指标

指标turbovecFAISS IndexPQ (LUT256)
压缩比 (2-bit)16x (1536-dim: 6KB→384B)8x
压缩比 (4-bit)8x (1536-dim: 6KB→768B)4x
ARM 搜索速度 (M3 Max)快 12–20%baseline
x86 4-bit (Sapphire Rapids)快 1–6%baseline
召回率 R@1 (OpenAI d=1536)领先 0.4–3.4ppbaseline
训练需求零需要 K-Means 训练
过滤搜索SIMD 内核内执行（零召回率损失）后过滤

4.4 过滤搜索的 SIMD 原生实现

┌──────────────────────────────────────────────┐
│              32 向量块                        │
│  block_mask → 全 0 → 整块跳过 LUT+评分    │
│  非全 0 → 评分 → 槽级丢弃 → 堆插入             │
└──────────────────────────────────────────────┘

白名单越小 → 跳过的 SIMD 计算越多 → 搜索越快

4.5 与 Orbit 的对接点

Orbit 模块现状turbovec 可替代价值
knowledge/vector.py::VectorStore 纯 Python TF-IDF，中文 bigram 分词，内存索引。注释写"3.4d 迁移到 Qdrant" 直接替换 TF-IDF → TurboQuantIndex 零外部服务（匹配 Tauri 桌面部署）。语义搜索替代关键词匹配。16x 压缩节省内存
memory/ BM25 关键词搜索 + HyDE 假设问答（需 LLM 调用） 嵌入+HyDE → turbovec 向量检索 BM25 只匹配精确词。turbovec 语义搜索不受词汇差异影响
graph/engines/code_graph.py Tree-sitter AST 结构化查询 代码嵌入 → turbovec 语义代码搜索 补充结构化查询——找"类似模式的代码"而非"相同 AST 节点"

5.2 与 Orbit 的关系

ECC 已在 Orbit 部署（双插件 ECC+Compound）。以下分析聚焦 ECC 有哪些能力是 Orbit 可以内建而非依赖外部插件的。

PreToolUse Hook — Bash 执行前质量检查 security/validators.py + goal/verifier.py::_validate_commands() — 命令白名单+shell 元字符检测。但 仅在验证阶段触发，不在每次 Bash 执行前触发 可内建——Orbit 有 security/guard.py 框架，只需接到工具调用拦截点
AgentShield 1282 规则 — 通用安全漏洞扫描 hallucination/ L1-L9 — 专门检测 LLM 输出幻觉，不是通用安全漏洞扫描。Orbit 缺少 SQL 注入/XSS/命令注入的静态规则库 不易内建——1282 条规则需要专门维护。ECC 插件已覆盖，内建是重复造轮子
Context Monitor — 上下文窗口监控 compression/budget.py + compression/cascade.py — Token 预算跟踪+级联压缩。已覆盖 已有
Cost Tracker — Token 消耗统计 observability/metrics.py + resource_guard/budget_guard.py — 成本追踪+资源守护。已覆盖 已有
Session 管理 — 会话历史+模式提取 sessions/registry.py + memory/decision_log.py — 会话注册+决策日志。但缺少"从 session 提取 pattern"的能力（ECC 的 /learn 命令） 中等 — Orbit 蒸馏引擎做类似事但更结构化

5.3 Orbit 可借鉴清单

#借鉴什么怎么实现优先级
1 PreToolUse 安全门禁——扩展到所有工具调用 现有 security/guard.py 只在 goal/verifier.py 验证阶段调用。将其接入 scheduler/task_runner.py 的工具调用拦截点，每次 Bash/文件写入前触发 P1
2 HITL 触发条件——从 ECC 的 context-monitor 学阈值设定 现有 metacognition/hitl.py + metacognition/triggers.py 已定义触发条件。参考 ECC 的窗口利用率阈值（context_window_used > 85% → 触发总结）调优参数 P2
⚡ 结论: ECC 不需要大量借鉴。 Orbit 的安全/监控/成本体系已自建（security/observability/resource_guard/compression）。ECC 是外部补充插件，核心价值（AgentShield 1282 规则）继续用插件即可，不需要内建。

八、总结

Orbit 在基础设施层（编排/防幻觉/沙箱/熔断/图谱/自蒸馏）碾压四个开源项目。
但在"让 Agent 思考得更好"这件事上，pm-skills 和 Compound 有 Orbit 没有的东西。

8.1 一句话结论

Orbit 的定位
多 Agent 协同的运行时基础设施——确保 Agent 不把系统搞崩。
pm-skills 的定位
多 Agent 协同的思维基础设施——确保 Agent 在正确的方向上做事。

两者不竞争——两者互补。 Orbit 的引擎 + pm-skills 的思想 = 真正完整的 Agent 编排系统。

8.2 最大的 3 个发现

  1. 技能设计哲学是 Orbit 最致命的短板。 不是技能数量问题（6 vs 68）——是技能内容问题。Orbit 技能告诉 Agent "做什么步骤"，pm-skills 技能告诉 Agent "按什么框架思考"。这不是加功能能解决的——是思维方式的改变。
  2. 蒸馏引擎是最可惜的"差一步"。 Orbit 的 DistillationEngine 和 Compound 的 /ce-compound 做的是同一件事。Orbit 的引擎甚至更结构化（效用评分+剪枝+语义去重）。但 Compound 的经验会自动注入下一轮 Agent 的 system prompt——Orbit 的蒸馏引擎产出的原则没有人读。就差一个 60 行的 context builder。
  3. turboVec 完美匹配 Orbit 的部署约束。 Orbit 是 Tauri 桌面应用——零外部服务依赖是硬约束。Qdrant/Chroma/Milvus 都需要额外服务。turboVec 在同约束下做到了 16x 压缩+SIMD 加速+零训练。比当前规划"3.4d 迁移到 Qdrant"更匹配 Orbit 的部署模型。

8.3 不该做的事

  • ❌ 不要内建 ECC 的安全规则库——1282 条规则需要专门维护，ECC 插件已覆盖
  • ❌ 不要照搬 pm-skills 的 68 个技能——大部分是产品管理领域，Orbit 只需要软件工程领域
  • ❌ 不要引入 GPU 向量索引——Orbit 是桌面应用，CPU 方案(turboVec)更匹配
  • ❌ 不要重写 ComposeOrchestrator——现有实现扎实（Kahn 排序+RUNE 注入+Worktree 隔离+重试），加推荐逻辑即可，不需要推倒
