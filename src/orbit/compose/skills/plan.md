# compose:plan

## 思考框架（必须遵守——这是 Marty Cagan 8 章 PRD 框架的工程适配版）

### 1. 问题定义
用一句话说清用户痛点，不是功能描述。
- ❌ 反面: "实现导出功能" ← 这是功能描述，不是问题
- ✅ 正面: "会计人员月末需向税务局提交增值税申报表，当前手动整理耗时 2 小时"

### 2. 约束清单
显式列出所有约束（技术/业务/时间），标注哪些是可协商的：
- 硬约束（不可协商）: ...
- 软约束（可协商）: ...

### 3. 方案对比（至少 2 个可行方案）
每个方案标注 trade-off:
- 方案 A: ... （优势: ... / 代价: ... / 适用场景: ...）
- 方案 B: ... （优势: ... / 代价: ... / 适用场景: ...）

### 4. 推荐方案
选一个，说清为什么。必须回答:
- 为什么这个方案优于替代方案？
- 最大的风险是什么？如何缓解？
- 如果这个方案失败，fallback 是什么？

### 5. 风险矩阵（至少 3 个风险）
每个风险标注 likelihood（高/中/低）× impact（高/中/低）:
| 风险 | L | I | 缓解措施 |
|------|---|---|---------|
| ... | 高 | 中 | ... |

### 6. Non-Goals（防止范围蔓延）
显式标注本次不做的事:
- ❌ ...（理由: ...）

## 流程
1. 读取 spec 文件——理解项目目标和约束
2. 分析现有代码结构——read_file + grep + glob 了解上下文
3. 按上述思考框架输出架构设计方案——每个章节必须有实质内容，信息不足标注"待确认"
4. 方案保存为 plan.md——供后续技能引用

## 输出格式
```markdown
# 实现方案: [标题]

## 1. 问题定义
[一句话 + 一段背景]

## 2. 约束清单
- 硬约束: ...
- 软约束: ...

## 3. 方案对比
### 方案 A: ...
### 方案 B: ...

## 4. 推荐方案: [方案 X]
[选择理由 + 最大风险 + fallback]

## 5. 风险矩阵
| 风险 | L | I | 缓解 |
|------|---|---|------|

## 6. Non-Goals
- ❌ ...

## 7. 架构设计
### 模块划分
### 数据流
### 接口定义
```
