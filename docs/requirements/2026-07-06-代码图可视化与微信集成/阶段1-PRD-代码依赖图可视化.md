# 阶段1 PRD — 代码依赖图可视化

> 基于 Marty Cagan 8 章 PRD 模板。数据源：Orbit 现有 CodeGraph（ast 模块），可视化引擎：Cytoscape.js。

## 1. 背景与问题

### 当前状态
Orbit 已有代码图谱（`CodeGraphEngine`），用 Python `ast` 模块解析源码，提取符号定义（函数/类/变量）和调用关系，存入 `code_nodes` + `edges` 表。但**没有可视化界面**——用户只能通过 API（`/codegraph/definition`、`/codegraph/references`）逐个查询符号，无法全局审视代码结构和依赖关系。

### 用户问题
- "这个项目的模块依赖关系是什么样的？"
- "改了文件 A，哪些文件会受影响？"
- "有没有循环依赖？"
- "这个项目随时间是怎么演化的？"

### 业务价值
- 代码审查加速：一眼看清改动影响范围
- 新人上手：可视化项目结构，降低认知负载
- Agent 输出可视化：Agent 生成的代码改动可在图谱上高亮展示

### 参考报告
`docs/research/代码链接可视化.html` —— Cytoscape.js vs vis.js vs D3.js vs Note Graph 深度对比。选型结论：**Cytoscape.js**（首选，图分析能力最强，elements 格式与现有 Edge 表直接映射）。

## 2. 用户故事

### P0（核心——必须有，否则无法交付）

| # | 用户故事 | 验收标准 |
|---|---------|---------|
| P0-1 | 作为开发者，我想看到项目完整的代码依赖关系图，以便理解模块结构 | 打开页面后，展示以文件/模块为节点、导入关系为边的力导向图，节点可拖拽，边有方向箭头 |
| P0-2 | 作为开发者，我想点击节点查看详情（文件路径、符号数量、出入度），以便快速定位 | 点击节点弹出 tooltip/Panel，显示 file_path、定义符号数、被引用数、引用数 |
| P0-3 | 作为开发者，我想按文件路径/符号名搜索过滤，以便聚焦特定模块 | 搜索框输入→匹配节点高亮→未匹配节点半透明。支持正则。 |
| P0-4 | 作为开发者，我想切换不同布局算法（力导向/层级/同心圆），以便不同场景选择最佳视图 | 布局下拉框，至少 3 种布局（cose/breadthfirst/concentric），切换后平滑过渡 |

### P1（重要——提升体验，P0 完成后做）

| # | 用户故事 | 验收标准 |

| P1-3 | 作为开发者，我想筛选边类型（imports/calls/inherits），以便只看特定关系 | 图例/Checkbox 切换边类型显隐 |

## 3. 验收标准（汇总）

| # | 标准 | 验证方式 |
|---|------|---------|
| AC-1 | 页面加载后 3 秒内渲染完成（500 节点以下项目） | Playwright 计时 |
| AC-2 | 节点拖拽流畅（60fps），无视觉卡顿 | Chrome DevTools FPS 计数 |
| AC-3 | 搜索过滤响应 < 200ms | 单元测试计时 |
| AC-4 | 布局切换动画 ≤ 1 秒 | Playwright 计时 |
| AC-5 | 时间轴滑块拖拽→图谱更新 ≤ 500ms | Playwright 计时 |
| AC-6 | 数据来源为 CodeGraph API（`/api/codegraph/*`），不直接读 DB | 代码审查确认 |
| AC-7 | 支持 Python 项目（ast 解析）和 TypeScript 项目（预留接口） | 集成测试 |
| AC-8 | 空项目/无代码文件时显示 empty state 提示 | Playwright 截图 |

## 4. 成功指标

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 图谱渲染性能 | 500 节点 < 3s，1000 节点 < 6s | 前端 Performance API 打点 |
| 交互响应 | 点击/搜索/布局切换 < 200ms | 同上 |
| 用户理解速度 | 新人用图谱找到指定文件的平均时间减少 50% | 可选：用户测试（非 MVP 硬指标） |

## 5. 边缘情况

| # | 场景 | 预期行为 |
|---|------|---------|
| E-1 | 项目无 Python 文件/未构建索引 | 显示 empty state："尚未构建代码索引，请先在项目设置中运行代码解析" |
| E-2 | 单文件无任何引用关系 | 显示孤立节点（无边），tooltip 显示"无引用关系" |
| E-3 | 超大项目（>5000 节点） | 默认只显示顶层包/模块节点，下级符号按需展开；显示性能警告横幅 |
| E-4 | 边数量极大（密集依赖） | 自动启用边 bundling（多条同向边合并显示为一条粗边） |
| E-5 | 图谱数据更新中（重新解析） | 显示 loading overlay，不允许交互，防止脏读 |
| E-6 | 浏览器 tab 后台切换回来后 | 图谱保持原状态，不重新渲染 |
| E-7 | 时间轴无 git 历史数据 | 时间滑块 disabled，tooltip "需要 git 仓库才能启用时间维度" |
| E-8 | 窗口缩放 | 图谱自适应容器大小，debounce 重绘（300ms） |

## 6. 图谱影响分析

### 现有 CodeGraph
- **不修改** `code_nodes` / `edges` 表结构
- **新增 API 端点**：`GET /api/codegraph/graph-data` —— 返回 Cytoscape elements 格式的节点+边数据
- **可能新增**：`GET /api/codegraph/graph-snapshots` —— 返回 git 历史时间点的图谱快照列表（P1 时间轴需要）

### 跨图谱协作
- 代码图谱与元图谱（`meta_graph.py`）协作：元图谱存储图谱快照时间戳→git commit 映射
- 不涉及 database_graph 和 config_graph（本期范围外）

## 7. 调度器/防幻觉影响

- **不影响调度器状态机**：可视化只读，不触发任务
- **不影响防幻觉层**：可视化数据来自 CodeGraph（确定性解析，非 LLM 输出）
- P2-1（Agent 修改高亮）需要通过 WebSocket 接收调度器事件，但不修改调度器逻辑

## 8. 待确认问题

| # | 问题 | 初步建议 |
|---|------|---------|
| Q-1 | 图谱页面入口在哪？ | 驾驶舱左侧导航新增"代码图谱"菜单项 |

| Q-6 | 时间轴的粒度？ | 按 git commit。无 commit 信息的项目禁用时间轴 |

## 9. Non-Goals

- ❌ 不修改代码图谱的解析和存储逻辑
- ❌ 不支持在线编辑图谱节点/边（只读可视化）
- ❌ 不实现数据库图谱和配置图谱的可视化（本期仅代码图谱）
- ❌ 不实现跨项目依赖分析
- ❌ 不实现协作功能（多人同时编辑图谱）
- ❌ 不替换 CodeGraphEngine（ast 模块），也不引入 Tree-sitter（那是另一个需求）
