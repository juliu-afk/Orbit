# PRD+ADR：系统优化——复杂度路由 + 快车道 + UX 简化

> 依据：客观评价报告三个维度（工程化差距/过度工程化/交互复杂）的优化建议。

## Step O1：任务复杂度路由与快车道模式

| PRD (产品需求文档) |  |
| --- | --- |
| **背景** | 当前所有任务走完整 6 步流水线（PARSING→PLANNING→CODING→VERIFYING→REVIEW→DONE），无论复杂度。"改一行日志级别"和"实现支付网关"花费同样的流程开销。简单任务浪费 5000+ Token 和 3 分钟等待，用户体验差。需要复杂度感知的路由机制——简单任务走快车道，复杂任务走完整流水线。 |
| **用户故事** | 我说"把日志级别改成 debug"，系统识别出这是单文件单行修改，自动走快车道——跳过方案对比、跳过 Z3 验证、跳过多候选生成——30 秒内完成。我说"实现支付网关"，系统识别出这是高复杂度需求，走完整 6 步流水线，确保质量。 |
| **需求描述** | ① **复杂度评分器**：在 PARSING 阶段评估任务复杂度（0-100），维度：涉及文件数(×0.3)、修改范围(×0.25)、风险等级(×0.25)、是否核心逻辑(×0.2)。② **快车道路由**：评分<30 → 快车道（PARSING→CODING→DONE，跳过 PLANNING/REVIEW，轻量验证）。评分≥30 → 完整流水线。③ **用户模式选择**：API 参数 `mode: "auto" | "fast" | "full"`——auto=系统决策，fast=强制快车道，full=强制完整。④ **历史相似度复用**：检测到与历史任务相似度>90%时，直接复用推理链并提示用户"检测到相似任务，是否直接应用？"。⑤ **Step 0.3 增强**：澄清流程不超过 3 轮，第 3 轮后强制总结。 |
| **范围 (Do/Don't)** | **Do：**复杂度评分器/快车道路由/用户模式选择/历史复用/澄清轮次上限。<br>**Don't：**不修改已有验证层逻辑；不修改 Agent 角色定义。 |
| **数据契约** | **复杂度评分输出:** `{ score: 0-100, file_count: int, scope: "single_line"|"single_file"|"multi_file"|"multi_module", risk: "low"|"medium"|"high", is_core: bool, recommended_mode: "fast"|"full" }` |
| **异常定义** | ① 历史任务匹配失败 → 不影响正常流程，静默跳过。② 复杂度评分异常 → 默认走完整流水线（保守策略）。③ 用户强制 fast 模式但任务实际复杂 → 提示警告但仍执行快车道（用户自主决策）。 |
| **成功标准→验收** | **SC1:** 简单任务 Token -70% → **AC1:** "改一行日志"任务走快车道，Token 消耗<500，完成时间<30s。<br>**SC2:** 复杂任务不走快车道 → **AC2:** "实现支付网关"评分>60，走完整流水线。<br>**SC3:** 澄清 ≤3 轮 → **AC3:** 任何输入 3 轮内强制总结。<br>**SC4:** 历史复用 → **AC4:** 重复提交已完成的简单任务，系统检测到并提示复用。 |

| ADR (架构决策记录) |  |
| --- | --- |
| **技术栈** | Python stdlib；复用已有 `ClarificationEngine`；零新依赖。 |
| **决策** | 复杂度评分器实现为独立 `ComplexityScorer`，在 `Scheduler._agent_cycle(PARSING)` 中调用。评分<30 时修改状态转换图跳过 PLANNING/REVIEW。历史复用在 `ClarificationEngine` 中增加 `find_similar_task()` 方法。 |
| **理由** | ① 评分器独立可测试。② 状态跳转利用已有 `_transition()` 机制，不引入新框架。③ 历史复用复用已有 `LessonStore`，不新增存储。 |
| **架构位置** | `scheduler/complexity.py`（新增）、`scheduler/orchestrator.py`（修改——PARSING 后调用评分器）、`scheduler/clarifier.py`（修改——增加历史复用+轮次上限）。 |
| **实施细节** | **评分算法（Phase 0 启发式）：** 关键词检测 "修改"/"改一行"/"加个日志" → 低分倾向；"实现"/"重构"/"架构" → 高分倾向；涉及文件数从 PRD 文本推断；涉及核心模块（术语定义中的核心列表）→ +30 分。**快车道跳过列表：** PLANNING、REVIEWING、L3 熵监控、L5 Z3 验证、L6 合约验证。保留：CODING、L1 静态校验、L4 类型检查。 |
| **风险** | 评分低估导致复杂任务走快车道。缓解：保守默认——评分不确定时走完整流水线。 |
| **依赖链** | 依赖 Step 0.3（ClarificationEngine）、Step 5.1（Scheduler 状态机）。 |

## Step O2：UX 简化——智能默认值 + 批量确认

| PRD (产品需求文档) |  |
| --- | --- |
| **背景** | 用户从输入需求到收到结果需要 6-8 轮交互（项目选择→Issue 关联→PRD 确认→方案选择→等待验证→交付确认）。每多一个确认步骤，中断率增加 15-20%。移动端体验尤其糟糕。需要减少交互轮次。 |
| **用户故事** | 我说"修一下支付超时"，系统自动识别当前项目、自动关联最近 Issue（或跳过）、自动确认 PRD（仅展示摘要）、自动选择最佳方案——我只需要确认一次"开始？"即可。 |
| **需求描述** | ① **智能默认项目**：ContextMatcher 返回的第一候选 >0.8 分时自动确认，不要求用户选择项目。② **Issue 可选**：无匹配 Issue 时静默跳过（当前是阻塞步骤）。③ **PRD 摘要而非全文**：确认卡片展示 3 行摘要而非完整 PRD。④ **方案默认选最优**：单方案时自动选择；多方案且最优与其他差距>30% 时自动选择。⑤ **批量确认**：将所有待确认项（项目/Issue/方案/风险）合并为一张卡片——"准备就绪，开始？" 一键确认。 |
| **范围 (Do/Don't)** | **Do：**自动确认逻辑/PRD 摘要/Issue 可选/批量确认卡片。<br>**Don't：**不移除任何已有能力（用户仍可手动展开完整 PRD/方案）。 |
| **数据契约** | **批量确认卡片:** `{ project: str, issue: str|null, prd_summary: str, plan_summary: str, risks: [str], estimated_time: str, estimated_tokens: int }` |
| **异常定义** | ① 所有候选分数均 <0.5 → 不自动确认，展示候选列表。② 自动确认后用户不满意 → 提供"撤销并手动选择"按钮。 |
| **成功标准→验收** | **SC1:** 简单任务 1 轮确认 → **AC1:** "改日志级别"从输入到确认"开始"仅 1 轮交互。<br>**SC2:** 复杂任务 ≤3 轮 → **AC2:** "实现支付网关"交互轮次≤3（含项目选择+PRD确认+开始）。 |
| **待定决策** | **Q:** 自动确认阈值可调吗？→ **决议：** 0.8 默认，环境变量覆盖。 |

| ADR (架构决策记录) |  |
| --- | --- |
| **决策** | UX 简化通过前端 `ChatPanel` + 后端 `ContextMatcher` 的 `requires_confirmation` 字段驱动。后端标记 `requires_confirmation=False` 时前端自动跳过确认步骤。 |
| **架构位置** | `frontend/src/components/chat/ChatPanel.vue`（修改——批量确认卡片）、`frontend/src/stores/chat.ts`（修改——自动确认逻辑）、`orbit/context/matcher.py`（修改——阈值配置化）。 |
| **实施细节** | **批量确认卡片**在候选卡片之后展示，整合项目/方案摘要/风险/预估 Token，用户仅需一次点击。 |
