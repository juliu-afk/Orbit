# compose:review

## 思考框架（必须遵守——对标 Compound 12-Agent 审查 + Google Code Review 标准）

### 审查维度（按优先级排序）

#### 1. 正确性（权重 40%）——逻辑是否对？
检查清单:
- [ ] 正常路径——happy path 行为是否符合 spec？
- [ ] 边界条件——空输入/零值/负数/超长字符串/特殊字符？
- [ ] 异常处理——网络失败/DB 超时/文件不存在/权限不足？
- [ ] 状态一致性——并发场景下状态是否可能不一致？
- [ ] 类型安全——是否有隐式类型转换导致精度丢失（尤其金额/浮点数）？

#### 2. 安全性（权重 25%）——有没有漏洞？
检查清单:
- [ ] 注入风险——SQL 拼接？命令拼接？模板注入？
- [ ] XSS——用户输入直接渲染到 HTML？
- [ ] 硬编码密钥——API key / Token / Password 在代码中？
- [ ] 权限检查——每个端点/函数是否校验了调用者权限？
- [ ] 敏感数据泄露——日志/错误消息中是否包含密钥/PII？

#### 3. 性能（权重 15%）——会不会慢？
检查清单:
- [ ] N+1 查询——循环内访问数据库？
- [ ] 缺失索引——新表/新查询是否有对应索引？
- [ ] 大循环复杂度——O(n²) 或更差？
- [ ] 不必要的内存分配——大列表全量加载而非流式/分页？

#### 4. 可维护性（权重 20%）——以后能改吗？
检查清单:
- [ ] 命名——变量/函数名是否准确描述其用途？
- [ ] 单一职责——函数是否做太多事？（>50 行 → 考虑拆分）
- [ ] 三行相似不抽象——重复代码 3 次以上 → 提取
- [ ] 过早抽象——用了一次就提取 → 合并回去
- [ ] 注释——业务逻辑/非直觉代码是否注释了 WHY？

## 审查原则
- **宁错杀不放过**——不确定也算作问题，标注 severity=minor 并说明不确定
- **有例证**——每个问题引用具体文件:行号
- **无视作者解释**——只看代码本身，不参考 commit message 或 PR 描述
- **diff-only**——仅审查变更部分，不审查全文件

## 流程
1. 读取变更文件——git diff 或 file list
2. 逐文件按上述 4 维度审查
3. 每个发现标注 severity + 维度 + 位置 + 修复建议

## 审查报告格式（每行精确格式）
```
path:line: <severity_emoji> <severity>: [<维度>] <问题描述>. <修复建议>.
```

severity:
- 🔴 critical: 必须修（安全漏洞/数据丢失/金钱错误）
- 🟡 major: 应该修（逻辑错误/性能问题/可维护性差）
- 🟢 minor: 建议修（命名/风格/注释）

示例:
```
src/api/routes/users.py:42: 🔴 critical: [安全] 用户输入直接拼接到 SQL 查询——SQL 注入风险。改用参数化查询: session.execute(select(User).where(User.id == user_id)).
src/services/tax.py:128: 🟡 major: [正确性] tax_rate 从 float 转为 Decimal 时可能丢失精度——金额计算必须全程 Decimal。改用 Decimal(str(tax_rate)).
```
