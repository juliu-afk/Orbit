# B 系列技术方案：孤儿路由接入 + 桩变真（分批 PR 计划）

> 日期：2026-07-11
> 背景：生产验收契约审计发现 14 条后端路由已实现但前端无入口（孤儿），另有 3 个后端桩返回假数据。
> 用户决策：全建 UI + 桩变真（真数据非 mock）+ backup 恢复强制建 + 拆多个 PR。
> 前置：PR1（5 断链修复）已合并 master(#272)。

## 一、总原则

- 每个 PR 独立可交付、可回滚、可单独验收。
- 按"后端就绪度"和"依赖链"排序：先接现成的（低风险），后建重工程。
- 每 PR 走完整 Orbit 流程：分支 → 实现 → 测试 → exe 验证 → PR → 审查 → 合并。
- graph-snapshots（大工程）单独立 epic，不塞进快 PR。

## 三、分批 PR 计划

### PR2 — 洞察接线（tier1，最小最安全）
- **范围**：codegraph outline + insights impact
- **后端**：无改动（两端点就绪）
- **前端**：
  - codegraph store 加 `fetchOutline(file)` + `fetchImpact(symbol)`
  - OutlinePanel 接入编辑器（打开文件→拉 outline→点击跳行）
  - ImpactGraph 接入 CodeGraphDrawer 详情区（选中节点→symbol→拉 impact）
- **风险**：低。纯前端接线，端点已验证。

### PR3 — tasks 桩变真 + DAG 取消
- **后端**：tasks.py HTTP 端点接真实 Scheduler（删 `_mock_store`，POST→`Scheduler.run_task`，GET→checkpoint 读，cancel→调度器取消）。~50-80 行。
- **前端**：Dashboard DAG 节点加"取消"按钮（调 POST /tasks/{id}/cancel）。
- **风险**：中。触及调度器接线，需回归测试任务生命周期。

### PR4 — loop 定时任务管理 UI
- **后端**：无改动（5端点就绪）。
- **前端**：新 loop store + 定时任务面板（列表/新建/暂停/恢复/删除），挂设置页或运维区。
- **风险**：中。全新 UI。

### PR5 — git blame gutter
- **后端**：无改动。
- **前端**：编辑器 gutter 叠加层（每行 author/time/is_agent），开关切换。
- **风险**：中。编辑器集成。

### PR6 — codegraph test-gaps UI
- **后端**：无改动。
- **前端**：代码查看页"测试覆盖分析"按钮 + 空洞面板（参数×已有测试）。
- **风险**：低-中。

### PR7 — ReviewView 子系统（含 risk + 行评论）
- **后端**：无改动（6端点就绪）；建议顺手修 insights/risk 的 `task_id` 参数名误导（实为 review_id）。
- **前端**：
  - 新建 ReviewView.vue + 路由 `/review`（参照现有测试模板：toolbar/文件树/diff/Approve-Reject）
  - review store 接线（createReview/fetchReview 现无调用方）
  - RiskPanel 作为 ReviewView 侧栏（review_id → /insights/risk）
  - ReviewCommentPanel 接入行级评论
- **风险**：大。这是完整子系统，本 PR 最重。risk 数据需 ComposeOrchestrator 跑过审查才有内容。

### PR8 — insights/health 死代码清理
- insights/health 是死代码，前端已用 observability/health（真数据）。
- **动作**：删除 insights/health 端点（或改别名指向 observability）。可并入 PR2 或单独小 PR。
- **风险**：极低。

### PR9 — backup 在线恢复 UI（PRD 冲突，用户强建）
- **后端**：restore 端点就绪。
- **前端**：恢复按钮 + **二次确认弹窗** + **恢复前自动快照**（缓解 PRD 安全顾虑）。
- **风险**：中。误操作会覆盖生产数据，安全护栏是核心。

### EPIC（不进快 PR）— graph-snapshots 历史图谱
- 后端无历史图引擎。需：CodeNode/Edge 加 git_sha 列 + DB 迁移 + commit 钩子触发建图 + 快照存储 + 按 rev 查询。多人周。
- **建议**：单独立项，出独立 PRD/技术方案。前端时间轴滑块保持 disabled 直到后端就绪。

## 四、推荐执行顺序

低风险接现成的先走，重工程后置：
`PR2(接线) → PR8(清理,可并PR2) → PR6(test-gaps) → PR3(tasks) → PR4(loop) → PR5(blame) → PR9(backup) → PR7(ReviewView大件) → EPIC(snapshots)`
