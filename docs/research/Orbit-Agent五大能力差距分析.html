执行摘要

基于对「AI Agent 五大基础能力深度技术报告」（感知记忆、规划推理、工具调用、闭环容错、自我进化与对齐）的逐项对标，Orbit v0.25 的 42 个落地模块被逐一评估。

总判断

Orbit 在工具调用（③）上领先业界，在感知记忆（①）和规划推理（②）上有扎实基础但缺关键进阶，在闭环容错（④）上有检测回滚但缺诊断自愈，在自我进化（⑤）上几乎空白——而这一项恰好是 FDE「碎石路→高速公路」方法论的工程化实现。

#能力整体评级Orbit 覆盖率致命缺口数
①感知与记忆⚠️ 有基础缺分层~60%0
②规划与推理⚠️ 有线性缺反思探索~45%1 (ReflAct)
③工具调用✅ 领先~90%0
④自主决策与闭环容错⚠️ 有检测回滚缺诊断自愈~40%1 (元认知层)
⑤自我进化与对齐🚫 几乎空白~10%1 (EvolveR 闭环)

3 个致命缺口 = 3 个必须补的能力。 不补的后果：Agent 会在财务/审计场景中（P0）自信地做出错误决策且无法自察，同时（P1）无法从每次客户部署中自动进化——FDE 模式无法规模化。

差距：能做线性规划，不会反思、不会探索备选方案

文档要求Orbit 现状为什么缺这个会出问题优先级
ReflAct
持续反思"Agent 当前状态是否仍对准目标"		 🚫 无 致命。 ReAct 的经典失败模式——Agent 在行动中逐渐漂移而自察不到。UIUC+斯坦福 AgentDebug 研究证实：Agent 会遗忘初始目标、混淆历史步骤、自信地做错事。在财务场景——一个审计程序走偏了但 Agent 不反思 = 执业风险。
goal/ 有结果验证但那是事后检查，不是过程反思。		 P0 致命
PreAct
行动前先预测可能结果		 🚫 无 Orbit 直接行动→错了回滚。没有"先想想行动可能的结果是什么"的预测阶段。回滚有成本（时间+Token）。预测可以避免回滚。 P2
Tree-of-Thought / MCTS
多路径探索		 🚫 无 规划是单路径线性的。复杂审计场景（舞弊风险评估）需要多路径探索。MCTS 在物流调度中效率比 A* 高 3 倍。 P2
⛔ P0 致命缺口

ReflAct 是必须补的。 没有持续反思对准目标的 Agent 在财务场景下不可接受。具体实现：在每一个 ReAct 循环的 Observation 之后插入一个 Reflection 阶段——"我现在离目标更近了吗？当前步骤是否仍在目标方向上？如果不是，我应该回溯还是调整？"

差距：能发现错误能回滚，但不能诊断、不能自愈、不能预判

文档要求Orbit 现状为什么缺这个会出问题优先级
元认知层 (Metacognitive Layer)
次级 Monitor Agent 监控主 Agent，预测失败、提前拦截		 🚫 无 致命。 9 层防幻觉防止了代码层面的幻觉——但没有防止Agent 决策层面的错误：目标遗忘、上下文混淆、自信地做错事。这需要一个独立于主 Agent 的次级监控 Agent——它不执行任务，它监控执行任务的 Agent 是否在偏离目标。文档原话："在任务到达不可恢复的错误状态之前提前发起人工移交"。Orbit 目前只能事后回滚，不能事前预判。 P0 致命
Agent 级错误诊断
AgentErrorTaxonomy + 根因溯源		 🚫 无 Orbit 知道"出错了"——不知道"错在哪个阶段、什么类型"。防幻觉层针对代码质量错误——不对 Agent 决策错误分类。 P1
自愈运行时 (VIGIL)
观察→诊断→修复自身行为		 🚫 无 Orbit 出错→回滚→重试。VIGIL 是出错→诊断原因→修复病因→继续。回滚重试可能遇到同样的错误——因为病因没修。 P1
ReSeek
每步有效性评估 + 单步回溯		 🚫 无 Orbit 回退整个检查点——粒度太粗。ReSeek 可以只回退一步。 P2
预判式 HITL
在失败发生前主动移交人类		 ⚠️ 被动 HITL 当前 HITL 是"等失败了再找人"。预判式 HITL 是"我快不行了，现在叫人"。财务场景下，某些错误不能等到发生——必须在临界点之前拦截。 P1
⛔ P0 致命缺口

元认知层 (Metacognitive Layer) 是必须补的。 这是一个独立于主 Agent 的次级 Monitor Agent。它的唯一职责是：监控主 Agent 的行为，基于预定义触发器（过度延迟、重复动作、目标漂移）预测失败，在任务到达不可恢复状态之前触发人工移交或自动纠正。

这个能力和 9 层防幻觉是互补的——防幻觉管代码质量，元认知层管 Agent 决策质量。

⑤ 自我进化与对齐 — 🚫 几乎空白，这是 FDE 飞轮的引擎

差距：EvolveR 闭环——文档描述的就是 FDE 方法论的技术实现

EvolveR 阶段Orbit 现状为什么缺这个会出问题优先级
① 在线交互
Agent 执行任务，收集成功与失败的完整轨迹		 ⚠️ 轨迹被记录 (observability/events)，但不结构化收集 存了日志但不是"可提炼的轨迹" —
② 离线自蒸馏
将轨迹蒸馏为抽象、可复用的策略原则		 🚫 无 这恰恰是 FDE 方法论的技术实现。 Sankar 的"碎石路→高速公路"循环——FDE 在客户现场建碎石路，总部抽象泛化成高速公路——EvolveR 就是这条循环的自动化版本。Orbit 让 FDE 执行任务，但 Orbit 自己不从任务中学习。每个客户的部署经验没有被自动蒸馏成可复用策略。没有自进化 = FDE 模式无法规模化 = 每个客户都是从头开始。 P1 关键
③ 经验库维护
基于效用分数剪枝与更新		 🚫 无
④ 策略进化
用 GRPO 强化学习优化行为		 🚫 无
对齐监控 (ANCHOR)
在自进化各阶段注入人工监督		 🚫 无 没有自进化所以暂时不需要对齐监控。但一旦加了自进化必须先有对齐监控。文档揭示的 Alignment Tipping Process——自进化可能导致对齐崩溃。必须先有 ANCHOR 再有 EvolveR。 P1 关键
⚠️ P1 关键缺口

EvolveR 闭环是 FDE 规模化的技术前提。 当前 FDE 的"碎石路→高速公路"靠人脑抽象——一个人类 FDE 驻场后回去跟团队说"我发现了一个模式"。EvolveR 让 Agent 自己从执行轨迹中提炼模式。这是"一个 FDE 管 5 个客户"升级到"一个 FDE 管 50 个客户"的前提。

但对齐监控必须先于自进化——ANCHOR 框架必须嵌入进化过程的每个阶段。不能让 Agent 在没有安全护栏的情况下自主进化。

开发优先级：3 致命 + 2 关键 + 2 增强

Phase A：闭环容错强化（P0 致命 · 2-4 周）

任务产出对标能力

Phase C：自进化闭环 + 对齐护栏（P1 关键 · 6-10 周）

目标版本 v0.30–v0.31新增模块 2 个
