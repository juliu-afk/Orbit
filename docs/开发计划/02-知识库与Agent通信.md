## 1. 总览：借鉴理念与融入原则

 
本补充章节的借鉴方法遵循以下三条核心原则：

 
- 原则一：理念对齐优先于代码照搬。先理解每个开源项目的核心思想，判断它解决的是不是你当前最痛的问题，再设计融入方案。
- 原则二：差异分析后融合，而非替换。对比 V14.1 架构与开源项目设计的异同，找到可以融合的点，而不是推翻现有设计。
- 原则三：最小化落地，先 PoC 后推广。选一个最相关的模块，先做小规模概念验证，验证效果后再逐步扩展到整个系统。

 
所有借鉴方案均已映射到 V14.1 的具体 Step 模块，确保与已有开发计划完全兼容。

## 2. Fable 5 System Prompt 逆向工程剖析

 
Anthropic 为 Claude Fable 5 设计的 System Prompt（约12万字符、1585行、72个命名章节）是目前公开领域最完整的AI Agent“操作系统配置”。以下提炼其核心设计原则及对V14.1的启发。

 

### 2.1 七大核心设计原则

 | 原则 | Fable 5 实现方式 | V14.1 借鉴落地 |
| --- | --- | --- |
| 1. 结构先行 | 使用类似XML的闭合标签构建绝对语义边界；ANTML语法调用工具 | Agent Prompt 使用 &lt;instructions&gt;、&lt;rules&gt;、&lt;tools&gt; 标签分隔；工具调用使用 {{invoke:name}} |
| 2. 防御性编程 | 开篇即最高优先级热修复指令；明确红线清单；拒绝危险请求话术 | 每个Agent Prompt顶部设紧急指令区；L1-L8验证层作为“红线引擎” |
| 3. 模块化设计 | 外部只读技能文档（SKILL.md），按需加载；容器化操作环境 | 五Agent共享底层框架，差异化管理角色定义和工具列表；与Docker沙箱（Step MVP-03）理念一致 |
| 4. 自适应思考 | effort参数（max/high/medium）动态控制思考深度 | Agent System Prompt 加入Effort控制指令；简单任务用low/medium，复杂任务用high |
| 5. 降级路由 | 无法处理或高风险请求自动改由其他模型回答；“感觉有风险则少说” | L3高熵触发后Agent进入“沉默模式”；调度器状态机增加降级路径 |
| 6. 防蒸馏机制 | 内置推理提取分类器，检测到白嫖推理逻辑则拒绝或提供垃圾代码 | V14.1审计链核心推理步骤可标记为confidential，输出时打码或脱敏 |
| 7. 双版本策略 | Fable和Mythos共享底层模型，仅System Prompt不同（带/不带安全措施） | 五Agent共享底层System Prompt框架，仅“角色定义”和“可用工具列表”差异化 |

## 3. 六大开源项目的借鉴与落地

 
以下每个项目均按“核心理念 → 可借鉴点 → 具体落地步骤”框架分析，并映射到V14.1的具体Step。

 

### 3.1 OpenRath：用 Session 重构状态管理

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | Session 是“一等公民”，Agent 只是 Session 上的变换层。所有证据、状态、分支都附着在 Session 上。 |
| 可借鉴点 | V14.1 当前的“共享上下文”本质上是 Session 的雏形，但还不够系统和强大。 |
| 落地步骤 | 1. 将 CheckpointData.context 升级为 Session 对象（`/src/scheduler/session.py`），包含 task_id、state、evidence（证据链）、lineage（血缘关系）。
 2. 实现 Session 的 Fork/Merge：在调度器状态机中，当进入分支状态时调用 session.fork()，合并时调用 session.merge()。
 3. 将审计表升级为 Session 的持久化层：task_audit_trail 表作为 Session 证据链的持久化存储。 |
| 影响Step | Step 2.2（检查点）、Step 5.1（调度器）、Step 1.2（审计表） |

 

### 3.2 Parlant：用“细粒度指南”替代“巨型Prompt”

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | 用动态匹配的“细粒度指南”（Guidelines）替代臃肿的System Prompt。每条指南包含触发条件和具体行动。 |
| 可借鉴点 | V14.1 五个Agent的System Prompt正在变得臃肿，可以用Parlant的思路拆分和组织。 |
| 落地步骤 | 1. 将 System Prompt 拆解为 Guidelines 列表，存储在配置文件或数据库中。
 2. 实现 Guideline 匹配引擎（`/src/scheduler/guidelines/matcher.py`），根据当前任务上下文动态匹配相关 Guideline。
 3. 实现 ARQ（Attentive Reasoning Queries）机制，在构造LLM请求时让模型逐条“思考”如何满足匹配到的Guidelines。 |
| 影响Step | Step 5.2（Agent角色与Prompt）、Step 0.3（需求澄清） |

 

### 3.3 HALO：用“执行轨迹分析”实现Agent自我进化

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | 收集Agent执行轨迹 → 分析失败模式 → 生成修复建议 → 应用到Harness → 循环。 |
| 可借鉴点 | V14.1 有审计表但只用于“看”，没有用于“分析”和“自动改进”。 |
| 落地步骤 | 1. 接入 OpenTelemetry，在调度器关键路径埋点导出Traces。
 2. 实现 HALO 分析引擎（`/src/analysis/halo_engine.py`），定期分析跨执行的共性失败模式。
 3. 实现“修复-部署”闭环：将分析报告转化为代码修改建议，自动或半自动应用到V14.1配置中。 |
| 影响Step | Step 7.2（AgentOps）、Step 2.2（审计表） |

 

### 3.4 MetaGPT：将SOP编码为Agent协作流程

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | 将标准化操作程序（SOP）编码为Prompt，让Agent像专业人士一样按流程协作。 |
| 可借鉴点 | V14.1 状态机是粗粒度SOP，每个状态内部（如CODING）的Agent行为还不够标准化。 |
| 落地步骤 | 1. 为每个Agent定义详细的SOP步骤（如DeveloperAgent：查询图谱→生成Diff→应用Diff→运行测试→提交）。
 2. 将SOP编码为Prompt模板，要求Agent输出每一步的执行结果。
 3. 实现SOP执行监控（`SOPValidator`），检查Agent是否按照SOP步骤执行，跳过关键步骤则驳回重试。 |
| 影响Step | Step 5.2（Agent角色）、Step 5.1（调度器） |

 

### 3.5 ReDel：用“事件日志+交互式回放”提升可调试性

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | 支持自定义工具、委派方案、基于事件的日志记录和交互式回放。 |
| 可借鉴点 | V14.1审计表是“事件日志”，但缺少“交互式回放”能力。 |
| 落地步骤 | 1. 丰富审计表事件类型粒度，细化到“工具调用开始/结束”、“LLM请求/响应”等。
 2. 在Vue3驾驶舱中增加“任务回放”页面，按时间线逐帧回放完整执行过程。
 3. 实现“单步执行”和“断点”功能，允许开发者在调试模式下介入观察。 |
| 影响Step | Step 6.1（驾驶舱）、Step 1.2（审计表） |

 

### 3.6 OpenAgents：用向量数据库构建长期记忆

 | 维度 | 内容 |
| --- | --- |
| 核心理念 | Agent加入共享网络，通过向量数据库实现长期记忆的语义检索。 |
| 可借鉴点 | V14.1教训库目前还是概念，可以用OpenAgents思路做实。 |
| 落地步骤 | 1. 部署Milvus或Qdrant向量数据库。
 2. 将“教训”向量化：失败原因、修复方案、上下文通过Embedding模型存入向量库。
 3. 在PLANNING阶段用当前PRD向量检索相似历史任务及其教训，注入Prompt。 |
| 影响Step | Step 5.2（Agent角色）、Step 3.1-3.3（三图谱） |

## 1. 总览：双模架构设计理念

 

> 核心洞察：“图谱化之后仍然需要RAG，不是因为图谱不够好，而是因为两者解决的是不同的问题。”

 | 维度 | 精确查询（图谱） | 语义检索（RAG） | 适用场景 |
| --- | --- | --- | --- |
| 查询方式 | Cypher/SPARQL 精确匹配 | 向量相似度检索 | — |
| 返回结果 | 确定性的实体和关系 | 相关性排序的文本片段 | — |
| 准确率 | 100%（数据存在时） | 60-85%（依赖embedding质量） | — |
| 适用问题 | “流动比率的公式是什么？” | “最近关于流动比率的监管要求变化” | 精确 vs 探索 |
| 未知处理 | 查不到就返回空 | 总能返回“最像”的结果 | 确定 vs 模糊 |
| Token开销 | 零Token | 低（向量检索+LLM摘要） | 成本敏感 vs 探索优先 |

 

> 核心原则：图谱负责“事实锚定” RAG负责“上下文扩展”
 两者通过MCP协议（Model Context Protocol）统一暴露给Agent，Agent根据问题类型自动选择或组合使用。

### 3.1 为什么图谱化后仍需要RAG？

 

#### 场景1：用户的问题本身就不精确

 

 

#### 场景2：图谱无法覆盖所有知识

 

 

#### 场景3：多跳推理需要“从图谱出发，到文本落地”

 

 

### 3.2 双模查询对比

 | 查询类型 | 使用技术 | 何时使用 | 输出示例 |
| --- | --- | --- | --- |
| 精确概念查询 | 图谱（Neo4j） | Agent需要确认“流动比率”的准确定义 | {"name":"流动比率","formula":"流动资产/流动负债","range":"1.5-2.0"} |

## 5. 来源筛选与质量控制机制

 

### 5.1 权威来源白名单

 | 领域 | 可接受来源 | 拒绝来源 |
| --- | --- | --- |
| 会计 | 财政部会计准则委员会、IASB（国际会计准则） | 个人博客、未经验证的论坛 |
| 金融 | 证监会（CSRC）、上交所/深交所、Wind/彭博公开数据、人民银行 | 股吧、雪球评论、自媒体 |

### 5.2 五级筛选流程

 
1. 源过滤：仅允许白名单来源。
2. 格式校验：检查文档结构、编码、元数据完整性。
3. 一致性校验：与已有知识库交叉比对。
4. 专家抽检：随机抽取10%新知识人工审核。
5. 版本背书：记录版本号、更新人、审核人、时间戳。

| 代码图谱 | 代码中引用的专业术语自动关联到知识图谱 | Agent生成代码时，可验证业务术语的正确性 |
| 数据库图谱 | 数据库字段与会计科目/金融指标的映射 | 确保数据模型与专业语义一致 |

## 7. 完整PRD/ADR

 

### 7.1 PRD：领域知识图谱模块

 | PRD · 领域知识图谱模块 |
| --- |
| 背景 | 系统需要处理会计、金融、法律等专业性任务，通用LLM的专业知识不足且存在幻觉风险。需构建外挂式领域知识图谱，通过“精确查询+语义检索”双模架构，为Agent提供权威、可验证的专业知识。 |
| 用户故事 | 作为DeveloperAgent，我在编写财务计算代码时，通过MCP调用query_knowledge(domain="finance", concept="CurrentRatio", mode="exact")获取流动比率的准确定义和公式；同时通过query_knowledge(mode="semantic")检索相关的监管要求全文。 |
| 需求描述 | ① 三层知识架构（通用层→领域层→实例层）。
 ② 支持会计/金融/法律三个领域的本体设计。
 ③ 实现五级来源筛选流程。
 ④ 通过MCP协议暴露 query_knowledge 工具，支持 exact/semantic/hybrid 三种模式。
 ⑤ 每次知识查询记录到审计表（task_audit_trail）。
 ⑥ 提供合规验证工具 validate_compliance。 |
| 范围 | Do：会计、金融、法律三个领域的本体设计+实例层注入；Neo4j+向量库双存储；MCP协议集成；零Token精确查询。
 Don't：不支持领域知识的自动演进（仍需人工审核）；不支持Agent自动更新知识库（只读）。 |
| SC→AC | SC1: 精确查询零Token → AC1: MCP调用 mode="exact"，返回确定结果，耗时&lt;50ms，无LLM调用。
 SC2: MCP协议兼容 → AC2: Agent通过标准化MCP工具调用知识图谱。
 SC3: 来源可追溯 → AC3: 每个知识点返回source_uri。 |

 

### 7.2 ADR：知识图谱双模架构

 | ADR · 知识图谱采用“精确查询 + 语义检索”双模架构 |
| --- |
| 决策 | 知识图谱模块同时支持两种查询模式：
 1. 精确模式：通过Neo4j Cypher查询，返回确定性结果（零Token，&lt;50ms）。
 2. 语义模式：通过向量检索，返回相关性排序的文档片段。
 3. 混合模式：图谱锚定 + 向量扩展，联合检索。 |
| 理由 | 精确查询解决“是什么”（概念定义、法规条文、公式验证）。
 语义检索解决“还有什么”（开放探索、全文检索、上下文扩展）。
 两者通过统一的MCP接口暴露，Agent无需感知底层差异。 |
| 技术栈 | Neo4j 5.x（结构化知识存储）；Milvus/Qdrant（向量检索）；SentenceTransformers（语义向量化）；MCP Python SDK（协议集成）。 |
| 实施细节 | 见第4节 MCP Server实现和代码示例。 |
