核实结论：11/12 条代码确实不存在；#5 部分（导入边已记录，控制流/eval/exec 无 —— 详见附录）。本报告只收录价值≥「中」的 8 项。

3. 逐项详情

⭐ #3 多语言防幻觉（TypeScript / JavaScript）

代码现状：确实仅 Python。l1_graph.py 无 typescript/ts 处理；l4_type.py 仅 mypy，无 tsc。
能做：中。L1 加 TS/JS 的 Tree-sitter 语法；L4 用 tsc --noEmit 替代 mypy；L2 沙箱换 node runtime。图谱 Schema 已语言无关，主要是解析器与类型检查器适配。
值得做：高。当前只服务 Python 项目——若目标用户写前端/Node，这是硬门槛。扩语言直接放大适用面，是 8 项中 ROI 最高的。
源文件：src/orbit/hallucination/l1_graph.py · l4_type.py · graph/engines/code_graph.py

#2 文档图谱 DocGraph

代码现状：确实无（docgraph/document_graph/pgvector/neo4j 全 0 命中）。仅存在于设计文档。
能做：中。设计已给 pgvector + 图引用 Schema；需落地向量存储 + 文档↔符号跨图谱边。
值得做：中。补全「五图谱」体系，实现需求/设计文档 ↔ 代码的双向追溯，契合 Orbit「治理/追溯」定位。代价：引 pgvector/Neo4j 依赖，破坏「零依赖 SQLite 单机启动」的设计优势——需权衡。
源文件：设计见 docs/产品路线图.md:121；无实现代码

#12 自然语言交互 —— Issue 写操作

代码现状：确实无（无 close-issue 逻辑）。当前仅查询/关联 Issue。
能做：易。GitHub/GitLab REST API 加写操作端点。
值得做：中。闭环体验好（对话里直接闭 Issue）。但写操作需权限校验 + 人工审批门禁，否则风险高。做需谨慎。
源文件：src/orbit/projects/ · NL 交互层

#10 驾驶舱 —— 任务管理 / 历史回放

代码现状：确实无（前端无 task-manage/replay/mobile）。
能做：易。纯前端功能。
值得做：任务管理（创建/取消）中——UX 提升；历史回放 中——结合已有审计链 task_audit_trail + Trace 数据可做「时间旅行调试」，价值放大。移动端仍低。
源文件：frontend/src/；数据源已有 observability/audit.py · trace.py

#6 数据库图谱 —— 多主库

代码现状：确实无（无 mongo/nosql/multi-db）。当前仅单主库、仅关系型。
能做：多库 易（连接池 + 库标识）；NoSQL 中（schema-less 反射难）。
值得做：多主库 中——真实项目常多库（主库+只读+分片）。NoSQL 低——无 schema，图谱价值弱。
源文件：src/orbit/graph/engines/（DBGraph 反射）

#8 DAG —— 跨任务依赖

代码现状：确实无（dag_runner.py 无 dynamic-topology/cross-task）。当前静态图、任务内 DAG。
能做：跨任务依赖 中；运行时改拓扑 难（状态一致性 + 检查点复杂度爆炸）。
值得做：跨任务依赖 中——大项目多 Goal 间有真实依赖。动态拓扑 低——收益不抵复杂度。
源文件：src/orbit/scheduler/dag_runner.py · goal/dependency_analyzer.py

#5 代码图谱 —— 控制流分析 (CFG)

代码现状：部分。导入边已记录（code_graph.py:444-448 _record_import_edge(alias.name)）；CFG 控制流分析、eval/exec 确实无。
能做：CFG 中难（AST→CFG 构建，已有 graph/engines/slicer.py 程序切片可复用）；eval/exec 理论不可判（动态语言本质，静态分析上限）。
值得做：CFG 中——提升 L2 动态追踪与程序切片精度。eval/exec 低——投入产出比差。
源文件：src/orbit/graph/engines/code_graph.py · slicer.py
⚠️ 文档 vs 代码出入：Non-Goal 写「不解析导入别名」，但代码 _record_import_edge(alias.name) 已记录导入别名。或为「记录≠解析」措辞差异，或文档滞后，建议复核后更新说明书。

#4 防幻觉 —— L6 gRPC / L8 K8s ConfigMap 验证

代码现状：确实无（l6_contract.py 无 grpc；l8_config.py 无 configmap/k8s）。L6 仅 OpenAPI，L8 仅文件配置。
能做：gRPC 易（加 .proto 解析）；ConfigMap 易（加 YAML kind 识别）；L5 IO 密集 难（形式化验证 IO 本质困难）。
值得做：中——云原生/微服务项目适用；否则小众，按需加。
源文件：src/orbit/hallucination/l6_contract.py · l8_config.py
