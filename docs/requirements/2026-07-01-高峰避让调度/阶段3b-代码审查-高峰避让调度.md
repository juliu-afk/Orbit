## 🔴 Bug

### B1: `_check_windows` 窗口到期判断使用了窗口开始时间而非结束时间

**位置**: `offpeak_scheduler.py:886-895`

```python
# 当前代码
for window_start_iso, tasks in windows.items():
    window_start = datetime.fromisoformat(window_start_iso)
    minutes_since_start = (now - window_start).total_seconds() / 60
    if minutes_since_start < 0:
        continue  # 窗口还没到
```

**问题**: 当窗口开始时间在未来时（`minutes_since_start < 0`），直接 `continue` 跳过。但 `windows` dict 的 key 是 `target_window_start`——所有排队任务按此分组。如果 watcher tick 发生在窗口开始后，`minutes_since_start >= 0` 成立，代码进入释放逻辑。**这本身是正确的**。

**实际 bug**: 第 916 行 `if now >= window_start` 条件与上面 894 行的 `continue` 互斥——894 已经跳过了 `minutes_since_start < 0`（即 `now < window_start`）的情况，所以 916 的 `if now >= window_start` 永远为 True。这不是 bug，是冗余条件。

**撤回**: 不是 bug。逻辑正确。

**重新审查**: 真正的潜在问题在 `pop_for_window` —— 它按 `target_window_start` 精确匹配字符串，而不是范围匹配。如果窗口被 reschedule 到不同时间，旧窗口开始时间的任务不会被匹配。**这是设计意图**——每个窗口内任务只在该窗口释放。

**结论**: 无 Bug。审查通过。

## 🟡 性能

### P1: `_check_windows` 每次 provider 迭代都查询全量 queued 任务

**位置**: `offpeak_scheduler.py:873`

```python
for provider in self._peak.providers:
    all_queued = await self._queue.list_all("queued")  # ← 每次循环都查
    provider_tasks = [t for t in all_queued if t.provider == provider]
```

**问题**: 有 N 个厂商时，执行 N 次全量 SQL 查询。

**修复**: 查询移到循环外，按 provider 分桶。

```python
all_queued = await self._queue.list_all("queued")
by_provider: dict[str, list[DeferredTask]] = {}
for t in all_queued:
    by_provider.setdefault(t.provider, []).append(t)

for provider in self._peak.providers:
    provider_tasks = by_provider.get(provider, [])
    if not provider_tasks:
        continue
    ...
```

**严重度**: 低——厂商数 ≤4，每秒仅 1 次查询。修不修都行。

## 🟢 代码质量

### Q1: 未使用变量 `priority_order`

**位置**: `offpeak_scheduler.py:521`

```python
priority_order = {"CRITICAL": 0, "HIGH": 1, "NORMAL": 2, "LOW": 3}
```

定义了但 SQL 的 CASE WHEN 语句内联了排序逻辑。变量未使用。

**修复**: 删除该行。

### Q2: `_push_sync` 用两个独立连接

**位置**: `offpeak_scheduler.py:490-514`

INSERT 用一个连接（`with self._connect()`），SELECT COUNT 用另一个。两个连接在不同事务中，可能读到不一致的状态（虽然 WAL 模式下问题不大）。

**修复**: 合并到一个 `with` 块。

### Q3: `PeakWindow` 动态属性脆弱

**位置**: `offpeak_scheduler.py:231-237`

`starts_at_iso` 和 `ends_at_iso` 在 `next_offpeak_window()` 中通过 `setattr` 动态附加到 dataclass 实例。调用方必须检查 `hasattr`。

**建议**: 用 NamedTuple 或 TypedDict 替代，或定义一个包含 ISO 字段的新返回类型如 `ResolvedWindow(NamedTuple)`。

**优先级**: P2——当前工作正常，但未来维护者容易踩坑。

### Q4: `enqueue()` 高低峰分支逻辑重复

**位置**: `offpeak_scheduler.py:772-787`

```python
if self._peak.is_peak(provider, now):
    window = self._peak.next_offpeak_window(provider, now)
else:
    window = self._peak.next_offpeak_window(provider, now)  # 完全相同的调用
```

**修复**: 合并为一个 `window = self._peak.next_offpeak_window(provider, now)`。

### Q5: `_on_task_done` 成本节省可能不准

**位置**: `offpeak_scheduler.py:971-994`

- 使用 `deferred_task.estimated_tokens` 而非 `actual_tokens` 算成本节省
- `_MODEL_COSTS.get(provider, 0.001)` 的 key 是 provider 名（如 "deepseek"），但 routing.py 的 key 是完整模型 ID（如 "deepseek/deepseek-v4-pro"）——永远 miss

**修复**: 改 key 映射或直接取 mid-point 价格。
