## 1. 问题背景

Thariq 提出核心框架：**Map（地图）≠ Territory（疆域）**。

| Map（你给 Agent 的） | Territory（真实执行环境） |
|---------------------|------------------------|
| 提示词、Skill、上下文 | 代码库、边界条件、现实约束 |

差距 = **Unknowns（未知项）**。Agent 遇到未知项时只能猜测你的意图。模型越强，瓶颈越不在 Agent 的能力，而在你澄清未知项的能力。

**四象限分类**：
- **已知的已知** — 写进提示词的
- **已知的未知** — 知道自己还没搞清楚的
- **未知的已知** — 太熟悉忘了写、但看到能认出来
- **未知的未知** — 完全没考虑到的

Thariq 总结了八种发现未知项的方法。经评估，**五种可直接落地到 Orbit**，按优先级分三批。

## 2. 用户故事

### P0 —— 偏离日志（Deviation Log）

**作为** Orbit 用户/开发者，
**我希望** Agent 在执行任务过程中自动记录偏离原计划的决策及原因，
**以便** 任务结束后能追溯"为什么这么做"，并将偏离数据喂给 evolution 模块优化后续 prompt。

**Why P0**: 改动最小（checkpoint 模块加一个字段 + 一个 Markdown 生成器），
直接提升 evolution 质量。偏离日志是"实现中"发现未知项的唯一记录机制——没有它，所有执行中的信息差都会丢失。

### P1 —— 盲区扫描（Blindspot Scanner）

**作为** Orbit 用户，
**我希望** 在任务创建时，Monitor Agent 能自动做一次前向盲区扫描：知识图谱缺失什么？代码图谱未覆盖哪些路径？哪些决策点需要我确认？
**以便** 在执行前消除尽可能多的"未知的未知"。

**Why P1**: metacognition 模块已有次级 Monitor Agent，基础设施现成。
盲区扫描是 Thariq 方法论的**第一步**——"先让 AI 当镜子，照出你的无知"。

### P1 —— 测验模式（Quiz Mode）

**作为** Orbit 用户，
**我希望** 任务完成后 Agent 能生成一份改动理解测验（5 道判断题），我必须全对才能标记任务完成，
**以便** 确保我真正理解每处改动，而非"看着差不多就合并"。

**Why P1**: 高频价值场景（merge 前最后一道防线）。
Thariq 原话："不理解每一处改动，就不配合并。这条规矩让事故率暴跌。"

### P2 —— 原型优先（Prototype First）

**作为** Orbit 用户，
**我希望** 对 UI/设计类任务能选择"先出独立 HTML 原型→反馈→再接入真实代码"的工作模式，
**以便** 在改动成本最低的阶段（原型）对齐视觉预期。

**Why P2**: 只对 UI/设计类任务有价值，适用面窄。但 Orbit 驾驶舱自身也是 UI，后续 UI 翻新可直接受益。

### P2 —— 跨语言语义迁移（Cross-language Semantic Transfer）

**作为** Orbit 用户，
**我希望** 指定"参考 X 语言的库 Y 的 Z 功能，用我们项目的语言重新实现相同语义"，
**以便** 用已有优质实现作为参照物，消除"不知道怎么做才算好"的未知项。

**Why P2**: 需要元图谱新增 `SEMANTICALLY_EQUIVALENT_TO` 关系类型，改动大。
Thariq 原话："最好的参考资料是源代码。即使参考代码是另一种语言写的。"

## 3. 验收标准

### 偏离日志 (P0)
- [ ] AC-P0-1: 任务执行过程中检测到偏离原计划时，自动写入 `implementation-notes.md`
- [ ] AC-P0-2: 偏离日志包含：时间戳、偏离决策、原因、替代方案
- [ ] AC-P0-3: 任务终止（成功/失败/熔断）时，偏离日志写入 checkpoint
- [ ] AC-P0-4: EvolutionEngine 可消费偏离日志优化 agent 指令（SCOPE + GEPA）
- [ ] AC-P0-5: 偏离日志对人类可读（Markdown 格式，含代码片段引用）

### 盲区扫描 (P1)
- [ ] AC-P1-1: 任务创建时 Monitor Agent 执行前向 blindspot pass
- [ ] AC-P1-2: 扫描输出：① 知识图谱缺失概念 ② 代码图谱未覆盖模块 ③ 用户需确认的模糊决策点
- [ ] AC-P1-3: 扫描结果以交互式问答呈现（用户可逐条回复或批量跳过）
- [ ] AC-P1-4: 扫描在 5 秒内完成（对知识图谱 + 代码图谱的只读查询）
- [ ] AC-P1-5: 扫描结果可选择性纳入任务上下文

### 测验模式 (P1)
- [ ] AC-P2-1: 任务完成后生成 HTML 测验报告（5 道判断题，覆盖核心改动）
- [ ] AC-P2-2: 测验题目基于 diff + 实现笔记自动生成
- [ ] AC-P2-3: 用户必须全对才能触发 `task_complete` 状态变更
- [ ] AC-P2-4: 答错显示正确答案 + 解释，允许重试
- [ ] AC-P2-5: 测验历史可追溯（存入 checkpoint + audit）

### 原型优先 (P2)
- [ ] AC-P3-1: compose 编排新增 `prototype_first` 策略
- [ ] AC-P3-2: 策略流程：生成独立 HTML → 用户反馈 → 接入真实代码
- [ ] AC-P3-3: 原型阶段不触碰真实代码库（沙箱隔离）
- [ ] AC-P3-4: 用户反馈后 Agent 自动提取关键设计决策写入任务上下文

### 跨语言语义迁移 (P2)
- [ ] AC-P4-1: 元图谱新增 `SEMANTICALLY_EQUIVALENT_TO` 边类型
- [ ] AC-P4-2: 支持 `参考 X 实现 Y` 语义查询，跨 Tree-sitter 语言解析
- [ ] AC-P4-3: 迁移结果经沙箱执行验证（语义等价 ≠ 语法相同）

## 4. 已确认决策

1. **偏离日志粒度**: ✅ 只记 major/critical。偏离定义 = 原计划（技术方案/实现计划）中的决策被实际执行改变，且影响架构/数据模型/API 签名/安全。命名调整、import 路径等机械性调整不算偏离。
2. **测验模式触发条件**: ✅ 仅核心模块改动触发。核心模块定义：`scheduler/`, `graph/`, `hallucination/`, `checkpoint/`, `agents/`, `gateway/`, `compliance/`, `evolution/`, `metacognition/`, `sandbox/`, `memory/`, `compose/`, `communication/`。用户可在 task config 中 `quiz: true` 手动开启非核心任务测验。
3. **盲区扫描触发时机**: ✅ 自动触发。任务创建时 Monitor Agent 默认执行 blindspot pass。用户可 `--skip-blindspot` 跳过。
4. **P2 实施时序**: ✅ P2（原型优先 + 跨语言语义迁移）放入 backlog，V15.3+ 独立迭代。本次 V15.2 仅 P0+P1。

## 5. Non-Goals

- 不替代现有 review 流程——测验模式是 review 后的额外门禁
- 不实现"AI 自动修复盲区"——盲区扫描只暴露问题，不做自动修复
- 不改变现有 checkpoint 回滚逻辑——偏离日志是追加记录，不影响回滚路径
- 不实现 Thariq 文章中的"引用式通信"全部类型——仅落地源代码语义迁移，不含截图/设计稿引用

## 6. 边缘情况

| 场景 | 预期行为 |
|------|---------|
| 盲区扫描时知识图谱为空 | 输出"知识图谱未初始化，跳过概念扫描"，仍完成代码图谱扫描 |
| 偏离日志写入失败（磁盘满） | 降级到内存缓存，任务结束前最后尝试一次写入 |
| 测验模式下 Agent 任务失败/熔断 | 跳过测验（没有改动需要理解） |
| 用户跳过盲区扫描后发现未知项 | 支持中途发起 `--blindspot` 补充扫描 |
| 原型优先策略应用于非 UI 任务 | compose 自动降级为普通策略并提示 |
| 跨语言迁移的源语言无 Tree-sitter 支持 | 降级为纯文本参考（不建立图谱关系） |
| 多个偏离同时发生（并发 Agent） | 每个 Agent 独立写偏离日志，checkpoint merge 时去重合并 |

## 7. Orbit 模块影响评估

| 模块 | 影响 | 风险等级 |
|------|------|---------|
| `checkpoint/` | 偏离日志字段 + merge 逻辑 | 中——核心模块，回滚兼容性需仔细测试 |
| `metacognition/` | 新增 BlindspotScanner | 低——Monitor Agent 已有基础设施 |
| `evolution/` | GEPA/SCOPE 消费偏离日志 | 低——新增数据源，不改现有逻辑 |
| `modes/` | 新增 `quiz` 交互协议 | 低——纯新增，不影响已有模式 |
| `compose/` | 新增 `prototype_first` 策略 | 低——策略模式扩展 |
| `graph/meta_graph.py` | 新增 `SEMANTICALLY_EQUIVALENT_TO` | 中——Schema 变更，需 migration |
| `sandbox/` | 原型优先的沙箱隔离 | 低——已有沙箱基础设施 |

## 8. 成功指标

- 偏离日志启用后，evolution 模块的 prompt 迭代周期缩短 20%（定性观察）
- 盲区扫描在至少 60% 的任务中产出了用户未主动提出的确认点
- 测验模式上线后，因"未理解改动"导致的回滚/热修复减少
