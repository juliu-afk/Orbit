Fable 5 方法论 — Orbit 落地影响总结

2026-07-14 | 4 PRs (#304, #306, #308, #311) | 来源: Thariq Shihipar "A Field Guide to Fable"

一、融入 Orbit 业务流程的方式

P0 偏离日志 — 执行中记录

任务开始→ wiring.on_task_start() 创建 DeviationLogger→ Agent 执行中调用 deviation_record MCP tool→ 任务结束 flush → checkpoint→ GEPA/SCOPE 消费偏离数据自动进化 prompt

时机：每个 Agent 任务的全生命周期。偏离日志夹在任务开始和结束之间，Agent 在执行中主动记录偏离。

触发条件：Agent 发现实际执行的决策与计划不同，且涉及数据模型/API/安全（白名单规则，零 Token）。

P1 盲区扫描 — 执行前诊断

任务创建→ start_monitor 注入 BlindspotScanner→ runner 调用 preflight_blindspot()→ 三维度并行扫描 (知识/代码/模糊)→ 结果注入 Agent 上下文

时机：每个任务开始执行前（PARSING 阶段之前）。

触发条件：自动。用户可通过 --skip-blindspot 跳过。

P1 测验模式 — 执行后门禁

任务完成→ on_task_end 检查 changed_files→ 核心模块? → 推送 quiz_required 事件→ 前端展示测验 → 用户作答→ 全对 → 标记完成

时机：任务执行完成后，合并之前。

触发条件：改动触及核心模块（scheduler/graph/hallucination/checkpoint/evolution 等 13 个）。

P2-1 原型优先 — UI 类任务策略

Spec 解析→ strategy="prototype_first"?→ _run_prototype_first() 生成 HTML→ 用户反馈 → 提取设计决策→ 注入决策到实现阶段

时机：ComposeOrchestrator 解析 Spec 之后，派发 Task 之前。

触发条件：Spec 中设置 strategy: prototype_first。

P2-2 跨语言语义迁移 — Agent 工具

ToolRegistry 初始化→ _register_fable5_tools() 注册 3 个 MCP 工具→ Agent 通过 dispatch() 调用

时机：系统启动时自动注册。Agent 执行中按需调用。

触发条件：Agent 需要"参考 X 语言实现 Y 语言等价功能"时调用 semantic_transfer_find。

二、与用户交互方式的改变

之前

  • 任务创建 → Agent 执行 → 完成。中间无交互
  • 用户拿到 diff，靠肉眼判断是否正确
  • 偏离了计划？用户不知道，除非翻日志
  • 不懂的领域？Agent 硬做，用户看不出问题
  • UI 改动？直接写代码，改完才看到效果

之后

  • 执行前：盲区扫描告诉你"你还有这些不知道的"
  • 执行中：偏离日志告诉你"我改了这个决策，因为..."
  • 执行后：测验考你"你真的理解改动了吗？"
  • UI 任务：先看 HTML 原型，确认了再写代码
  • 跨语言参考：Agent 能查"这个 Rust 库有没有 Python 等价实现"
核心变化：从"提交任务 → 等结果 → 自己判断"变为"盲区扫描 → 偏离透明 → 测验验证"的三阶段交互闭环。

三、对 Orbit 的量化帮助

Token 节省

机制节省方式估算
盲区扫描 执行前暴露未知 → 减少 Agent 在未知领域反复试错的 turn 数 估算节省 15-30% agent turns（每轮 ~2000 tokens）
偏离日志 → SCOPE 进化 偏离模式自动升级为战略规则 → Agent 下一次不犯同样错误 累积效应：每 10 个任务减少 ~5% 修正性 turn
原型优先 HTML 原型零成本迭代布局 → 省去"写完代码→看了不对→改代码"的循环 UI 类任务节省 40-60% 实现阶段的 token
语义迁移 Agent 查已有等价实现 → 不需要从零推断实现方式 参考式实现比从零推断节省 ~30% token

代码质量

机制质量提升
测验模式"不理解就不配合并"——防止用户盲目合并不理解的改动。Thariq 原话："事故率暴跌"
偏离日志所有偏离原计划的决策有据可查——审计追溯完整。GEPA 用偏离数据定向进化 prompt
原型优先UI/UX 在原型阶段对齐——不会出现"代码写完了但交互不对"的返工
语义迁移"最好的参考是源码"——用已验证的实现作为模版，减少从零设计的错误

性能

模块性能特征
偏离日志零开销记录（内存缓存 → 批量 flush）。is_deviation() 白名单规则，毫秒级
盲区扫描三维度并行查询（asyncio.gather），5 秒超时 abort。纯只读，不修改图谱
测验模式仅核心模块触发（~30% 任务），非核心任务零开销。LLM 调用一次（~2-5s）
原型优先仅在 ComposeOrchestrator 中、且 strategy 显式设置时触发
语义迁移SQLite LIKE 查询，毫秒级。仅在 Agent 显式调用 MCP 工具时执行

四、总结

8 个新文件，10 个修改文件，~2300 行代码，~500 行文档，38 个单元测试。

Fable 5 方法论不是给 Orbit 加了一个新功能——是改变了 Agent 和用户之间的信息关系：

  • 执行前 → 扫盲区（减少未知的未知）
  • 执行中 → 记偏离（透明化"为什么改主意"）
  • 执行后 → 考理解（不理解的代码不合入）
  • UI 类 → 先原型（成本最低的阶段对齐）
  • 跨语言 → 有参考（不重复造轮子）

Thariq 原话："模型越强，瓶颈越不在代码，而在'不知道自己不知道'。" 这五个模块就是 Orbit 对抗信息差的武器。
