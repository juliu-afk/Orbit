# Orbit 业务层减熵——阶段 1 PRD

> 基线文档：[[06-减熵体系]] | 日期：2026-06-29
> 关联文档：[[阶段2-技术方案-业务层减熵]]

## 一、背景

Orbit 是 AI 编程工具。它的核心价值不是"Orbit 自己的代码有多好"，而是"用 Orbit 写出来的用户项目代码有多好"。

当前 Orbit 已在编码阶段落地 24 个模块，但缺少**系统性的减熵策略**来降低用户项目的熵。Agent 生成的代码如果本身就高熵（依赖膨胀、上下文过载、决策信息丢失），那用户接手后的维护成本反而更高。

**目标**：将 8 项减熵策略植入 Orbit 的 Agent 调度管线，让每个 Agent 产出的代码天然低熵。

## 二、用户故事

### P0（上下文 + 输出质量）

| ID | 故事 | 价值 |
|----|------|------|
| US-B1 | 作为用户，我不希望 Agent 看到和当前任务无关的代码，导致生成偏离需求的方案 | Agent 只收到相关函数/类，不是整个文件。上下文越少，幻觉越少。 |
| US-B2 | 作为用户，我不希望 Agent 生成长篇大论——能 3 行写清楚的事不要写 30 行 | Agent 的 system prompt 植入简洁规则，生成后自动做简洁审计。 |

### P1（知识复用 + 稳定性）

| ID | 故事 | 价值 |
|----|------|------|
| US-B3 | 作为用户，我不希望 Agent 从零写 CRUD 端点——每次风格都不一样 | Agent 优先调用已验证的代码模板，填充参数而非自由生成。 |
| US-B4 | 作为用户，我不希望同一个文件被 Agent 反复修改，说明需求可能不对 | 自动检测高熵文件 → 触发需求重确认，避免越改越乱。 |
| US-B5 | 作为用户，我不希望 Agent 之间互相不知道对方做了关键设计决策 | 决策日志持久化 → 后续 Agent 自动加载，不会重选一遍。 |

### P2（自动治理）

| ID | 故事 | 价值 |
|----|------|------|
| US-B6 | 作为用户，我不希望 Agent 随手装新包导致依赖爆炸 | 安装前检查已有替代方案 → 拦截不必要的新依赖。 |
| US-B7 | 作为用户，我不希望测试覆盖了 happy path 但边界全是洞 | 自动分析函数参数 × 已有测试输入值 → 找空洞。 |
| US-B8 | 作为用户，我不希望每次新建项目都要手写一整套 CLAUDE.md | Orbit 的六图谱自动生成初始 CLAUDE.md + 迭代中更新。 |

## 三、范围

**Do**：
- `prompt/builder.py` RULES_BLOCK 加第 9 条简洁规则
- `context/matcher.py` 实现 Tree-sitter 相关性打分
- `knowledge/` 下建 `templates/` 模板库（8-10 个高频模式）
- `events/bus.py` 加文件变更事件 → 编辑摇摆检测
- `memory/` 模块扩建设计决策存储与检索
- `resource_guard/` 加依赖膨胀拦截检查
- 代码图谱加测试覆盖空洞检测查询
- 六图谱 → CLAUDE.md 自动生成 pipeline

**Don't**：
- 不改动已有 Agent 的核心行为逻辑（只加约束，不改推理）
- 不强制所有用户项目都必须用这些策略（可配置开关）
- 不引入新的外部依赖（用已有 Tree-sitter / 代码图谱能力）

## 四、Non-Goals

- 不追求"全自动"——依赖安装/关键决策仍需用户确认
- 不覆盖非 Python 项目的测试空洞检测（初版）
- 不做 CLAUDE.md 的格式标准化（只生成内容，不规定用户怎么组织）

## 五、边缘情况

| 场景 | 预期行为 |
|------|---------|
| Tree-sitter 解析失败（语法错误的文件） | 降级到全文注入 + 标注"解析失败，上下文可能不精确" |
| 模板命中但参数不匹配（如 CRUD 模板需要 5 个字段但任务描述只有 3 个） | Agent 自由生成 + 记录未命中原因到知识库供后续优化 |
| 决策日志中相同问题出现矛盾答案 | 标记冲突 → 触发 Clarifier 重新澄清 → 记录最新决策并链接旧版本 |
| 编辑摇摆检测在上线第一天没有历史数据 | 渐进式：1 天数据启动，14 天达到完整判定能力 |
| 依赖拦截误判（如 difflib 确实不够用需要 fuzzywuzzy） | 提示替代方案，但仍允许用户确认后安装 |
| 测试空洞检测误判（参数类型标注错误导致假空洞） | 只报 >0 的空洞，不报"参数无类型标注"的情况 |

## 六、验收标准

| ID | 标准 | 测量方法 |
|----|------|---------|
| AC-B1 | 上下文注入 token 量 -30%（同任务对比） | 截取 `prompt/builder.py` 的 context 层输出 token 数 |
| AC-B2 | Agent 生成的 Python 文件平均行数 -15% | 统计 10 个任务的文件行数对比 |
| AC-B3 | 模板命中率 ≥40%（高频任务） | 命中次数 / (命中 + 自由生成) |
| AC-B4 | 编辑摇摆检测触发后，需求重新澄清率 ≥70% | 被标记"高熵"的文件后是否有 PRD 修订 |
| AC-B5 | 依赖拦截准确率 ≥90%（不过度拦截不过度放行） | 被拦截的安装中用户同意的比例 |
| AC-B6 | 测试空洞检测报告 ≥1 个空洞/文件（有复杂参数的文件） | 空洞检测输出数量 |
| AC-B7 | CLAUDE.md 自动生成的内容覆盖六维度 ≥4/6 | 人工评审生成结果的完整性 |
| AC-B8 | 所有改动不影响已有功能的 CI 通过 | `pytest tests/ -q` exit 0 |

## 七、配置开关

所有业务层减熵策略可独立开关（`configs/orbit.toml` 或环境变量）：

```toml
[entropy_reduction]
context_pruning = true       # P0
code_conciseness = true      # P0
template_library = true      # P1
edit_churn_detection = true  # P1
decision_log = true          # P1
dependency_guard = true      # P2
test_gap_detection = false   # P2（初版实验性）
claude_md_generation = false # P2（初版实验性）
```

**默认**：P0/P1 全开，P2 关。

## 八、待确认问题

| # | 问题 | 默认答案 |
|---|------|---------|
| Q1 | 简洁规则会不会过于限制 Agent 创造力？ | 只限制行数不限制方案——能短写就短写，需要长写就长写。阈值 200 行/文件只是触发审查，不是硬截断。 |
| Q2 | 决策日志存哪里？要不要新表？ | 复用 `memory/` 的存储。加一个 `decision` 类型的 MemoryFileType。不新增 SQL 表。 |
| Q3 | 模板库初始模板来源？ | 从 Orbit 自身已实现的代码提取（API 路由、Service、测试）——吃自己的狗粮。 |
| Q4 | 测试空洞检测是否支持 TypeScript？ | 初版只支持 Python（Tree-sitter Python grammar）。TS 后续加。 |
