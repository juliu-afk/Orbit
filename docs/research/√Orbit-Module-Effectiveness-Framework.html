2. 模块契约模型

每个模块必须声明 4 种契约。测试 = 验证契约是否成立。优化 = 在满足契约的前提下提升性能。

2.1 契约模板

ModuleContract:
  name: "ModuleName"

  input_contract:
    required: [param1: Type, param2: Type]     # 如果缺失→立即失败
    optional: [param3: Type = default]          # 缺失→使用默认值
    preconditions: [cond1, cond2]               # 调用前必须满足的条件

  output_contract:
    success: { shape: ..., invariants: [...] }   # 成功返回的形状和不变式
    failure_modes:                               # 每种失败模式
      - mode: TimeoutError
        condition: "执行超过 timeout 秒"
        return: ErrorType
      - mode: InvalidInput
        condition: "输入不符合 preconditions"
        return: ErrorType

  performance_contract:
    p50_latency_ms: number
    p99_latency_ms: number
    max_memory_mb: number

  side_effects:
    - type: "db_write" | "http_call" | "file_write" | "none"
      description: "..."

3.2.3 逐层 KPI 定义

层KPI公式目标测量频率
L1 符号漏报率 FN / (TP + FN) 其中 FN=已知不存在的符号未被检测到 ≤ 1% 每次 CI
L1 符号误报率 FP / (FP + TN) 其中 FP=正确符号被标记为不存在 ≤ 3% 每次 CI
L3 熵阈值 AUC ROC AUC of entropy_threshold vs actual hallucination label ≥ 0.70 每周
L4 类型错误检出率 检测到的类型错误 / 总注入的类型错误 ≥ 90% 每次 CI
L7 运行时错误捕获率 沙箱中代码运行失败且被 L7 标记的 / 沙箱中代码运行失败总数 ≥ 95% 每次 CI
ALL 管道整体 F1 2 × Precision × Recall / (Precision + Recall) ≥ 0.85 每次 CI

3.3 优化策略

🔴 立即: 逐层消融实验

方法: 在基准数据集上，逐层禁用→测量 F1 变化→量化每层贡献

Layer_Contribution[i] = F1(all_layers) - F1(all_layers \ {i})

预期: L5 贡献应该最低。如果确实如此→降级 L5 为默认关闭（需显式开启）。

行动: 写一个 50 行的消融实验脚本，跑完即知哪些层是 token 黑洞。

🟠 短期: L1 阈值调优

当前 L1 是二值的（符号存在/不存在）。改为三级:

  • DEFINITELY_ABSENT: 在当前项目+依赖+标准库中都不存在 → 高置信阻断
  • UNCERTAIN: 仅在部分路径中不存在（可能动态导入）→ 告警不阻断
  • PRESENT: 存在 → 通过

预期: 误报率下降 50%+

🟠 短期: L3 阈值校准

当前 L3 的熵阈值是硬编码的 (DeepSeek 0.75 / GLM 0.70)。需要:

  1. 收集 1000+ 个正确输出和 100+ 个幻觉输出的熵值分布
  2. 用 Youden's J 统计量计算最优阈值（而非硬编码）
  3. 每个模型独立校准

🔵 中期: L5 范围限定

Z3 形式化验证不可能对所有代码有效。限定:

  • 仅验证 纯函数（无副作用，无 IO）
  • 仅验证 ≤ 30 行的函数
  • 自动检测可形式化的性质（如 range check, null check）

不满足条件的函数直接跳过 L5，不浪费 token

🔵 中期: 管道并行化

当前是串行执行 L1→L4→L3→L2→L6→L8→L7→L5

优化: L1+L4+L3+L6+L8 可以并行运行（均无外部依赖），L2+L7 串行（需沙箱），L5 可选

预期延迟下降 40-60%

3.4 防幻觉管道契约

HallucinationPipeline.contract:
  input:  code: str (LLM生成的代码片段)
  output: ValidationResult { passed: bool, errors: [str], warnings: [str], level }

  performance:
    validate_quick(): p99 ≤ 500ms   # 仅 L1+L4+L3
    validate_full():   p99 ≤ 5000ms  # 全量

  invariants:
    - passed=true AND code 含 NameError → BUG (漏报)
    - passed=false AND code 在沙箱中成功运行 → 误报 (需记录)
    - 单层崩溃不影响其他层 (fail-open per layer)

4. LLM 网关 & 熔断器 测量与优化

4.1 CircuitBreaker

契约项内容
输入model_key: str, 每次调用的成功/失败标记
核心行为CLOSED→连续5次失败→OPEN(60s冷却)→HALF_OPEN→探测→CLOSED
关键不变量OPEN 状态下 必须 拒绝所有请求 (bypass率=0)

4.1.1 测量方法

测试: 故障注入

Mock LLM 返回连续失败 → 验证:

  1. 第 5 次失败后状态=OPEN
  2. OPEN 期间所有请求被 CircuitOpenError 拒绝
  3. 60s 后自动进入 HALF_OPEN
  4. HALF_OPEN 探测成功→CLOSED
  5. HALF_OPEN 探测失败→重回 OPEN

测试框架: pytest + asyncio + freezegun (控制时间)

已有: tests/unit/ 应已有部分覆盖

测试: 并发 RMW 安全

10 个协程同时请求 HALF_OPEN 状态→验证仅 1 个获得探测权

assert sum(1 for probe in results if probe) == 1

4.1.2 KPI

KPI目标测量方法
故障检测延迟≤ 5 次调用内检测到failure_count 到 failure_threshold 的调用数
恢复时间≤ 60s (cooldown)opened_at 到 HALF_OPEN 的时间差
误熔断率0% (不容忍)OPEN 期间模型实际可用的次数 / 总 OPEN 次数
状态持久化重启不丢失进程重启后 SQLite 中的状态与重启前一致
半开冲突0 次并发探测probe_in_flight > 1 的情况

4.1.3 优化

  • 自适应阈值: 根据历史失败模式动态调整 failure_threshold。如果某模型稳定→提高阈值避免误熔断；如果某模型不稳定→降低阈值快速隔离
  • 预热机制: OPEN 状态下定期（如每 30s）对模型发送低成本探针（如简单 echo），提前检测模型恢复

5. 编排层 (Goal / Scheduler / Compose) 测量与优化

5.1 MetaOrchestrator

🔴 核心KPI: 任务拆分准确率

定义: 生成的 TaskDAG 中，子任务划分正确且依赖关系与实际代码改动匹配的比例

测量: 人工标注 20 个典型需求，定义"标准 TaskDAG"→对比 MetaOrchestrator 生成的 DAG vs 标准 DAG

Accuracy = |generated_edges ∩ reference_edges| / |reference_edges ∪ generated_edges|

目标: ≥ 80%

🟠 核心KPI: 并行效率

定义: 层内并行的实际加速比

Speedup = Σ单任务时间 / 实际层完成时间

目标: 3个并行任务时 speedup ≥ 2.0x

瓶颈排查: 如果 speedup 低→检查 asyncio.gather 是否被某个慢任务拖累→检查 Semaphore 是否正确释放→检查 Worktree 创建是否串行化

🟠 核心KPI: 预算分配效率

定义: BudgetAllocator 分配预算与实际消耗的相关性

Corr = Pearson(allocated_budget[], actual_consumption[]) 对所有子任务

目标: ≥ 0.7

相关性低→BudgetAllocator 在瞎猜→改进: 基于历史任务复杂度（代码行数/文件数/依赖深度）而非纯 PRD 长度

7. 上下文系统 测量与优化

7.1 核心 KPI: 上下文 ROI

ROI = (Agent 任务成功率提升%) / (额外 Token 消耗%)

如果 ROI < 1.0 → 上下文系统在消耗 Token 但没带来等比例的收益。

5 个 Prebuilder 各自的上下文被 Agent 实际引用的比例

收集: Agent 输出中引用了 Prebuilder 提供的哪些信息

目标: 引用率 ≥ 20%。低于 10% 的 Prebuilder→考虑移除

Builder 有效性

7 个 Builder各自构建的上下文是否被 Agent 使用

同样追踪引用率。低引用率 Builder → 移除或降级为可选

5 个 Scanner发现的文件是否覆盖了 Agent 最终修改的文件

覆盖率 = |Scanner发现文件 ∩ Agent实际修改文件| / |Agent实际修改文件|

目标: ≥ 80%

10. 自我进化系统 (evolution/) 测量与优化

这是 Orbit 最独特的模块，也是最难测量的。进化系统声称"让系统越用越好"——如何量化？

🔴 GEPA 进化效果

A/B 测试: 同一批 50 个任务，分别用 GEPA 进化后的 Prompt vs 初始 Prompt

测量: 任务成功率差异

GEPA_gain = SuccessRate(after_GEPA) - SuccessRate(before_GEPA)

目标: ≥ +5% (如果 ≤ 0% → GEPA 无效)

🟠 SCOPE 双流有效性

SCOPE 声称"分离成功路径和失败路径"→ 需要验证:

  1. 成功路径上的 Agent 成功率是否高于平均水平
  2. 失败路径是否被逐渐废弃（使用频率下降）

🟠 蒸馏保真度

方法: 蒸馏后的 Agent vs 原始 Agent 在 50 个测试任务上对比

测量: 成功率保持率 + 延迟下降率

Fidelity = SuccessRate(distilled) / SuccessRate(original)

目标: ≥ 0.95 (蒸馏后质量不应明显下降)

🔵 ANCHOR 对齐度

定义: Agent 行为与 ANCHOR 基准的偏离程度

测量: 定期运行 ANCHOR 对齐测试套件 (预定义的 20 个关键行为检查点)

目标: 对齐度 ≥ 90%

10.1 进化系统核心挑战: 归因

最难的问题: 如果任务成功率从 70% 提升到 75%，这 5% 的提升应该归因于:
  • GEPA 进化了 Prompt？
  • SCOPE 优化了路径选择？
  • LLM 模型本身升级了？
  • Context 系统改进了？
  • 随机波动？

解决方法: 每次只改一个模块，保持其他模块不变。建立对照时间线——每周固定一天跑全量基准测试，记录所有模块版本和表现。用因果分析（如 DoWhy 库）而非纯相关性。

核心原则

原则 1: 无测量不优化

任何模块在被量化评估之前，不投入优化资源。直觉驱动优化 = 随机游走。

原则 2: 消融实验是金标准

证明模块有效的唯一方法: 拿掉它→系统变差。F1 贡献 < 0.03 的模块应降级或移除。

原则 3: 基准数据集防污染

评估数据 ≠ 训练数据。用同一份数据训练和评估 = 自欺欺人。

原则 4: 故障安全优先

ProcessGuard、CircuitBreaker、Sandbox 等安全关键模块→先验证它们不会坏。其他模块可以慢慢测。

原则 5: 持续闭环

不是"一次评估→优化→结束"。是每迭代重新评估→找出新的瓶颈→优化→再评估。

最终目标

6 个月后，对于 Orbit 的每一个模块，都能回答:

模块名称: _________
它的契约 (应该做什么): _________
它实际在做什么 (KPI 当前值): _________
它的贡献度 (消融实验 F1 差值): _________
它每任务消耗 (Token/延迟): _________
它的 ROI (贡献度 / 消耗): _________
→ 决策: [保留并优化 / 降级为可选 / 移除]
这一张表的答案 = Orbit 从"研究项目"到"工程工具"的转折点。
