本章回答"为什么这么设计"。研究者优先阅读。所有论断均可回溯至 docs/charter.md（项目"宪法"）与 docs/开发计划/00-架构总览.md。

1.1 核心命题：治理，而非生成 · Governance, Not Generation

中文：Orbit 的根本命题是——LLM 生成代码的能力已经足够，缺的是让这种能力可信、可控、可追溯的治理体系。因此 Orbit 把重心放在"生成之后"和"生成之间"：多 Agent 如何协作、状态如何流转、幻觉如何拦截、决策如何追溯。

English: Orbit's core thesis: LLMs are already *capable enough* at code generation; what's missing is a governance system that makes that capability trustworthy, controllable, and traceable. So Orbit invests in what happens *between* and *after* generation — how agents collaborate, how state transitions, how hallucinations are intercepted, how decisions are traced.

这解释了架构锚定声明的存在（charter.md:87-90）：任何把 Prompt 写成"你是一个 Python 专家"的执行层退化，都会被评审打回，因为它偏离了编排层的定位。

1.2 为什么锚定编排层 · Why the Orchestration Layer

单 Agent 工具（Claude Code / Codex）解决的是"一次生成质量"；Orbit 解决的是"一群 Agent 跑完一个软件项目"。两者不是竞争关系——Orbit 可以把单 Agent 工具当作执行节点，自己专注于上层治理：

单 Agent 工具关心 Orbit 关心
这段代码写得好不好谁来写、谁来审、谁来测、出错回滚到哪
单轮上下文塞什么五图谱全局记忆、跨 Agent 上下文传递
生成结果生成过程的每一步可否被审计

1.3 为什么自研调度器（而非 CrewAI）· Why a Self-Built Scheduler

中文：现成多 Agent 框架（CrewAI 等）是黑盒，调度不可审计、Token 开销大、延迟高，与 Orbit 的"极致性能 + 透明可控"哲学冲突。基准对比（docs/开发计划/00-架构总览.md:83-91）：

维度 CrewAI 自研调度器
单任务 Token~4.5K≤35（目标，可优化至 1/128）
单次耗时（含验证）~32s~1.5s（目标）
内存~200MB 框架开销~50MB
可控性黑盒，难审计完全可控、可修改

结论：零黑盒依赖是不可妥协项。所有调度逻辑用 Python asyncio 有限状态机自己实现，换来的是每个状态转换都能存检查点、能回滚、能审计。

1.4 六项设计哲学 · The Six Principles

来源 docs/开发计划/00-架构总览.md:72-78：

# 哲学 Principle 含义 Meaning
1极致性能 Extreme performance单任务 Token ≤35，调度层延迟 ≤1.5s（不含验证层）
2透明可控 Transparent & controllable所有逻辑可审计、可修改，零黑盒依赖
3全维度覆盖 Full-dimension memory代码+数据库+配置+知识+元图谱 = 五图谱完整记忆
4动静结合 Static + dynamic确定性分析（Tree-sitter/Z3）+ 运行时反馈（追踪/熵监控）
5渐进式复杂 Progressive complexity先跑通核心闭环，再逐步优化，不追求一次性完美
6决策可追溯 Traceable decisions每次动作、每个状态转换、每次验证均可追溯至原始上下文

1.5 关键设计取舍 · Key Trade-offs

1.5.1 追溯 vs 完整回滚（审计链的取舍）

Orbit 的审计链（task_audit_trail）放弃了全量状态回滚（低频，<1% 场景），保留了推理链追溯（高频，100% 任务需要）。代价对比：千次任务审计链约 50MB，而全量 MVCC 快照需 50GB——节约 1000 倍存储。回滚能力改由检查点（checkpoint）在关键状态点提供。

来源 docs/开发计划/00-架构总览.md:430-453。

1.5.2 静态分析的理论上限

核心认知：纯静态分析拦截幻觉的理论上限只有 48.5%–77%，必须与动态验证（沙箱执行、运行时追踪）结合，才能把错误率压到 <3%。这直接决定了 9 层防幻觉"从快到慢、静动结合"的分层结构（见 03 防幻觉）。

来源 docs/开发计划/00-架构总览.md:540-544,607-609。

1.6 不可妥协项 · Non-Negotiables

这些是架构"红线"，任何改动不得违反（来源 AGENTS.md:130-141 + charter.md）：

# 红线 Invariant 理由 Rationale
1锚定编排层Prompt 必须为协作编排服务，退化为执行层则打回（Step 0.4）
2零黑盒依赖所有组件确定性可控、可审计、可修改
3LiteLLM 网关唯一入口禁止直连 provider——统一追踪 + 降级
4Docker 沙箱强制LLM 生成代码不得在宿主机直接执行——安全基线
5防幻觉链路完整性改防幻觉层不能只改一层——单层改动影响上下游判定
6调度器状态一致性状态机改动必须全路径回归（转换/检查点/回滚）——否则任务卡死
7密钥仅环境变量API key / Token 禁止硬编码
8Charter 即宪法架构决策争议时以 charter.md 为准
